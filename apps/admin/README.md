# 后台管理端

后台管理端直接重写旧后台，不迁移旧后台 UI。

后台长期用于成员管理、权限管理、资源管理、审核、导出、审计和异常处理；第一阶段仅作为端到端主线审核控制台落地。后台访问不等于 `998/999`，业务人员只能进入自己有权限的后台模块。

## 第一阶段 P0 前置

后台主线审核控制台必须先接入真实登录、`/auth/me` 会话恢复、`/permissions/me` 权限摘要和按权限点过滤的最小菜单。

后台 P0 壳只做最小可用后台：

- 未登录只展示登录页；
- 已登录后调用 `/auth/me` 和 `/permissions/me` 恢复会话与权限；
- 菜单只展示当前用户有权限的第一阶段主线入口；
- 没有可访问模块时展示无权限空态和退出入口；
- 不提前铺完整后台导航、成员管理、项目审核、运营大盘或系统配置静态页。

静态菜单、静态表格或 mock 审核页不能算后台审核入口完成。`borrowing.application.review` 才能显示物资借用审核/归还入口，`resources.material.manage` 才能显示物资台账维护入口。

后台端必须复用 `@makershub/api-client` 的认证、权限、响应解析、错误处理和令牌续签能力。
不要在 `apps/admin` 内重新写一套后台专用 `fetch` 客户端，否则后续会和成员网页端、小程序端出现错误码、权限摘要和会话恢复口径分叉。
