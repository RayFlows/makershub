# 环境、部署与发布周期

本文面向后续开发、测试、运维和项目负责人，说明 MakersHub 从本地开发到生产发布的环境分层、部署流转、灰度策略、回滚与日常运维要求。

这份文档不替代具体部署脚本。它定义项目周期中的环境边界和操作原则，后续 `infra/` 下的 Docker、CI/CD、服务器配置和运维脚本都应遵守这里的约定。

## 总体原则

MakersHub 的环境设计遵循以下原则：

- 开发环境要方便重置、方便热更新、方便定位问题；
- 预发布环境要尽量接近生产，但不承载真实用户流量；
- 生产环境要稳定、可回滚、可备份、可追踪；
- 同一版本应先在预发布环境验证，再进入生产环境；
- 能用配置解决的差异，不通过改代码解决；
- 数据库迁移、对象存储、权限规则和积分规则上线前必须明确验证和回滚方案；
- Docker 是本地开发、联调和部署交付的基础入口，但开发环境和生产环境不能混用同一套运行方式。

## 环境分层

项目至少分为三个环境：

| 环境 | 推荐名称 | 主要用途 | 面向对象 |
| --- | --- | --- | --- |
| 本地开发环境 | `local` / `dev` | 开发、调试、联调、快速重置数据 | 开发人员 |
| 预发布环境 | `staging` | 真实部署验证、内部验收、发布前检查 | 开发、测试、管理员 |
| 生产环境 | `production` / `prod` | 正式对外服务 | 真实用户 |

后续如项目规模扩大，可以增加：

- `preview`：每个分支或 Pull Request 的临时预览环境；
- `demo`：用于展示的演示环境；
- `backup`：灾备或只读备份环境。

第一阶段不强制建设所有环境，但必须在命名、配置和文档上为这些环境预留位置。

## 本地开发环境

本地开发环境的目标是让开发者在 WSL 或 Linux 中快速启动完整依赖。

当前推荐入口：

```bash
docker compose -f infra/docker/compose.dev.yml up --build
```

本地环境可以采用两种模式。

### 全容器模式

全容器模式通过 Docker 同时启动：

- MySQL；
- MinIO；
- FastAPI 后端；
- 成员网页端；
- 后台管理端；
- 文档站。

适用场景：

- 新成员第一次启动项目；
- 验证 Dockerfile 和 compose 配置；
- 做端到端联调；
- 模拟服务器部署方式。

全容器模式允许源码挂载、热更新和开发默认密码，但不得直接用于生产环境。

### 基础设施容器模式

基础设施容器模式只通过 Docker 启动 MySQL、MinIO 等外部依赖，应用本身在 WSL 中本地运行。

适用场景：

- 高频后端开发；
- 前端开发和调试；
- 需要 IDE 直接调试 Python 或 TypeScript；
- 不想每次重建应用镜像。

后续可以增加脚本：

```bash
pnpm docker:infra
pnpm dev:api
pnpm dev:web
pnpm dev:admin
pnpm dev:docs
```

本地开发环境允许：

- 使用开发密钥；
- 重置数据库；
- 重建 MinIO bucket；
- 开启接口文档；
- 开启热更新；
- 输出较详细日志。

本地开发环境不要求：

- 高可用；
- HTTPS；
- 多副本；
- 严格限流；
- 完整备份策略。

## 预发布环境

预发布环境用于验证即将上线的版本。

预发布环境应该尽量接近生产环境：

- 使用和生产相同的 Docker 镜像；
- 不使用源码挂载；
- 不开启 `--reload`；
- 使用独立数据库；
- 使用独立 MinIO bucket；
- 使用独立域名或内网地址；
- 使用独立环境变量；
- 使用真实的反向代理和 HTTPS 配置；
- 执行和生产相同的数据库迁移流程。

预发布环境不承载真实用户数据，除非经过脱敏。

预发布环境的主要验收内容：

- 镜像能否正常启动；
- 健康检查是否可用；
- 数据库迁移是否成功；
- MinIO bucket 是否初始化；
- 前端静态资源是否正确加载；
- API 文档和接口路径是否符合契约；
- 登录、权限、积分、借用、项目等核心流程是否可演示；
- 日志是否能定位错误；
- 回滚方案是否可执行。

预发布环境通过后，才能进入生产发布。

## 生产环境

生产环境承载真实用户访问，必须遵守更严格的稳定性要求。

生产环境要求：

- 使用固定版本镜像，不在服务器上临时构建；
- 使用生产环境变量和 secret；
- 禁止使用开发默认密码；
- 禁止开启热更新和 debug；
- MySQL 和 MinIO 原则上不直接暴露公网；
- API、Web、Admin 通过反向代理暴露；
- 所有公网入口启用 HTTPS；
- 数据库和对象存储有备份策略；
- 日志和错误可追踪；
- 关键操作有审计记录；
- 发布前有回滚方案。

生产环境不应依赖：

- 本地 `.env` 中的默认值；
- 开发机上的 `node_modules`；
- 源码挂载；
- 临时手动修改容器内部文件；
- 未记录的服务器手工配置。

## 配置管理

不同环境通过配置区分行为，不通过改代码区分行为。

推荐配置来源：

- 本地开发：`.env` 或 `.env.local`；
- 预发布：服务器环境变量或部署平台 secret；
- 生产：服务器环境变量、secret 管理服务或受控配置文件。

配置文件必须区分：

- 应用基础配置，例如 `APP_ENV`、`APP_NAME`；
- 数据库配置，例如 `DATABASE_URL`；
- 对象存储配置，例如 `MINIO_ENDPOINT`、bucket 名称；
- 认证配置，例如 `JWT_SECRET_KEY`、`ACCESS_TOKEN_EXPIRE_MINUTES`、`REFRESH_TOKEN_EXPIRE_DAYS`；
- 邮件验证码配置，例如 `EMAIL_DELIVERY_MODE`、`SMTP_HOST`、`SMTP_USERNAME`、`SMTP_PASSWORD`；
- 跨域配置，例如 `CORS_ORIGINS`；
- 外部服务配置，例如 `WECHAT_APP_ID`、`WECHAT_APP_SECRET`、邮件、短信；
- 前端构建时配置，例如 API base URL。

### 邮件验证码发信

第一阶段真实邮箱验证码发信服务暂定使用 Resend，项目域名记录为 `scumaker.com`。本地开发仍默认
使用 `EMAIL_DELIVERY_MODE=log`，只有预发布或生产环境需要切到真实 SMTP。

Resend 免费层适合当前阶段验证码流量，但它有每日发送上限；如果后续协会成员规模或通知量增长，
需要重新评估额度、国内邮箱送达率和是否升级付费计划。

Resend 通过 SMTP 接入时推荐配置：

```env
EMAIL_DELIVERY_MODE=smtp
SMTP_HOST=smtp.resend.com
SMTP_PORT=465
SMTP_USE_SSL=true
SMTP_USERNAME=resend
SMTP_PASSWORD=<RESEND_API_KEY>
SMTP_FROM_EMAIL=auth@scumaker.com
SMTP_FROM_NAME=MakersHub
EMAIL_HOME_URL=https://scumaker.com
EMAIL_BRAND_IMAGE_URL=https://static.scumaker.com/public/brand/SCUMAKER_logowithtext_email.png
```

接入步骤：

1. 在 Resend 的 Domains 中添加并验证 `scumaker.com`。
2. 按 Resend 页面给出的 DNS 记录配置 SPF 和 DKIM；建议补充 DMARC 记录提高送达可信度。
3. 在 Resend 创建只用于 MakersHub 的 API Key。
4. 把 API Key 写入部署环境变量 `SMTP_PASSWORD`，不要写进 `.env.example` 或仓库文档。
5. 用预发布环境发送测试验证码，确认收件箱、垃圾箱、日志和 Resend Emails 页面都能追踪。

### 邮件品牌图与静态资源域名

验证码邮件中的品牌图不使用 CID 附件，而是使用公开 HTTPS 图片地址。推荐把品牌图放入
MinIO 的 public bucket，再通过静态资源域名暴露：

注意：邮件中使用的是从原始透明 PNG 生成的浅底版本 `SCUMAKER_logowithtext_email.png`。
这样可以避免 QQ 邮箱等移动客户端在夜间模式下把透明区域渲染成黑底，导致黑色字样不可见。

```text
对外地址:
https://static.scumaker.com/public/brand/SCUMAKER_logowithtext_email.png

MinIO 对象:
makershub-public-prod/brand/SCUMAKER_logowithtext_email.png
```

生产配置建议：

```env
MINIO_PUBLIC_BUCKET=makershub-public-prod
MINIO_PUBLIC_URL=https://static.scumaker.com
EMAIL_HOME_URL=https://scumaker.com
EMAIL_BRAND_IMAGE_URL=https://static.scumaker.com/public/brand/SCUMAKER_logowithtext_email.png
```

DNSPod 中需要新增一条记录：

| 主机记录 | 记录类型 | 记录值 |
| --- | --- | --- |
| `static` | `A` | 部署 Nginx/反向代理的服务器公网 IP |

如果静态资源入口由云平台、CDN 或对象存储网关托管，也可以把 `static` 做成 `CNAME`，指向平台给出的域名。

服务器侧可以参考 `infra/nginx/static-minio.example.conf`：

- `https://static.scumaker.com/public/...` 只映射到 MinIO public bucket；
- 管理控制台不暴露公网；
- 私有 bucket 不走这个域名，继续通过后端鉴权和短期签名 URL 访问。

当前第一版临时部署已经在 `132.232.2.227` 上启用：

- Nginx 配置文件：`/etc/nginx/conf.d/static-scumaker.conf`；
- MinIO public bucket：`makershub-public-prod`；
- 邮件品牌图对象：`brand/SCUMAKER_logowithtext_email.png`；
- HTTPS 证书：Let's Encrypt，使用 Docker 版 Certbot 申请；
- 证书续期：`/etc/cron.d/makershub-certbot` 每天尝试续期后重载 Nginx。

本地开发可以继续使用：

```env
EMAIL_BRAND_IMAGE_URL=http://localhost:9000/makershub-public-local/brand/SCUMAKER_logowithtext_email.png
```

这个本地地址只适合开发机预览。真实发给 QQ、学校邮箱或其他外部邮箱时，必须换成公网 HTTPS 地址，
否则邮箱客户端无法加载图片。

邮件列表或邮件详情中的发件人头像不由 HTML 模板控制。正文 logo、`EMAIL_BRAND_IMAGE_URL` 和
`SMTP_FROM_NAME` 只能影响邮件内容和发件人展示名，不能保证改变收件邮箱客户端里的头像。后续如果需要
域名级品牌头像，应单独规划 BIMI/DNS 记录、DMARC 策略和方形品牌图；不同邮箱客户端对 BIMI 的支持程度
也需要实测。

禁止将以下内容提交到仓库：

- 生产数据库密码；
- 生产 JWT 密钥；
- MinIO 生产访问密钥；
- 微信小程序生产密钥；
- Resend API Key、SMTP 密码或任何真实邮件服务密钥；
- 任何真实用户数据导出。

仓库中只保留 `.env.example`，用于说明变量名称、格式和开发默认值。

## 镜像与版本

推荐采用“一次构建，多环境推进”的策略：

```text
提交代码
  -> CI 构建 Docker 镜像
  -> 推送镜像仓库
  -> 部署到 staging
  -> 验收通过
  -> 同一镜像推进到 production
```

同一个版本不应在 staging 和 production 分别构建两次。否则可能出现预发布验证通过，但生产镜像内容不同的问题。

镜像标签建议同时包含：

- Git commit hash；
- 语义化版本号或日期版本号；
- 环境无关的镜像名。

示例：

```text
makershub-api:2026.04.29-abcdef1
makershub-web:2026.04.29-abcdef1
makershub-admin:2026.04.29-abcdef1
```

`latest` 可以用于开发，但不应作为生产发布的唯一依据。

## CI/CD 流程

长期目标流程：

```text
开发分支
  -> 提交代码
  -> 代码检查
  -> 单元测试
  -> 构建前端
  -> 构建后端镜像
  -> 构建前端镜像
  -> 推送镜像
  -> 自动部署 staging
  -> 人工验收
  -> 手动批准生产发布
  -> 部署 production
  -> 发布后检查
```

第一阶段可以先实现较轻的流程：

- 本地执行 `pnpm build`；
- 本地执行后端测试和格式检查；
- 本地构建 Docker 镜像；
- 手动部署到服务器；
- 文档记录每次发布内容。

等核心业务稳定后，再逐步引入自动化 CI/CD。

## 数据库迁移

数据库迁移是发布中风险最高的部分之一。

项目后端应使用迁移工具管理数据库结构，例如 Alembic。

迁移原则：

- 所有表结构变更必须形成迁移文件；
- 迁移文件随代码提交；
- staging 先执行迁移并验证；
- production 执行迁移前必须确认备份；
- 避免直接在生产库手工改表；
- 不轻易删除字段和表；
- 大表变更需要单独评估执行时间和锁表风险。

推荐上线顺序：

```text
备份生产数据库
  -> 部署兼容旧结构的新代码或先执行兼容迁移
  -> 执行数据库迁移
  -> 启动新版本服务
  -> 验证核心流程
```

如果迁移包含破坏性变更，例如删除字段、重命名字段、改变字段含义，必须单独写发布方案。

## 对象存储

MinIO 用于保存图片、附件、材料、项目文档等文件。

开发环境可以自动初始化 bucket。

生产环境要求：

- bucket 名称明确；
- 公共 bucket 和私有 bucket 分离；
- 上传权限和读取权限分离；
- 重要文件有备份策略；
- 不把 MinIO 管理控制台直接暴露公网；
- 文件 URL 生成方式和过期策略有明确约定。

公共静态资源可以使用同一个静态域名，通过路径映射到 public bucket；不同环境通过 bucket 名称或对象 key
前缀区分，不需要为了每个 bucket 单独创建公网域名。业务私有资源不应因为使用同一套 MinIO 而变成公开资源。

建议 bucket 按用途拆分：

- 头像；
- 公共图片；
- 物资和资源图片；
- 项目材料；
- 审批附件；
- 临时上传文件。

第一阶段可以先用较少 bucket，但文档和配置中要保留扩展空间。

## 灰度发布

灰度发布不是第一阶段必须完整实现的能力，但设计上要提前考虑。

常见灰度方式：

- 预发布环境先验收；
- 只对管理员或内部测试成员开放新功能；
- 通过功能开关控制新功能是否启用；
- 通过角色、部门或用户白名单控制可见范围；
- 多副本部署时按流量比例切换；
- 蓝绿部署，在旧版本和新版本之间切换入口。

MakersHub 早期推荐采用：

```text
staging 验收
  -> 管理员或内部成员白名单
  -> 功能开关
  -> 全量开放
```

不建议一开始就做复杂的百分比流量灰度。对协会平台来说，按角色和白名单控制更容易理解，也更容易追踪问题。

后端业务代码中涉及灰度时，应优先使用功能开关或权限点，而不是在代码中写死用户 ID。

## 回滚策略

发布前必须知道如何回滚。

代码回滚：

- 保留上一版本镜像；
- 生产发布时记录当前版本号；
- 出现问题时切回上一版本镜像；
- 回滚后执行健康检查和核心流程验证。

数据库回滚：

- 优先通过兼容性迁移避免回滚数据库；
- 生产迁移前必须备份；
- 破坏性迁移必须有单独恢复方案；
- 如果新代码写入了新格式数据，回滚前要评估旧代码是否能读取。

对象存储回滚：

- 文件上传通常不能简单回滚；
- 删除文件前要谨慎；
- 关键附件应保留审计记录和软删除策略。

配置回滚：

- 发布时记录关键环境变量变更；
- 不在服务器上做无法追踪的手工改动；
- 重要配置变更应随发布记录一起保存。

## 备份与恢复

生产环境至少需要备份：

- MySQL 数据库；
- MinIO 对象数据；
- 环境变量和 secret 的受控副本；
- 部署配置；
- 发布记录。

备份策略应包含：

- 备份频率；
- 备份保存位置；
- 保留周期；
- 恢复演练；
- 备份失败告警。

早期可以先执行每日数据库备份和对象存储定期备份。正式上线前，应至少做一次恢复演练，确认备份真的可用。

## 日志、监控与告警

生产环境需要最基本的可观测性。

至少应记录：

- API 请求错误；
- 登录失败和异常登录；
- 权限拒绝；
- 积分流水变更；
- 借用审批变更；
- 项目审核变更；
- 文件上传失败；
- 数据库连接异常；
- 外部服务调用失败。

后端运行日志默认按用途分流到 `logs/`：

| 文件 | 内容 | 默认保留 |
| --- | --- | --- |
| `app.log` | 普通运行信息、warning，不包含请求访问、debug 和 error 明细 | 30 天 |
| `error.log` | 所有 `ERROR/CRITICAL`，包含请求异常和业务异常 | 180 天 |
| `request.log` | HTTP 请求开始、结束、状态码、耗时和 `request_id` | 30 天 |
| `debug.log` | `DEBUG` 明细，本地和测试默认开启，生产默认关闭 | 7 天 |

日志文件默认每天轮转并压缩。生产环境不应无限保留运行日志，保留周期通过
`LOG_RETENTION`、`LOG_ERROR_RETENTION`、`LOG_REQUEST_RETENTION`、
`LOG_DEBUG_RETENTION` 调整；生产如需临时打开 debug 文件，应明确设置
`LOG_DEBUG_FILE_ENABLED=true`，排查结束后关闭。

API 应用层默认启用基础 HTTP 安全边界：

- `SECURITY_HEADERS_ENABLED`：是否写入基础安全响应头；
- `HSTS_ENABLED`：是否写入 HSTS；未显式设置时仅生产环境默认开启；
- `REQUEST_SIZE_LIMIT_ENABLED` / `MAX_REQUEST_BODY_BYTES`：普通接口请求体上限；
- `RATE_LIMIT_ENABLED`：是否启用应用层兜底限流；
- `RATE_LIMIT_WINDOW_SECONDS` / `RATE_LIMIT_MAX_REQUESTS`：普通接口限流窗口；
- `AUTH_RATE_LIMIT_WINDOW_SECONDS` / `AUTH_RATE_LIMIT_MAX_REQUESTS`：认证敏感接口限流窗口；
- `RATE_LIMIT_EXEMPT_PATHS`：健康检查和文档等限流豁免路径；
- `AUTH_RATE_LIMIT_PATHS`：使用严格认证限流桶的路径。

注意：应用层限流是单进程兜底能力。生产如果使用多 worker、多实例或反向代理，
必须在网关、Redis 或云安全产品侧配置集中限流，并保证真实客户端 IP 解析可信。

至少应监控：

- 服务是否存活；
- API 健康检查：`/health` 用于 liveness，`/api/v1/health` 用于 readiness；依赖异常时 readiness 返回 503；
- MySQL 可用性；
- MinIO 可用性；
- 磁盘空间；
- CPU 和内存；
- 错误日志数量；
- 备份任务结果。

第一阶段可以先使用 Docker 日志、服务器日志和健康检查。后续再接入集中日志、监控面板和告警工具。

## 权限与审计

运维和发布也属于系统权限的一部分。

生产环境操作应遵守：

- 最小权限原则；
- 管理员账号和普通账号分离；
- 服务器登录账号可追踪；
- 生产数据库不直接给普通开发人员开放写权限；
- 后台关键操作写入审计日志；
- 手工修复数据必须记录原因、操作者和影响范围。

涉及积分、权限、项目审核、借用押金等数据的手工调整，必须优先通过后台管理端或受控脚本执行，不建议直接改数据库。

## 发布检查清单

每次发布前检查：

- 代码已合并到目标分支；
- 构建通过；
- 测试通过；
- Docker 镜像已生成并标记版本；
- 数据库迁移已在 staging 验证；
- MinIO bucket 和权限已确认；
- 环境变量已确认；
- 前端 API 地址正确；
- 管理员账号和权限可用；
- 回滚版本已明确；
- 发布窗口和影响范围已通知相关人员。

发布后检查：

- API 健康检查通过；
- Web 和 Admin 可以访问；
- 登录流程正常；
- 核心业务流程正常；
- 日志无持续错误；
- 数据库连接稳定；
- MinIO 上传和读取正常；
- 版本号或构建信息可查。

## 目录和文件规划

当前已存在：

- `infra/docker/compose.dev.yml`：本地开发 Docker 编排；
- `infra/docker/*.Dockerfile`：本地开发镜像；
- `.env.example`：开发环境变量样例。

后续可以逐步增加：

- `infra/docker/compose.staging.yml`：预发布部署编排；
- `infra/docker/compose.prod.yml`：生产部署编排或生产部署模板；
- `infra/nginx/`：反向代理配置；
- `infra/systemd/`：服务器服务配置；
- `infra/scripts/`：部署、备份、恢复、巡检脚本；
- `apps/docs/docs/ops/`：长期运维手册和发布记录。

如果未来采用 Kubernetes、云托管数据库或对象存储，本文件中的环境原则仍然适用，只是具体落地从 Docker Compose 转为对应平台配置。

## 第一阶段落地建议

第一阶段建议先完成以下运维能力：

- 本地 `compose.dev.yml` 可一键启动；
- 增加 `minio-init` 自动创建开发 bucket；
- 补齐 `.env.example` 中的开发配置；
- 后端接入数据库迁移工具；
- 明确 staging 和 production 的配置变量清单；
- 手工部署 staging 并跑通完整业务链路；
- 编写第一版生产部署说明；
- 上线前建立数据库和对象存储备份流程。

灰度发布、自动 CI/CD、集中监控和蓝绿部署可以在核心业务闭环稳定后逐步引入。
