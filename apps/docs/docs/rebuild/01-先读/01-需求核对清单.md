# MakersHub 重构需求核对清单

## 目标

本文档用于持续核对 MakersHub 的需求边界，避免在项目结构、权限、端侧入口、接口和后续增量规划上做错方向。

本次整理已经把前面多轮讨论中的行内备注、冲突表述和重复问题合并为一份较稳定的需求基线。后续如果继续确认需求，优先更新本文档对应条目，不再把新答案追加到旧问题后面。

## 使用方式

- “已确认需求”可以直接作为架构设计、数据库设计和增量拆分的输入。
- “权限功能编号”用于继续核对权限和端侧入口。
- “后续细化问题”只保留不阻塞第一阶段开工、但实现前还需要补充细节的问题。
- “暂缓需求”表示后续会做，但不放进当前最小重构闭环。

## 一、已确认需求

### 1.1 重构方式

- 采用严格的软件工程增量模型。
- 第一阶段完成口径以端到端可验收业务闭环为准，不以单个后端业务域、接口文件、页面占位或文档表格单独落地为准。
- 后续描述进度时必须区分“文档规划”“后端已实现”“HTTP 接口已开放”“端侧已接入”“可本地演示”“可验收完成”，不能把这些状态混写成“已完成”。
- 不在旧仓库上继续大改，而是新建 `makershub`。
- 后端技术栈暂定继续使用 FastAPI、SQLAlchemy、MySQL、MinIO。
- `makershub` 采用单仓库多应用结构，顶层分为 `apps/`、`packages/`、`infra/` 和 `scripts/`。
- 后端依赖管理采用 `uv + pyproject.toml`。
- 前端采用 `pnpm workspace` 管理多应用和共享包。
- Web 入口应用和后台管理端拆成两个前端应用：`apps/web` 是公开主站首页、成员中心和协会工作台所在的 Web 应用，`apps/admin` 是后台管理端；它们共享 API 客户端、类型、UI 组件、认证机制和权限点，但不是同一个前端应用里简单按权限切换两套页面。
- 前端技术方向采用 React、Vite、TypeScript 和 Ant Design Pro 风格。
- 旧业务数据可以清空重建，不要求迁移历史业务数据。
- 所有项目文档默认使用中文。
- 需要维护仓库内 Markdown 文档。
- 需要规划文档网站，文档网站放在新仓库内部。

### 1.2 业务背景

- 项目服务对象是开源硬件协会。
- 当前协会有四个部门：
  - 宣传部；
  - 基管部；
  - 项目部；
  - 运维部。
- 项目后续会扩展很多能力，包括博客、社区、电控、AI、设备管理、积分制度和更多内部运营功能。
- 因为项目会长期增长，所以项目结构必须按业务域组织，不能只按当前页面或当前旧接口拆文件。

### 1.3 架构原则

- 部门不是代码模块，业务域才是代码模块。
- 身份与权限、组织与成员要分开。
- 积分制度要作为独立账本系统设计，不能只是用户表上的 `score` 字段。
- 资源和借用要分开。
- 内容和社区要分开。
- 项目和工作台要分开。
- 电控必须单独成域。
- AI 不能成为业务事实来源。
- 当前旧后台管理系统在重构初期可以完全删除，后续按新业务重写。
- 重构初期需要保留的是统一身份、权限系统，以及后续后台管理会调用的业务接口。

### 1.4 端侧规划

MakersHub 至少有三类用户入口：

| 入口 | 用途 |
| --- | --- |
| 小程序 | 高频短流程、轻表单、移动场景、简单审核、正常归还、查看和报名 |
| 网页端 | 长表单、材料上传、复杂内容编辑、需要反复修改的申请 |
| 后台管理系统 | 低频管理、复杂列表、批量操作、统计导出、敏感配置、审计追踪、异常处理 |

已确认原则：

- 端侧功能是包含关系，不是交叉关系：小程序用户端是成员网页端自助能力的高频轻量子集；小程序管理端是后台管理端高频轻操作子集；成员网页端覆盖更完整的成员自助操作，后台管理系统覆盖低频管理、复杂批量、敏感配置、异常处理和审计。
- 网页端、小程序端和后台管理端共享 `/api/v1` 业务接口契约、状态机和业务事实，不为小程序单独设计一套借用、积分、资源或审核后端语义。端侧差异主要体现在身份入口、权限点、页面密度、表单拆分、字段摘要和交互适配。
- 认证入口可以不同：小程序使用微信登录和邮箱绑定作为普通用户前置入口；成员网页端使用已绑定邮箱的首次验证码登录、首次设置密码和后续密码登录；登录后必须落到同一个用户主体、会话机制、权限点和业务 API。
- 后台管理系统不是高频日常操作的唯一入口。第一阶段可以先用后台主线审核控制台验证借用审核和归还 API；当接口、权限和状态机稳定后，待审核列表、通过/驳回和正常归还应优先接入小程序管理端，以匹配基管部日常移动使用场景。
- “网页端”在本文档中默认指 `apps/web` 这个 Web 入口应用，包含公开主站首页、成员中心和协会工作台；“后台管理端”指 `apps/admin`。二者都运行在浏览器里，但产品入口、信息架构和页面密度不同，不能把后台审核控制台理解为成员中心或协会工作台里多渲染几个权限按钮。
- `apps/web` 长期产品形态应分成“公开主站首页 / 社区入口”“成员中心”和“协会工作台”三层，而不是进站后直接进入登录框或任务工作台。
- 公开主站首页面向外部用户、游客和新成员，优先承载协会介绍、品牌视觉、公告/活动预览、项目/作品展示预览、社区内容预览、加入说明、小程序绑定邮箱引导、登录/首次登录入口、小程序入口，以及全局业务快捷入口。第一版不提供普通用户网页自助注册；网页端是完整上位入口，小程序后续承接其中的高频轻量子集；旧小程序只能作为现有业务入口和交互参照，不能反过来限定网页端能力。
- 当前已存在的 6 个全局业务入口是：个人借物、团队借物、场地申请、协会活动、项目立项、查看项目。这些入口在旧小程序首页已有对应入口，但在新版 `apps/web` 中应作为“首页业务快捷入口”重新组织；它们不是“我的”成员中心，也不是“协会工作台”。不登录也可以看到入口和公开摘要，但报名、发帖、申请借物、查看个人记录和参与项目等操作需要登录；第一阶段未接入的入口必须明确标记后续能力，不能伪装成可用功能。
- 成员中心面向普通已登录成员，承载个人资料、积分、物资借用、我的申请、我的项目、我的活动和常用自助入口；这些普通成员自助能力不要再统称为“工作台”。
- 协会工作台只面向协会工作人员或被授予对应权限的用户，承载值班、扫除、远程控制、部门工作、任务委派、简单审核等内部运营能力；旧小程序“我的”页里的“协会工作”可以作为入口语义参照，但新版网页端应覆盖更完整的内部工作能力。入口本身应按协会身份或权限点显示，内部模块继续按权限点过滤。
- 工作台不同于后台管理端：工作台是高频内部工作入口，后台管理端承接低频、复杂、敏感、批量、审计和异常处理。
- 第一阶段 `apps/web` 可以做轻量公开主站首页，让系统看起来像 MakersHub 网站；但只做静态或手工维护的协会介绍、品牌入口、登录/首次登录、小程序入口、6 个首页业务快捷入口和少量内容预览，不做完整 CMS、动态活动列表、动态项目展示、官方博客发布流或社区互动闭环。
- 完整公开官网、动态活动公告、项目展示、官方博客和社区能力，等内容域、项目域、社区域及对应审核/发布流程成熟后再增量接入。
- 登录后普通成员进入“成员中心”，不是进入工作台。成员中心优先展示资料状态、积分概览、我的待办、物资借用入口、我的借用记录和常用入口；协会工作台入口只对协会工作人员或有权限用户显示。
- 第一阶段优先实现成员网页端，用网页端验证登录、积分账本和物资借用等当前最小主线业务逻辑；项目域进入 1.5 阶段，不进入当前最小端到端验收主线。
- 第一阶段验收必须围绕真实用户路径串起来：小程序微信登录并绑定邮箱，网页端首次邮箱验证码登录并设置密码，成员网页端完成资料、积分、资源和借用操作，后台管理端完成对应审核或异常处理，最后能在积分流水和审计日志中追溯结果。
- 第一阶段成员网页端不是完整会员门户，而是“端到端验收台 + 可继续产品化的雏形”；它优先支撑当前主线，不追求完整个人中心、活动社区、项目入口或复杂视觉打磨。
- 成员网页端当前必须覆盖：登录、首次设置密码、退出登录、启动态 `/auth/me` 校验与续签、个人资料查看和最小自助更新、积分余额/冻结/近期流水、物资列表/详情、物资借用申请、我的借用记录。
- 成员网页端当前不做：完整个人中心、复杂协会工作台体验、项目立项/材料/结项页面、动态活动社区、通知中心、移动端全面产品化，以及和当前物资借用主线无关的复杂展示型页面。
- 如果某个能力只有后端模型或服务层，没有 HTTP 契约、端侧入口、测试和文档同步，只能标记为“后端能力已具备”，不能标记为“业务闭环完成”。
- 后台管理系统直接重写，不迁移旧后台前端。
- 第一阶段后台管理端先降级为“端到端主线审核控制台”，只服务当前验收主线，不追求完整后台管理系统。
- 第一阶段后台控制台必须优先接入真实登录、权限摘要、按权限过滤的菜单、物资借用审核/归还、必要的物资和成员只读信息、积分流水和审计查看。
- 成员批量管理、资源全量管理、项目审核、统计导出、复杂运营看板和系统配置属于后续完整后台，不应该阻塞第一阶段端到端验收。
- 旧后台的 React 和 Ant Design 方向可以保留，但不继续使用旧的 CRA/react-scripts 工程结构。
- 当前小程序已有的简单审核可以保留在小程序端。
- 大型表单管理、复杂材料提交、批量管理和数据导出更适合网页端或后台管理系统。
- 后台管理系统访问不等于 `998` 管理员身份；业务人员也可以进入自己有权限的后台模块。
- 网页端和后台管理系统第一版不能只依赖微信登录，需要支持账号密码登录。
- 小程序端继续采用微信无感登录，不要求普通小程序用户设置账号密码。
- 小程序端不能只凭本地存在 `auth_token` 判断已登录；必须同时检查令牌过期时间，并在启动或关键接口前通过后端 `/auth/me` 确认令牌仍然有效。
- 访问令牌失效后，小程序应只清理认证相关缓存并自动重新走微信登录流程，不能要求用户手动清空小程序缓存。
- 认证体系采用短期 `access_token` + 长期 `refresh_token` + 服务端 `auth_sessions` 会话表；网页端、后台管理端和后续小程序 API 客户端应统一使用该机制。
- `refresh_token` 必须只保存哈希值，每次续签时轮换，退出登录或后台撤销会话后，原 access token 也不能继续通过 `/auth/me`。
- 所有用户理论上都可以拥有账号密码，但微信端不强制使用账号密码。
- 账号密码登录主要服务网页端、后台管理系统、第一个 `999` 初始化，以及后续需要网页操作的用户。
- 账号密码登录和微信登录应绑定到同一个用户主体上：第一个 `999`、受控后台账号和其他被管理侧明确创建的账号可以先使用邮箱密码；普通用户第一版必须先通过小程序微信登录建立用户主体并绑定邮箱，再到网页端首次登录设置密码。
- 用户主体不要直接等同于微信 `openid` 或 `unionid`；建议内部使用稳定的用户主键，再挂接微信身份、邮箱密码等登录方式。
- 微信 `openid` 是当前小程序下的用户标识，`unionid` 是同一微信开放平台账号下跨应用的用户标识；`unionid` 可能暂时拿不到，不能作为第一阶段启动的硬依赖。
- 当前暂时拿不到稳定的微信 `unionid`，所以第一个 `999` 和网页端登录不能依赖 `unionid` 才能启动。
- 邮箱密码登录名采用邮箱。
- 账号绑定流程以小程序微信无感登录为前置：用户先进入小程序个人主页绑定邮箱，系统通过协会邮箱发送验证码。
- 网页端首次登录接受邮箱验证码，验证通过后强制设置密码，从而把网页端账号和微信小程序用户绑定到同一个用户主体。
- 第一版不考虑先注册网页账号再绑定微信；后续如果有独立 App 再重新评估。
- 第一版不允许解绑微信。
- 用户可以设置、修改密码，也可以通过邮箱验证码重置密码。
- 邮箱验证码有效期为 5 分钟。
- 邮箱验证码同一邮箱 1 小时最多发送 10 次，每次重新请求至少间隔 1 分钟。
- 本地开发阶段允许验证码通过日志模式发送；真实 SMTP 邮箱后续通过环境变量接入，不能把 SMTP 密码提交到仓库。
- 第一版允许用户更换绑定邮箱。
- 更换邮箱需要验证旧邮箱和新邮箱；更换成功后新邮箱作为登录名，原密码保持不变。

### 1.5 身份、职务和权限模型

当前确认的身份和职务编号如下：

| 代号 | 名称 | 类型 | 当前理解 |
| --- | --- | --- | --- |
| 0 | 外部成员、协会会员 | 基础身份 | 已加入协会生态，但不一定是部门干事 |
| 1 | 协会成员、干事 | 基础身份 | 各部门干事 |
| 2 | 部长、部门负责人 | 部门职务 | 各部门负责人，2 及以上属于核心成员 |
| 3 | 副会长 | 协会职务 | 权限高于部长、低于会长；可以和部长兼任 |
| 4 | 会长 | 协会职务 | 协会最高业务负责人 |
| 5 | 指导老师 | 特殊身份 | 通过权限点授权；可以参与所有审核类事项，但不负责运维 |
| 998 | 管理员 | 底层管理身份 | 系统运维、敏感配置、异常修复和受控兜底操作 |
| 999 | 超级管理员 | 底层管理身份 | 母账号；比管理员多“指定管理员”的能力，其余底层能力与管理员一致 |

已确认原则：

- 一般业务权限尽量做包含关系：高一级身份拥有前一级身份的大部分权限。
- `1` 和 `2` 是明显分水岭：`2` 及以上是核心成员，`1` 及以下不是核心成员。
- 副会长参与权限继承，不再仅作为展示标签。
- 副会长每周积分应高于部长，具体数值后续建模。
- 副会长的场地权限通过“场地管理关系”确定：如果副会长被分配或申请到某个场地的管理关系或对应工位，则拥有该场地相关权限。
- 指导老师是特殊身份，不按普通层级继承，但审核类事项权限较高。
- 指导老师可以参与积分人工扣减、积分异常撤回等审核或复核事项；最终落库操作需要结合积分部分的执行边界。
- `998` 和 `999` 确认为绑定到已有用户上的附加系统管理身份，不是独立后台账号。
- `998` 和 `999` 属于底层管理身份，不作为日常业务审批、业务查看、业务导出的默认角色。
- `998` 主要用于运维、系统配置、异常修复、敏感数据处理。
- `999` 主要用于初始化、权限恢复、指定管理员、破窗处理。
- 全系统只能有一个 `999`。
- `999` 平时一般不使用，它的核心工作是指定或恢复 `998`。
- 第一个 `999` 不应从普通业务界面产生，建议通过部署初始化命令或一次性初始化流程产生。
- 第一个 `999` 初始化时应优先创建或绑定邮箱密码登录方式，后续再支持绑定微信身份。
- 灾备重置 `999` 必须要求服务器 sudo 管理员操作。
- 灾备重置 `999` 后，旧 `998` 需要全部失效。
- 灾备重置必须记录执行人、原因、时间和目标账号。
- 代码中不能用 `identity_code >= 1` 这类数字比较直接做接口鉴权。
- 接口应该统一检查权限点，再由身份、职务、部门、作用域规则授予权限点。
- `0` 是外部成员/协会会员的基础身份，不是后台权限点；它用于区分“系统内普通用户”和“干事及以上成员”，不自动带来管理能力。

建议把以下概念分开：

- 账号身份：用户是谁，如何登录。
- 协会身份：是否为协会会员、协会成员、部门成员。
- 部门归属：宣传部、基管部、项目部、运维部。
- 职务身份：部长、副会长、会长、指导老师等。
- 系统权限：能不能执行某个功能。
- 作用域权限：只能管理本部门、某个项目、某个场地，还是全局。

“组织管理权限”说明：

- 它不是新的身份代号，也不是 `998/999`。
- 它是一组可以授予现有用户的权限点，用于维护成员资料、部门归属、职务身份等组织基础数据。
- 已确认可以保留这组权限，由会长授予可信用户。

### 1.6 小程序改造范围

已确认方向：

- 如果新后端不兼容旧接口，小程序至少需要重构 API 层和数据适配层。
- 小程序不一定需要完全重写 UI。
- 小程序源码可以预留在 `apps/miniapp`，微信开发者工具可以直接打开该子目录。
- 小程序接入顺序排在成员网页端验证核心业务逻辑之后。
- 当前最小端到端验收主线中，小程序只负责新版微信登录、令牌校验/续签和邮箱绑定。
- 当前阶段不迁移小程序物资、场地、项目、活动、任务、宣传链接等业务页面；这些页面最后再基于网页端和后台控制台已经验证过的 API 逐步改造。
- 物资借用、资源浏览、积分流水和后台审核等业务闭环当前统一在成员网页端和后台主线审核控制台验证，不要求小程序业务页同步完成。
- 小程序后续可以接回轻量查看、提交和状态跟踪，也可以接入小程序管理端的待审核列表、通过/驳回和正常归还等高频轻操作。
- 小程序不承担物资台账维护、物资类别多层管理、库存调整、资源状态处理、复杂筛选导出和异常归还这类复杂或高风险物资管理；这些能力放在浏览器端，管理类主入口是 `apps/admin`。
- 如果只是统一接口契约和字段，很多页面可以保留 UI，只换 API 调用和数据映射。
- 如果借用、积分、项目、值班、社区等流程重新设计，对应页面需要同步重做。
- 当前仓库已经导入旧小程序源码到 `apps/miniapp`，认证链路和邮箱绑定入口已开始按新版 API 改造。

建议策略：

1. 当前先保证认证链路统一：所有登录态相关请求走 `/api/v1/auth/*`。
2. 当前邮箱绑定页接入新版验证码发送和绑定接口，作为网页端首次登录的前置。
3. 业务页暂时允许继续保留旧调用或不参与验收，但交付说明中必须明确“未迁移”。
4. 等成员网页端和后台控制台验证核心 API 后，再按业务域逐个替换小程序页面数据来源。
5. 能保留 UI 的页面先保留，业务流程变化大的页面再重做。
6. 如果旧小程序必须继续上线运行，再考虑临时兼容层。

### 1.8 积分制度

已确认方向：

- 积分相当于协会内部货币。
- 每个用户都应有积分账户，外部成员 `0` 也有积分账户。
- 积分变动必须有流水。
- 不允许业务代码直接修改余额。
- 人工调整必须写原因。
- 重要积分操作必须进入审计。
- 同一业务事件发放积分必须幂等，不能重复发放。
- 不允许用户之间转账。
- 不允许积分透支。
- 积分永久有效。
- 积分数量是核心。
- 积分等级、称号可以作为后续激励能力规划，但不是第一优先级。
- 每周积分领取和固定积分规则第一版可以先统一用 `10` 作为简单占位值调试，但必须做成可配置规则，不能硬编码在业务代码里。
- 用户提交相关表单时，应该显示积分余额、预计扣除或冻结额度。
- 个人物资借用提交前必须展示预计冻结押金和当前可用积分；可用积分不足时，前端应阻止提交，后端提交接口也必须再次拒绝，避免生成无法审批的待审核申请。
- 借用押金采用冻结余额机制。
- 冻结余额不是扣除积分，而是暂时锁住一部分积分。
- 正常归还后解冻；逾期、损坏或丢失时，冻结积分的一部分或全部可以转为扣除。
- 第一阶段个人物资异常归还采用全额扣除本次申请冻结押金，不支持部分扣除、按损坏数量扣除或管理员手填扣除金额；这些细则进入 1.5 阶段再做。
- 普通固定规则内的积分发放应自动化，例如日常任务固定积分。
- 涉及改动积分规则、创建临时积分规则、发放规则之外的积分、人工补发和人工扣分，都必须进入受控审批或系统兜底流程。
- `998/999` 不作为日常积分规则审批角色，只负责系统级兜底、异常修复和受控人工调整。
- 积分规则审批默认由会长持有；指导老师可以参与复核；副会长可由会长授权参与审批。
- 特殊非模板任务不能直接发布，需要先提交临时积分规则申请；审批通过后生成一次性任务模板，再由任务发布人使用该模板发布任务。
- 临时积分规则至少需要记录任务原因、积分数、适用对象、最大发放人数、总积分上限、有效期、申请人、审批人和审批理由。
- 积分规则、临时积分规则、一次性任务模板的创建、审批、发布、撤回都必须保留日志。
- 已发布的积分规则或临时积分规则允许撤回；撤回不能删除原记录，只能标记撤回并记录撤回原因、撤回人、撤回时间和影响范围。
- 临时积分规则撤回不等于自动追回已经发放的积分；默认含义是停止该规则后续继续使用。
- 只有规则错误、重复发放、恶意滥用或审批失误等明确异常场景，才允许追回已经发放的积分。
- 如果确实需要追回已经产生的积分流水，必须通过反向流水修正，不能直接删除原流水。
- 临时积分规则撤回以及由撤回引发的任务取消、待处理、积分修正，都需要通知受影响用户。
- 后台人工发放积分前需要开会讨论，量化贡献后再发放。
- 违规扣分不需要二次确认，但需要允许撤回。撤回应该通过反向流水修正，而不是删除原流水。
- 违规扣分撤回的最终执行由 `998/999` 处理；指导老师可以参与积分异常的审核或复核。
- 积分获取和消耗需要后续进行数学建模，避免滥用和积分通货膨胀。

已确认的积分来源或消耗方向：

- 每周领取积分，按身份或职务区分。
- 打扫卫生或大扫除可以获得积分。
- 日常任务完成可以获得固定积分。
- 宣传链接通过、审核、审批等贡献可以获得积分。
- 审批失误、资源丢失、逾期未归还、损坏物资、恶意占用工位等可以扣积分。
- 借物、场地或工位押金通过借用流程处理，不属于“积分兑换审核”。
- 3D 打印额度等可以作为积分消费场景。
- 3D 打印积分消费第一版不需要库存、次数或额度限制，但需要支持“维护中/关闭消费”开关。
- 3D 打印、服务器借用等服务型资源主要由运维部维护和操作，对应的服务开关也应授予运维部相关权限。
- 3D 打印和服务器借用第一阶段保留业务域和接口占位，但不作为核心闭环优先实现。
- 服务型资源更像“服务订单/接单执行”，不是普通业务审批。
- 3D 打印流程倾向于：用户提交切片文件，系统按配置核算积分，运维负责人接单后打印，完成后通知申请人领取。
- 3D 打印字段后续再和运维部对接；当前不在核心重构闭环中强行定死。
- 服务器借用流程倾向于：按租用服务扣积分，可按时长或周期计费。
- 服务器借用申请字段后续再和运维部对接；当前不在核心重构闭环中强行定死。
- 3D 打印和服务器借用价格应由系统核算，用户提交时就能看到预计消耗，不需要“待计价”“待确认”状态。
- 3D 打印接单后冻结积分，打印完成后扣除积分。
- 3D 打印取消后释放冻结积分；未接单时用户可以取消，接单后需要联系运维处理人取消。
- 服务器借用按时间或周期扣积分。
- 服务器借用需要到期前提醒。
- 服务器借用没有传统逾期概念，可以保留 3 天宽限期；宽限期结束后直接释放对应实例。

### 1.9 场地和工位

当前工位类型调整为四类：

| 类型 | 积分变化 | 是否允许放个人物品 | 当前理解 |
| --- | --- | --- | --- |
| 流动工位 | 增加积分或额度 | 不允许 | 可通过 NFC 或后续视觉方案记录使用和值班情况 |
| 流动固定工位 | 扣积分 | 允许 | 包含部长、会长等固定使用诉求；促进成员在三个场地之间流转 |
| 管理工位 | 不扣积分，不加积分 | 允许 | 每个场地仅一个；只有会长或副会长可以申请；不是自动分配 |
| 项目工位 | 不直接影响积分 | 允许 | 通过项目或工位申请获得，需要关联项目并签订开源协议 |

已确认方向：

- 工位不是普通场地资源，需要独立字段描述类型、积分规则、占用规则、是否可存放个人物品、是否需要冷却期。
- 场地借用后续基本会收敛为工位申请。
- 不同工位类型需要不同字段，例如项目工位需要项目编号，管理工位不需要项目编号。
- 流动工位不需要用户填写申请；它更接近值班或到场使用场景，后续通过 NFC、视觉检测等方式记录使用并结算积分。
- 流动固定工位冷却期暂定 3 个月。
- 流动固定工位一次最长申请 3 个月，冷却期与申请时长对等。
- 管理工位按会长或副会长职务长期有效，不单独设置固定到期时间；职务变更后自动取消或到期。
- 项目工位周期与项目周期绑定。
- 项目工位原则上不允许随意换场地或换工位，但需要保留受控换位能力。
- 项目工位换位由项目负责人发起，需要重新审核。
- 项目工位换位后原工位需要立即释放。
- 项目工位换位原因必须记录。
- 当前场地名称可预填为 `101`、`i创街`、`208`，但场地可能变化，只能作为初始数据。
- 基地/场地/工位管理属于基管部，不属于运维部。
- 硬件相关能力暂时不考虑，仅保留业务域入口。
- NFC、WiFi 在线时长、视觉监控等打卡或检测能力放到后续增量。

### 1.10 资源和物资

已确认方向：

- 物资列表和物资详情第一阶段先在成员网页端验证；后续可以给小程序做轻量查看入口。
- 物资详情建议展示名称、分类、剩余数量、描述、位置、柜号、层数、可借状态、维护/下架状态、借用规则提示。
- 成员端物资列表默认只展示 `status=available` 的物资，不把维修中、停用或下架物资混入可借入口；后台物资台账可以按状态筛选查看全部物资。
- 资源读取必须由后端按权限兜底：没有 `resources.material.manage` 的普通成员只能进入可借浏览视角，列表默认且强制限制为 `status=available`；即使手动传 `status=maintenance`、`disabled` 或 `retired`，也不能返回管理台账数据。
- 物资详情也必须按权限区分：普通成员只能读取可借物资详情，拥有 `resources.material.manage` 的用户才可以读取维修中、停用或下架物资详情；普通成员直接访问不存在物资或非 `available` 物资详情时，都返回 `404 MATERIAL_NOT_FOUND`，按“不可见即不存在”处理。
- 端侧展示的“可借”由物资状态和当前可借数量共同决定：`status=available` 但 `available_quantity=0` 时可以只读展示库存不足，但不能允许提交或修改为借用明细。
- 物资新增、编辑、库存调整属于基管部日常管理，主入口放在后台管理端 `apps/admin`，不放在小程序端。
- 物资类别删除或整类下架风险更高，应限制为基管部 `2` 及以上。
- “调整物资库存”指修改某种物资的数量，用于补货、盘点、报废、丢失修正；它不是删除物资类别。
- “维修中、停用、下架”等物资状态可以并入物资编辑或下架功能，不必强行拆成复杂模块。
- 资源操作日志当前范围指场地和物资的新增、修改、数量调整、下架、删除等记录。
- 工程权限上物资台账维护和物资借用审核要拆成两个权限点：`resources.material.manage` 和 `borrowing.application.review`。
- 业务常态下这两个能力都由基管部负责，第一阶段可以用预置 `resource_manager` 角色打包授予基管部成员；代码和后台菜单仍然必须按权限点判断，不能硬编码“基管部成员一定拥有所有资源和借用能力”。

### 1.11 借用和预约

已确认方向：

- 物资借用和场地/工位借用的数据格式不同，不能粗暴合成一张完全相同的业务表。
- 可以统一生命周期和通用接口概念，例如申请、审批、取消、归还、逾期、押金、审计。
- 物资借用取消需要保留取消记录。
- 借用需要积分押金或信用分。
- 取消记录至少需要保存取消人、取消时间和取消前状态；取消原因是选填字段，用户未填写时不能阻塞取消，也不能伪造默认原因。
- 申请人可以取消待审核或已驳回的个人物资借用申请；取消已驳回申请只是用户侧收尾，不删除审核记录，也不撤销驳回结论。
- 旧小程序物资借用详情页提供“修改申请”，会跳转到申请页并携带 `edit=true&sb_id=...`，申请页回填原申请后调用旧 `/stuff-borrow/update/{sb_id}`；新版第一阶段应保留“修改原申请”的业务语义，而不是要求用户取消后新建。
- 旧小程序个人和团队物资申请都强制选择归还日期；新版第一阶段个人物资借用也必须要求 `expected_return_at`，前端提示不能代替后端校验。
- 申请人可以修改 `pending_review` 或 `rejected` 的个人物资借用申请，修改范围包括借用理由、预计归还时间和物资明细；修改已驳回申请后应回到 `pending_review` 等待重新审核。
- 修改原申请必须保留原申请 ID、原审核记录和操作审计；如果物资明细变化，可以在服务层替换明细快照，但不能删除旧审核结论。
- 提交和修改申请都必须由后端即时校验当前物资台账：物资必须存在、状态可借、借用数量为正、当前可借数量足够、押金积分足够；当前可借数量不足时直接拒绝提交或修改，不创建或更新申请。
- 提交或修改时当前可借数量不足必须返回稳定错误码 `BORROW_MATERIAL_STOCK_NOT_ENOUGH`，HTTP 状态为 `409`，`details` 至少包含 `material_id`、`material_name`、`requested_quantity` 和 `available_quantity`。
- 修改申请按“基于当前物资台账重新提交原申请”处理，后端必须重新校验申请人身份、申请状态、物资可借状态、借用数量、当前可借数量、当前押金和可用积分；前端可以提前提示或禁用按钮，但不能成为唯一判断边界。
- 修改申请不是局部补丁，而是完整提交可修改字段：`reason`、`expected_return_at` 和整份 `items` 明细都必须提交；后端不接受只增减某一条物资的增量语义，也不允许修改 `borrow_type`、`usage_type`、`project_id`、`applicant_id`、状态或押金字段。
- 修改已驳回申请后，当前主状态展示为待重新审核，旧驳回意见不能继续作为当前审核结果展示；成员端和后台详情页可以用“历史审核意见”折叠区展示 `borrow_reviews` 中的旧意见。
- 修改申请成功后必须刷新 `submitted_at`，语义是“最新提交/重新提交时间”；`created_at` 保留原始创建时间，后台待审核队列按 `submitted_at desc, id desc` 排序。
- 借用申请必须保存申请人信息快照，至少包含姓名、学号、手机号、联系邮箱、年级和专业；历史详情、后台审核和归还处理展示快照，不能被后续个人资料修改污染。
- 申请人信息快照来源于成员资料，不来自借用申请表自由填写；借用申请页只填写借用理由、预计归还时间和物资明细等业务字段。联系邮箱默认取登录邮箱，成员可以在资料中修改，修改联系邮箱不等同于更换登录邮箱。联系邮箱必须由后端做邮箱格式校验，但不需要验证码确认，因为它不是登录凭证。提交或修改借用申请时，后端优先使用成员资料里的联系邮箱；联系邮箱为空但登录邮箱存在时回退使用登录邮箱；姓名、学号、手机号、年级或专业缺失，或联系邮箱和登录邮箱均缺失时，后端拒绝提交或修改，端侧引导用户先完善资料。
- 后台审核列表指后台管理端/后台主线审核控制台，不是成员中心页面。
- 借用列表范围必须由后端兜底：普通成员即使不传 `mine=true`，也只能看到自己的申请；只有拥有 `borrowing.application.review` 权限的用户才能查看全量申请列表；拥有该权限的用户显式传 `mine=true` 时仍应切回本人视角。
- 借用列表字段也必须由后端兜底：成员本人列表和后台全量列表都只返回摘要，不返回学号、手机号、联系邮箱或 `applicant_current_contact`，不能依赖前端隐藏敏感字段。
- 普通成员查看本人借用详情时可以看到 `applicant_snapshot` 中的姓名、学号、手机号、联系邮箱、年级和专业，端侧必须标注为“申请时资料”；但不返回 `applicant_current_contact`。拥有 `borrowing.application.review` 权限的审核/归还人员查看详情时，默认返回 `applicant_current_contact` 作为人工联系辅助信息，不需要额外传开关参数。
- 后台审核列表沿用旧小程序摘要边界，但不要继续使用旧小程序 `年级-专业-姓名` 的拼接字符串，应拆成独立列：姓名、年级、专业、提交时间、状态、类型、物资摘要和操作；申请编号、学号、手机号、联系邮箱以及当前联系方式放详情页，不作为后台列表主列。
- 成员端“我的借用记录”列表也只展示摘要，但它是本人视角，不套用后台审核表格列；优先展示状态、提交时间、预计归还时间、物资摘要和可执行操作，学号、手机号、联系邮箱以及当前联系方式只进入详情页。
- 列表里的物资摘要使用借用明细快照生成：单项显示“物资名称 x 数量”；多项显示“首件物资名称 x 数量 等 N 项”，其中 `N` 是本次申请的明细项总数。完整物资清单、分类、单件押金和每项数量只在详情页展示。
- 后台审核、归还和历史详情页的主申请人信息必须展示申请时快照；如果申请人后来修改了手机号或联系邮箱，后台详情页可以额外只读展示“当前成员资料中的联系方式”，并明确标注来源。当前联系方式只用于人工联系，不用于改写申请快照、审核事实、归还记录或审计日志。
- 借用明细必须保存申请时快照，至少包含物资名称、分类名称和单件押金积分；历史详情、审核和归还展示使用快照，库存扣减和归还仍通过 `material_id` 操作当前物资台账。
- 成员端“我的借用记录”必须展示本人全部物资借用状态，包括 `pending_review`、`approved`、`rejected`、`cancelled`、`returned` 和 `exception_closed`，不能只展示待处理或可操作记录。
- 成员端只允许对 `pending_review` 和 `rejected` 申请执行修改或取消；`rejected` 的取消只是关闭用户侧待处理项，不代表重新提交或撤销驳回。
- 成员端不得提供“归还”“异常归还”“确认归还”按钮；`approved` 只展示“待后台处理归还”，`returned`、`exception_closed` 和 `cancelled` 均为只读结果。
- 物资借用和场地/工位借用都属于基管部业务。
- 物资借用审核入口依赖 `borrowing.application.review` 权限点；第一阶段基管部常态角色会同时包含物资台账维护和借用审核，但后续如果出现只审核、不维护台账的分工，应通过角色授权调整，不改业务状态机。
- 物资借用审核通过时必须再次校验押金可用积分和当前可借库存，因为申请等待审核期间用户积分或库存可能变化；如果审核时积分或库存不足，申请自动驳回并写入审核记录，不扣库存、不冻结押金；库存不足的系统审核意见必须写明物资名称、当前可借数量和申请数量。
- 人工驳回物资借用申请必须填写驳回理由；审核通过意见选填。系统因库存或积分不足自动驳回时，由后端写入明确系统原因。
- 第一阶段借用审核和归还的幂等口径是“防重复副作用”：重复审核不能重复扣库存或重复冻结押金，重复归还不能重复恢复库存或重复解冻/扣除押金；状态已流转后的重复请求可以返回 `409`，不要求返回第一次成功结果。
- 后台借用审核工作台默认是待办视角，只展示 `pending_review`，并按 `submitted_at desc, id desc` 排序；全部、已通过待归还、已驳回、已取消、已归还和异常关闭通过状态筛选查看。
- 后台审核动作只在 `pending_review` 申请上展示，正常归还和异常归还动作只在 `approved` 申请上展示，历史状态默认不混进待办列表。
- 查看全部借用记录应限制为 `2` 及以上。
- 物资借用审核可以由基管部 `1` 及以上处理。
- 场地/工位借用审核建议由基管部 `2` 及以上处理。
- 普通归还可以轻量处理；涉及损坏、丢失、扣分、异常记录时应进入后台管理系统。
- 确认归还不是普通借用人的自助动作，不暴露在第一阶段成员网页端；第一阶段先由后台主线审核控制台验证接口和状态机，后续应暴露给小程序管理端的基管部处理人。
- 第一阶段后台界面必须拆成“正常归还”和“异常归还”两个动作：正常归还给基管部 `1` 及以上处理，异常归还给基管部 `2` 及以上、会长或对应业务负责人处理。
- 第一阶段后端暂不新增 `borrowing.application.return_normal`、`borrowing.application.return_exception` 这类细分权限点，审核、正常归还和异常归还暂时共用 `borrowing.application.review`；这是为了先闭合端到端主线，不代表异常归还和正常归还业务风险相同。
- 当前后端归还接口可以继续使用同一个 `/borrowing/applications/{application_id}/return`，通过 `condition` 区分正常或异常；但端侧不能只做一个含糊的“确认归还”按钮。
- 后台正常归还点击后可以使用轻量二次确认弹窗，弹窗必须展示“将恢复可借库存并解冻本次冻结押金”，备注输入可选。
- 后台异常归还点击后必须使用高风险二次确认弹窗，弹窗必须展示“将全额扣除本次冻结押金，且不自动恢复可借库存”，并要求填写损坏、丢失或消耗原因。
- 后续当组织/职务/作用域授权稳定后，应把异常归还拆成独立高风险权限点，并按基管部 `2` 及以上、会长或对应业务负责人授权。
- 第一阶段异常归还包含损坏、丢失和消耗类情况，统一进入 `exception_closed`，全额扣除本次申请冻结押金，不自动恢复可借库存。
- 异常归还必须填写归还备注，说明损坏、丢失或消耗原因；正常归还备注可选。
- 第一阶段异常归还确认后是终态，不提供普通撤销或重新归还流程；误操作通过受控库存调整、积分反向修正和审计记录兜底，专门的异常归还撤销流程进入 1.5 阶段。
- 借用规则配置由会长 `4` 和基管部 `2` 及以上处理。
- 通过项目申请物资或工位时，可以走项目相关减免规则，不直接扣普通个人积分。
- 项目借用属于 1.5 阶段能力；第一阶段物资借用只要求个人借用闭环，不能因为项目借用未实现而阻塞当前主线验收。
- 项目相关的物资借用或工位申请需要关联项目，并要求负责人签署开源协议。
- 项目借用必须关联已经通过项目申请并获得项目编号的项目。
- 项目借用字段应尽量从项目和个人资料自动带出，避免申请人手填造成不一致。
- 个人借用不需要填写指导老师信息；项目借用的指导老师信息应从项目资料自动带出。
- 项目借用完全免扣个人积分，不额外占用个人积分。
- 项目负责人完成开源协议签署并通过项目申请后，后续该项目借用可基于项目编号走项目借用路径。
- 项目借用主要靠开源协议约束，一般问题不做积分惩罚。
- 违反开源协议或其他重要规定时，不只是项目拉黑，项目负责人和成员也应一起进入限制或封禁状态，不允许再向协会提起申请。
- 重大破坏或重要违规由拥有项目风控处理权限的人员结合实际情况手动判断；该权限只授予项目部部长及以上权限，需要系统级封禁时由 `998/999` 兜底执行。
- 项目、负责人和成员的限制或封禁允许申诉。
- 封禁或限制解除权限限制为会长及以上。
- 项目风控记录需要保存原因、证据、处理人、处理时间、影响范围和解除条件。
- 项目申诉记录需要保存申诉说明、附件、处理意见、处理人和处理时间。
- 项目中断记录需要保存中断原因、剩余资源处理、恢复条件和审核意见。
- 项目中断后，无法收回或已经损耗的借出物资可以由基管部部长及以上标记为已损耗，并从库存中扣除。

需要说明：

- “记录逾期事实”不是立即扣分，而是系统保存“是否逾期、逾期多久、实际归还时间”等事实数据。
- 扣不扣积分、扣多少积分，可以后续由积分模型决定。
- 第一版可以保留逾期扣分公式的配置占位，但默认不启用具体扣分公式。

### 1.12 项目

当前阶段说明：

- 项目域已经保留业务边界、需求草案、旧字段对照和后续接口方向。
- 项目端到端实现降级到 1.5 阶段，不进入当前最小端到端验收主线。
- 第一阶段不要求实现项目立项、项目审核、项目材料上传、结项和项目风控页面。
- 第一阶段涉及项目的地方只需要保留边界约束，例如项目工位审核不在项目域重复建流程、项目借用暂不开放。

已确认方向：

- 所有人都可以创建项目申请，项目申请中应加入开源协议签署相关内容。
- 项目创建后进入待审核，后续可以显式拆分草稿和提交审核。
- 项目结项需要审核。
- 项目结项可以被打回。
- 项目结项只需要一个有权限的审核人审核通过即可，不需要多方会签。
- 项目招募需要申请加入，但可以先作为后续待办。
- 当前代码已有 `is_recruiting` 类似招募状态，数据模型中应保留招募状态。
- 项目成员第一版只保留“负责人”和“成员”，不扩展维护者、硬件负责人等复杂角色。
- 项目负责人不允许随意变更；如果确实发生特殊情况，需要作为异常流程另行处理。
- 项目类型第一版保留个人项目和比赛项目，用于区分兴趣项目和比赛项目；协会项目暂缓。
- 项目完成是否奖励积分后续讨论。
- 项目团队需要每 3 个月提交一次项目进展。
- 项目进展至少需要填写一种材料，例如文字、图片、文件、链接或视频。
- 项目进展需要审核。
- 项目进展到期前 7 天开始提醒。
- 项目进展逾期 1 周再次提醒，逾期 2 周严重提醒，逾期 3 周作为最后期限。
- 项目进展逾期 1 个月后直接进入黑名单或限制名单。
- 项目进展逾期不扣积分，直接按提醒、限制、拉黑处理。
- 项目进展黑名单或限制名单只能由会长及以上权限解除。
- 项目进展不对外展示，但项目结项后需要对外展示。
- 项目材料需要长期归档，所有材料默认永久保存。
- 项目材料上传权限应限制为项目负责人。
- 材料较多、结项总结较长的场景，建议网页端提供更完整体验，小程序保留轻量上传。
- 项目工位申请只需要支持上传材料和展示文字提示，不需要系统固定材料类型。
- 项目工位申请应在工位申请中选择项目编号，从而和项目关联。
- 项目工位审核不应在项目域重复建一套流程，应归入工位/场地借用审核。
- 项目公开范围、仓库地址、文档地址、展示链接更适合在项目结项时提交，不放在第一版立项申请里。
- 项目立项不需要新增“预期成果”字段。
- 开源协议由协会拟定，项目负责人需要下载、签字后上传签署文件。
- 开源协议签署文件限制为 PDF。
- 开源协议签署文件和协议版本都需要保存。
- 项目需要有“中断”状态，由项目部人员负责管理。
- 负责人毕业、退会或失联等异常情况，除特殊原因外按照项目进展逾期规则处理。
- 项目中断后，项目工位可以直接释放。
- 项目中断后，已经借出的资源需要按资源类型和实际情况处理；项目工位直接释放，无法收回或已经损耗的借出物资可以由基管部部长及以上标记为已损耗，并从库存中扣除。
- 项目中断状态允许恢复。
- 项目中断需要提交说明并经过审核。

### 1.13 工作台、任务和值班

已确认方向：

- 所有部门都需要排班。
- 排班基本单位是任务。
- 所有部门都支持自由发布任务。
- `2` 及以上可以发布任务。
- 发布指定任务时，原则上只能高权限给低权限发布，不能平级发、给自己发、向上发。
- 会长和指导老师的发任务权限不受上述上下级限制。
- 指定任务不需要接收，发布后直接安排，并通过微信服务通知提醒。
- 指定任务发布后直接进入“待完成”，不需要单独保留“已发布”状态。
- 悬赏任务可以不指定人，需要“待领取”状态；被领取后直接进入“待完成”，不需要单独保留“已领取”状态。
- 悬赏任务需要区分协会内可见和公开可见；`0` 外部成员只能领取公开任务。
- 任务完成流程应拆成“执行人提交完成申请或材料”和“任务发布人审核完成”。
- “待审核”表示执行人已经提交完成材料，等待任务发布人审核；它比“待确认”更清晰。
- 最终标记任务完成的人应是任务发布人。
- 宣传部的文案、推文、新闻稿是便捷任务类型；不是这三类时使用自由任务。
- 宣传链接即秀米链接，通常由宣传部长派任务给干事做，再由部长审核。
- 值班申请是有人发布值班消息，然后成员报名。
- 值班消息需要和场地关联，不需要绑定部门。
- 值班消息可以选择多个场地。
- 多场地值班的人数上限由各场地的流动工位数量决定。
- 成员报名多场地值班时需要选择具体场地。
- 同一个值班消息下，一个成员只能报名一个场地。
- 值班报名后允许退出。
- 值班人数满员后自动关闭报名。
- 值班记录正常业务中不开放随意增删改查，只有错误修正时才由 `998/999` 介入。
- 打扫卫生或大扫除属于统一任务的一种任务类型，不单独建立清洁业务域。
- 悬赏任务发布人可以选择公开范围。
- 允许公开给 `0` 外部成员的任务类型后续由积分规则审批权限在模板配置中维护。
- 外部成员完成公开悬赏任务后可以获得积分，同时保留完成记录。
- 部门工作应在部门内入口发布，包括部门内悬赏工作和指定工作。
- 能公开的工作应在外部入口发布，属于公开悬赏工作。
- 任务模板按当前理解分为三类：部门指定任务、协会指定任务或协会悬赏任务、对外悬赏任务。
- 悬赏任务类型和积分条目只能由积分规则审批权限维护，不由普通任务发布人临时修改。
- `998/999` 只负责系统级兜底，不作为日常任务模板和积分规则审批角色。
- 悬赏任务类型和积分条目修改后，只对新任务生效，不影响已经发布的任务。
- 发布任务时必须完全使用模板积分，不能在模板基础上临时改积分。
- 特殊非模板任务不能直接发布，需要先提交临时积分规则申请；审批通过后生成一次性任务模板。
- 临时积分规则和一次性任务模板必须保留审批日志，并允许撤回；异常需要追回积分时通过反向流水修正。
- 临时积分规则撤回默认只停止后续使用，不自动追回已经发放的积分。
- 只有规则错误、重复发放、恶意滥用或审批失误等明确异常场景，才允许通过反向流水追回积分。
- 临时积分规则撤回后，未领取任务直接关闭；已领取未提交任务通知执行人并取消；待审核任务进入“规则撤回待处理”；已完成且已发放积分的任务默认保留积分和完成记录。

任务状态建议：

| 任务类型 | 状态流转 |
| --- | --- |
| 指定任务 | 待完成 -> 待审核 -> 已完成；异常分支为已打回、已取消 |
| 悬赏任务 | 待领取 -> 待完成 -> 待审核 -> 已完成；异常分支为已打回、已取消 |

说明：

- “已发布”更适合作为操作记录或发布时间，不适合作为指定任务的业务状态。
- “已领取”也更适合作为操作记录；领取完成后，任务对执行人来说就是“待完成”。

### 1.15 文档和运维

已确认方向：

- 文档网站放在新仓库内部。
- 文档站可以采用 VitePress。

- 需要故障记录模板。

VitePress 说明：

- VitePress 是一个基于 Markdown 的文档网站生成工具。
- 写文档时主要还是写 `.md` 文件。
- 它可以把 Markdown 自动生成一个静态文档网站。
- 它适合放在仓库内部，和代码一起版本管理。
- 它支持导航、侧边栏、全文搜索、代码高亮和静态部署。
- 对当前项目来说，它适合作为开发文档、接口说明、运维手册、发布记录、故障记录的文档站底座。

运维环境说明：

## 四、后续细化问题

本节只保留不阻塞第一阶段开工、但实现前还需要补充细节的问题。已经确认的回答已经移动到前面的“已确认需求”和“权限功能编号”。

### 4.1 服务型资源

Q001：服务订单字段后续再和运维部继续对接。

当前已确认：

- 3D 打印和服务器借用第一阶段只保留业务域和接口占位。
- 这部分不如核心的借物、借场地和项目重要，不作为当前核心重构闭环。
- 3D 打印和服务器借用价格由系统核算，用户提交时就能看到预计积分。
- 不需要“待计价”“待确认”状态。
- 服务订单状态采用待接单、执行中、待领取或使用中、已完成、已取消。
- 3D 打印接单后冻结积分，打印完成后扣除积分。
- 3D 打印取消后释放冻结积分；未接单时用户可以取消，接单后需要联系运维处理人取消。
- 服务器借用按时间或周期扣积分。
- 服务器借用需要到期前提醒，没有传统逾期概念，允许拖 3 天，3 天后释放对应实例。

后续需要继续确认：

- 3D 打印是否还需要材料类型、颜色、备注等字段，以及这些字段是否由用户填写还是运维接单时补充。
- 服务器借用需要哪些配置字段，例如用途、规格、系统镜像、周期、访问方式；这部分暂时留空，等运维部确认。

### 4.2 临时积分规则

Q002：临时积分规则的表单和撤回处理规则。

当前已确认：

- 任务模板按当前理解分为三类：部门指定任务、协会指定任务或协会悬赏任务、对外悬赏任务。
- 普通任务必须完全使用模板积分，发布人不能临时改积分。
- 特殊非模板任务不能直接发布，需要先提交临时积分规则申请。
- 临时积分规则审批通过后生成一次性任务模板。
- 临时积分规则默认由会长审批；指导老师可以复核；副会长可以由会长授权；`998/999` 只做系统兜底。
- 临时积分规则必须保留审批、发布、撤回日志。
- 临时积分规则允许撤回；撤回默认停止后续使用，不自动追回已经发放的积分。
- 临时积分规则撤回需要通知受影响用户。
- 只有规则错误、重复发放、恶意滥用或审批失误等明确异常场景，才允许追回已经发放的积分。
- 如果确实需要追回已经产生的积分流水，必须通过反向流水修正，不能删除原流水。

建议申请表单字段：

- 基础信息：规则名称、任务类型、所属部门或协会级别、申请人、申请原因、为什么不能使用现有模板。
- 积分信息：单次完成积分、最大发放人数、总积分上限、有效期。
- 任务要求：适用对象、完成条件、需要提交的材料、审核人或审核范围。
- 风控信息：是否公开给外部成员、是否允许多人完成、是否允许重复领取、撤回影响说明。

撤回后的任务处理建议：

| 任务状态 | 处理方式 |
| --- | --- |
| 未领取 | 关闭任务，展示为“规则已撤回，任务已关闭”，不产生积分 |
| 已领取但未提交 | 通知执行人，任务标记为“规则撤回取消”，不产生积分；如工作已经实际发生，发布人可重新申请临时积分规则补偿 |
| 已提交待审核 | 进入“规则撤回待处理”，暂停审核和发分；由审核人判断取消、打回，或重新申请临时积分规则 |
| 已完成但未发分 | 暂停发分，进入“规则撤回待处理”；确认任务仍有效后可用新规则结算 |
| 已完成且已发分 | 默认保留积分和完成记录；只有规则错误、重复发放、恶意滥用或审批失误等异常场景，才通过反向流水追回积分 |
