# MakersHub 第一阶段实施路线图

## 目标

第一阶段目标是交付一组可以运行、可以验收、可以继续扩展的核心业务闭环。

第一阶段不追求把所有规划功能一次性做完，而是先把系统底座和高频主流程做稳：身份、组织、权限、积分账本、物资资源、物资借用、成员网页端和后台主线审核控制台。项目域降级到 1.5 阶段，当前只保留业务边界和设计草案。

## 第一阶段完成定义

第一阶段的“完成”以端到端可验收为准，而不是以后端文件数量、接口数量或页面数量为准。

最小端到端验收主线如下：

1. 普通用户从小程序进入，通过微信无感登录建立内部用户主体。
2. 用户在小程序个人主页绑定邮箱，后端发送并校验协会邮箱验证码。
3. 用户在成员网页端使用已绑定邮箱进行首次验证码登录，并被强制设置密码。
4. 用户在成员网页端查看和维护个人资料，查看积分余额、冻结积分和积分流水。
5. 用户在成员网页端浏览物资资源并提交物资借用申请。
6. 有对应权限的后台用户在后台管理端查看借用申请，完成审核、归还或异常处理；这是第一阶段验证入口，不代表后台管理端是后续高频操作唯一入口。
7. 借用申请提交前展示预计押金并校验可用积分；审核通过时再次校验可用积分，扣减可借库存并冻结押金，正常归还时恢复库存并解冻押金，异常归还时全额扣除本次申请押金。
8. 积分流水、借用状态和审计日志能共同追溯“谁在什么时间对什么业务对象做了什么”。

只有同时满足以下条件，某条能力才能标记为“可验收完成”：

- 数据库模型和迁移已落地；
- 服务层实现了完整业务状态流转；
- HTTP 接口开放，并且使用统一认证、权限点和错误响应；
- 成员端或后台端至少有一个真实入口接入，不是静态占位或 mock 数据；
- 核心路径有自动化测试覆盖；
- 对应需求、API、数据库或运维文档已同步更新；
- 本地 lint/test 通过；涉及文档时文档构建通过。

进度描述必须使用下面的状态词，避免把半成品误写成完成：

| 状态 | 含义 |
| --- | --- |
| 文档规划 | 只有需求、设计或 README，没有可运行代码 |
| 后端能力已具备 | 模型、服务或测试已落地，但缺少 HTTP、端侧或完整联调 |
| HTTP 已开放 | 接口可调用，但端侧还没有真实入口 |
| 端侧已接入 | 页面或小程序已调用新接口，但还没有串通完整验收主线 |
| 可本地演示 | 本地环境可以手动走通主流程，但测试、文档或异常分支仍可能缺口 |
| 可验收完成 | 满足端到端验收主线和上面的完成条件 |

## 第一阶段范围

必须交付：

- 身份与登录：邮箱账号、密码、微信身份挂接、会话、唯一 `999` 初始化；
- 组织与成员：部门、职务、成员资料、成员状态、花名册；
- 权限系统：权限点、角色或授权关系、作用域权限、后台访问权限；
- 积分账本：积分账户、可用余额、冻结余额、流水、幂等业务事件；
- 资源基础：第一阶段完成物资分类、物资台账、库存和资源状态；场地/工位进入后续增量；
- 借用基础：第一阶段完成个人物资借用、修改申请、审批、取消、归还和押金冻结；场地/工位申请和项目借用进入后续增量；
- 项目边界：保留业务域、需求草案、旧字段对照和接口方向，端到端实现进入 1.5 阶段；
- 成员网页端：用网页端跑通登录、资料、资源浏览、物资借用申请、积分流水；
- 后台管理端：第一阶段作为端到端主线审核控制台，只覆盖真实登录、权限菜单、物资借用审核/归还、必要只读信息、积分流水和审计查看；
- 成员网页端和后台管理端是两个前端应用，分别对应 `apps/web` 和 `apps/admin`；它们共享同一套认证、权限点和业务 API，但不是同一个前端根据权限切换页面；
- 共享 API 客户端：第一阶段必须把新版响应解析、业务错误、令牌刷新、`/auth/me`、`/permissions/me` 和核心业务客户端沉淀到 `packages/api-client`；网页端和后台端复用同一套 `fetch` 适配，小程序后续通过 `wx.request` 适配共享同一套契约；
- 后台管理端第一阶段用于把审核、归还、异常处理、积分流水和审计追溯闭合成可验收主线；后续高频轻操作不应长期被后台端独占，待审核列表、通过/驳回和正常归还可以迁回小程序管理端；
- 文档站：持续维护架构、接口、数据库、开发和运维文档。
- Docker 本地环境：至少编排 MySQL、MinIO、后端服务、成员网页端、后台管理端和文档站。

暂不深入：

- 3D 打印和服务器借用具体字段；
- 项目立项、审核、材料上传、结项和项目风控的端到端实现；
- 完整项目招募流程；
- 项目进展自动提醒和自动拉黑执行；
- 博客、社区、电控、AI；
- NFC、WiFi 在线时长、视觉检测等硬件联动；
- 完整积分经济模型、积分等级和称号。

## AI 协作执行口径

后续 AI 或人工继续开发时，必须按端到端主线推进，不要只在某个业务域内堆代码。

每次接任务前先判断当前工作落在哪个状态词上，并在文档或交付说明中说清楚。例如：

- “后端能力已具备”：只表示服务层、接口或测试完成，不代表用户能在端侧走通；
- “端侧已接入”：必须是真实接口调用，不是静态表格、mock 数据或按钮占位；
- “可本地演示”：必须能说明使用哪个账号、哪个入口、哪些步骤演示；
- “可验收完成”：必须同时具备端侧入口、权限、状态流转、积分/审计追溯和测试。

以下做法禁止直接标记为完成：

- 只新增数据库表或模型，没有迁移、接口和测试；
- 只写服务层函数，没有 HTTP 入口或调用方；
- 前端只放静态数据、占位按钮或“待接入”表格；
- 小程序页面继续调用旧 `dev-api`，却声称已经接入新后端；
- 用 `998/999` 代替日常业务审批角色；
- 用职务数字大小或部门数字直接鉴权；
- 业务代码直接修改积分余额；
- 改了需求、接口、数据库或运维行为，却没有同步更新文档。

- 权限检查不使用 `identity_code >= 1` 这类数字比较；
- 借用申请提交时可用积分不足会被拒绝，审核时可用积分不足会自动驳回并保留审核记录；
- 待审核或已驳回借用申请可以由申请人修改或取消，取消原因选填；已驳回申请被修改后回到待审核，并刷新 `submitted_at` 作为最新提交时间；取消已驳回申请不删除审核记录、不撤销驳回结论；
- 借用申请可以冻结积分，取消或归还可以解冻；
- 第一阶段异常归还只支持全额扣除本次申请冻结押金，部分扣除进入 1.5 阶段；
- 异常归还备注必填，正常归还备注可选；
- 异常归还确认后是终态，不做普通撤销；误操作通过受控积分反向修正、库存调整和审计兜底；
- 项目立项、审核、材料上传和结项不进入当前最小端到端验收主线，后续作为 1.5 阶段独立闭环实现。

### 5. 成员网页端

第一阶段成员网页端定位为“端到端验收台 + 可继续产品化的雏形”，不是完整会员产品。
它的任务是把身份、资料、积分和物资借用主线跑通，帮助后续小程序业务页和完整会员门户复用已经验证过的 API。

成员网页端的长期产品形态应拆成“公开主站首页 / 社区入口”“成员中心”和“协会工作台”三层。公开主站首页面向外部用户、游客和新成员，并包含 6 个全局业务入口：个人借物、团队借物、场地申请、协会活动、项目立项、查看项目；这些入口在旧小程序首页已有对应实现，但网页端是完整上位入口，小程序后续承接其中的高频轻量子集。成员中心面向普通已登录成员；协会工作台只面向协会工作人员或被授予对应权限的用户，并覆盖更完整的内部工作能力。BBS-like 讨论区属于未来社区模块，不能在第一阶段反过来挤占资料、积分、资源和借用主线。

第一阶段可以做轻量公开主站首页，让 `apps/web` 不再只是登录框；未登录用户可以浏览协会介绍、品牌视觉、登录/首次登录、小程序入口、6 个首页业务快捷入口和少量手工维护的内容预览。但第一阶段不做完整 CMS、动态活动列表、动态项目展示、官方博客发布流或社区互动闭环。登录后普通成员进入成员中心，优先服务资料状态、积分、待办、物资借用入口和我的借用记录。

交付物：

- 登录、首次设置密码、退出登录；
- 启动态 `/auth/me` 校验、access token 自动续签和认证缓存清理；
- 复用 `packages/api-client` 的认证、权限、错误响应和令牌刷新封装；`apps/web/src/api.ts` 只保留成员网页端 base URL 配置，`apps/web/src/auth-storage.ts` 只保留 storage key 配置；
- 公开主站首页保留小程序首页的 6 个全局业务快捷入口，其中第一阶段已接入能力进入真实流程，未接入能力必须明确标记为后续能力；
- 登录后普通成员第一屏改为“成员中心”，不再以工作台任务作为唯一主视图；
- 成员中心展示资料状态、积分概览、我的待办、物资借用入口、我的借用记录摘要和常用入口；
- 协会工作台入口只对协会工作人员或有对应权限的用户显示，工作台内部模块继续按权限点过滤；
- 个人资料查看和最小自助更新；
- 积分余额、冻结积分和近期流水；
- 物资资源列表和详情；
- 物资借用申请；
- 我的物资借用记录：展示本人全部状态，只有待审核和已驳回申请可修改或取消，不提供归还入口；
- 项目入口只保留后续 1.5 阶段规划说明，当前不做项目立项、材料上传和结项提交页面。

第一阶段前端执行拆分：

| 顺序 | 任务 | 归属范围 | 完成口径 |
| --- | --- | --- | --- |
| 1 | `apps/web` 信息架构壳 | `apps/web/src` | 匿名默认进入轻量公开主站首页；首页展示协会介绍、登录/首次登录、小程序入口和 6 个业务快捷入口；未接入入口明确标记后续能力 |
| 2 | 认证入口迁移 | `apps/web/src` | 复用现有邮箱密码登录、首次验证码登录、设置密码、会话恢复和退出逻辑；登录成功进入成员中心，不再直接进入任务验证页 |

| 9 | 端到端联调 | `apps/web`、`apps/admin`、`packages/api-client` | 能从成员中心提交借用申请，在后台审核/归还后回到成员中心看到状态、积分流水和记录变化 |

第一阶段不做：

- 完整个人中心、成长体系、复杂资料运营和会员门户首页；
- 动态活动、社区互动、通知中心和复杂展示型内容页面；
- 完整公开官网、动态活动列表、动态项目展示、官方博客发布流和 CMS；
- BBS/论坛主站、发帖评论、举报审核、社区积分和内容治理；
- 项目立项、项目材料、项目结项和项目工作台页面；
- 复杂工作台任务体验。当前已有工作台任务验证页可以保留，但它不是当前最小端到端验收主线的阻塞项；
- 大范围移动端适配和完整视觉产品化。当前优先保证桌面网页端可验收、接口真实、状态清楚。

当前进展：

- 已完成成员网页端认证页面；
- 已接入邮箱密码登录、首次邮箱验证码登录和首次设置密码；
- 已接入 access token 启动校验、refresh token 自动续签和退出登录；
- 已完成当前用户最小资料展示；
- 已接入组织域成员资料读取与自助更新；
- 已接入积分余额和最近流水；
- 已接入工作台任务端到端验证页面，覆盖任务发布、领取、提交和审核；但它属于协会工作台能力，不是普通成员中心主入口；
- 已把成员网页端原本的本地请求封装、认证存储语义和工作台状态标签抽入 `packages/api-client` 第一版；
- 当前代码中第一屏仍偏“登录后任务验证页”，需要重排为公开主站、成员中心和协会工作台三层入口；
- 资源借用、资源管理仍待实现；项目页面进入 1.5 阶段，不阻塞当前最小端到端验收。

验收标准：

- 成员网页端可以跑通第一阶段核心业务，尤其是“资料/积分查看 -> 物资浏览 -> 借用申请 -> 我的借用记录”；
- 我的借用记录必须展示 `pending_review`、`approved`、`rejected`、`cancelled`、`returned`、`exception_closed` 的清晰标签；用户只可修改或取消 `pending_review` 和 `rejected`，不能在成员端归还；
- 长表单和文件上传优先在网页端验证；
- 页面不依赖旧接口响应；
- 页面不能直接解析旧式 `code/msg/data`，也不能绕过共享调用层自行解析新版错误响应；
- 页面必须调用真实 API，不允许用静态卡片、mock 表格或占位按钮代替可验收主线；
- 网页端和后续小程序共享 API 契约。

### 6. 后台管理端

第一阶段后台管理端不是完整后台管理系统，而是端到端主线审核控制台。它的目标是让
“成员提交借用申请 -> 有权限的人审核和归还 -> 积分/审计可追溯”这条主线闭合。

交付物：

- 后台登录和会话恢复，复用统一 `access_token`、`refresh_token` 和 `/auth/me`；
- 权限摘要读取和按权限过滤的最小菜单；
- 后台 P0 壳只做最小可用后台：未登录展示登录页；已登录后读取当前用户和权限摘要；只显示当前用户有权限的第一阶段主线入口；没有可访问模块时展示无权限空态和退出入口；
- 后台 P0 壳的菜单先限于第一阶段主线：物资借用审核/归还、必要的物资台账入口、积分流水和审计追溯入口；不要为了像完整后台而提前铺成员管理、项目审核、运营大盘、系统配置等静态模块；
- 后台端必须复用 `packages/api-client`，不能另写一套后台专用的 `fetch`、错误解析、令牌续签和权限摘要读取逻辑；
- 后台真实登录、会话恢复和权限菜单是 P0 前置；不能先做一个脱离认证和权限的静态审核页来声明后台主线控制台完成；
- 物资借用申请列表、详情、审核、正常归还和异常归还；
- 借用审核工作台默认展示 `pending_review` 待办，并按 `submitted_at desc, id desc` 排序；其他状态通过筛选查看；
- 审核动作只在待审核申请展示，归还动作只在已通过待归还申请展示，历史状态默认不混入待办列表；
- 正常归还和异常归还必须是两个明确动作，不做一个含糊的“确认归还”通用按钮；
- 正常归还使用轻量二次确认弹窗，提示将恢复库存并解冻押金，备注可选；
- 异常归还使用高风险二次确认弹窗，提示将全额扣除本次冻结押金且不自动恢复库存，并强制填写原因；
- 审核时需要展示的成员、物资、押金和积分余额只读信息；
- 积分流水和审计日志查看入口，用于验证押金冻结、解冻或扣除是否可追溯。
- 权限菜单按权限点拆分：物资台账维护看 `resources.material.manage`，物资借用审核/归还看 `borrowing.application.review`。
- 第一阶段暂不拆正常归还和异常归还的后端权限点；后台 UI 必须拆动作和风险提示，后续再把异常归还升级为独立高风险权限点。
- 第一阶段可以用 `resource_manager` 预置角色代表基管部常态职责，同时打包两个权限点；后台代码不能依赖角色名或部门名判断菜单和操作权限。

第一阶段不做：

- 成员批量管理、复杂筛选、统计导出和完整花名册运营；
- 资源全量管理、批量导入、盘点报表和复杂库存运营；
- 项目审核、项目风控和项目材料后台工作台；
- 通用异常处理中心、系统配置中心和运营大盘；
- 为了填满后台菜单而新增静态页面、mock 表格或占位按钮；
- 把后台 P0 壳做成完整后台框架，再用大量“待接入”模块伪装进度。

验收标准：

- 后台管理端直接重写，不迁移旧后台 UI；
- 业务人员只能进入自己有权限的模块；
- 后台主线审核控制台必须先证明真实登录、`/auth/me`、`/permissions/me` 和权限菜单可用；
- 后台 P0 壳必须能证明“同一个后台应用，不同权限账号看到不同入口”；不能靠前端固定菜单、角色名硬编码或静态表格模拟权限效果；
- 后台登录、权限菜单、审核和归还页面必须使用共享 API 客户端，保证和成员网页端使用同一套响应信封、错误码、令牌续签和权限判断基础能力；
- `998/999` 只作为底层系统兜底身份，不作为日常业务审批角色；
- 后台审核、归还和异常归还必须调用真实 API，不允许使用静态数据或 mock 数据；
- 归还动作只出现在后台主线审核控制台或后续小程序管理端，不出现在成员中心；
- 即使后端暂时共用 `borrowing.application.review`，后台页面也必须把正常归还和异常归还拆成两个动作；异常归还必须做高风险二次确认、展示扣押金和不恢复库存的后果，并备注必填；
- 后台控制台验收时可以给基管部测试账号授予 `resource_manager`，但接口和菜单仍要证明权限点拆分有效；
- 后台操作能产生审计记录；
- 后台控制台能支撑最小端到端验收主线，不要求覆盖完整后台管理系统。

### 7. 小程序接入准备

当前阶段小程序只承担身份前置，不承担业务闭环验证。物资借用、资源浏览、积分流水和
后台审核先在成员网页端和后台主线审核控制台闭合。

小程序后续定位为高频轻量入口，可以承接查看、提交、状态跟踪、简单审核和正常归还等移动场景。
物资台账维护、物资类别多层管理、库存调整、资源状态处理、复杂筛选导出和异常处理不放在小程序端，
统一放在浏览器端，管理类主入口为 `apps/admin`。

交付物：

- 小程序新版认证接入：微信登录、`/auth/me` 启动态校验、`/auth/refresh` 续签和认证缓存清理；
- 小程序邮箱绑定页接入：`/auth/email/send-code` 和 `/auth/email/bind`；
- 小程序后续业务页迁移时使用 `packages/api-client` 定义的统一响应、错误和业务类型，并通过小程序 `wx.request` 传输适配进入同一套契约；
- `apps/miniapp` 接入说明，明确当前只验收登录和绑定；
- 后续业务页迁移清单，只列待迁移页面，不在当前阶段实现。

验收标准：

- 微信开发者工具可以直接打开 `apps/miniapp`；
- 小程序启动或进入关键页面时不能只看本地 `auth_token`，必须结合过期时间和 `/auth/me` 校验；
- access token 过期时优先使用 refresh token 续签，refresh 失效后重新走微信登录；
- 邮箱绑定能完成验证码发送、验证码消费和绑定结果缓存更新；
- 小程序物资、场地、项目、活动、任务、宣传链接等业务页面不作为当前验收范围；
- 后续业务页迁移必须优先接入已经由成员网页端和后台控制台验证过的 API；
- 迁移后的业务页不能继续在页面内手写 `wx.request` 响应判断或解析旧式 `data.code === 200`，必须走统一小程序请求适配和新版错误响应。

## 验收总标准

第一阶段完成时，应满足：

- 后端、成员网页端、后台管理端、文档站都能本地启动；
- Docker 可以启动第一阶段所需基础服务；
- 最小端到端验收主线可以从小程序登录绑定邮箱一路走到网页端借用申请、后台审核归还、积分流水和审计追溯；
- 身份、权限、积分、物资资源、物资借用的核心闭环可演示，项目域明确标记为 1.5 阶段文档规划；
- 所有核心变更都有中文文档；
- 数据库模型和 API 契约能支撑后续小程序接入；
- 暂缓需求有明确边界，不阻塞核心闭环。
