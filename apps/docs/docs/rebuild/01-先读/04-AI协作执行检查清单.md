# MakersHub AI 协作执行检查清单

## 目标

本文档用于约束后续 AI 或人工继续开发时的执行口径，避免出现“代码写了很多，但端到端验收仍然走不通”的情况。

如果一个 agent 只能看少量文档，至少应先读：

1. `01-需求核对清单.md`：确认业务边界和身份链路；
2. `02-第一阶段路线图.md`：确认当前阶段的端到端验收主线；
3. `03-工程运维/02-后端代码规范.md`：确认后端注释、旧代码参照和测试要求；
4. 本文档：确认交付状态、禁区和交接说明。

## 端到端优先原则

第一阶段不以“某个后端域已经写完”作为完成标准，而以真实用户路径可以验收为准。

`apps/web` 指公开主站首页、成员中心和协会工作台所在的 Web 入口应用；`apps/admin` 指后台管理端。二者共享认证、权限点和业务 API，但不是同一个前端应用里按权限渲染两套页面。

当前最小验收主线是：

```text
小程序微信登录
  -> 小程序绑定邮箱
  -> 网页端首次邮箱验证码登录
  -> 网页端首次设置密码
  -> 成员资料和积分账户可查看
  -> 成员提交物资借用申请
  -> 后台按权限审核借用和归还
  -> 积分押金冻结、解冻或扣除
  -> 审计日志和积分流水可追溯
```

实现任何新能力时，都要回答：它推进了这条主线的哪一步？如果没有推进，需要说明为什么它是必要前置，而不是旁支扩张。

## 交付状态标签

后续进度说明必须使用清晰状态，不能只写“已完成”。

| 状态 | 可以这样说 | 不能这样说 |
| --- | --- | --- |
| 文档规划 | “项目域已完成需求和表设计规划” | “项目完成” |
| 后端能力已具备 | “物资借用服务层和接口已具备” | “物资借用端到端完成” |
| HTTP 已开放 | “接口已开放，等待端侧接入” | “用户可以使用” |
| 端侧已接入 | “网页端已调用真实接口提交借用申请” | “后台审核也完成了” |
| 可本地演示 | “可用本地账号手动走通主流程” | “生产可用” |
| 可验收完成 | “端侧、权限、业务状态、积分/审计、测试和文档都闭合” | “还差页面/测试/文档” |

## 开发前检查

开始写代码前先检查：

- 是否读过 `01-需求核对清单.md` 和后端代码规范；
- 是否需要查看旧 `makershub-backend` 或旧小程序页面来确认业务语义；
- 当前任务属于端到端主线的哪一步；
- 当前任务是否会修改身份、权限、积分、借用状态、数据库结构或接口契约；
- 如果会修改，是否同步准备更新对应文档；
- 端侧任务是否应该复用或完善 `packages/api-client`；如果涉及请求、认证、令牌、权限、错误码、分页或业务状态标签，不要先在页面里手写一份；
- 是否已有类似业务域、能力模块、测试写法可以复用。

## 评审和拷打前事实盘点

对方案、需求文档或其他 agent 产物做评审时，不能用追问代替读代码。开始拷打前先完成一轮事实盘点：

- 涉及身份链路时，先看小程序新版登录、令牌校验、邮箱绑定页面，以及网页端首次登录、设置密码和会话恢复实现；
- 涉及物资借用时，先看旧小程序的物资申请、我的借用详情、修改申请、取消、审核和归还页面，再看新版后端 router、schema、service、repository 和测试；
- 涉及资源台账时，先看旧小程序资源加载方式，再看新版资源接口、服务层和权限点；
- 涉及端到端验收时，必须同时看成员网页端和后台管理端当前是否接入真实 API，不能只看后端接口；
- 如果旧 `makershub-backend` 不在当前仓库，必须记录“旧后端未在仓库中找到”，并把旧小程序、现有文档和新版测试作为可用事实来源；
- 如果旧实现已有明确业务语义，新版决定保留或偏离都要写清楚；不能在没有证据的情况下把旧功能改成另一个流程；
- 旧代码、新代码或测试中已经能确认的行为，必须先查证并引用事实，不能用“如果旧系统如何”“如果前端如何”这类假设性问题让用户重复回答；
- 只有旧代码、新代码、测试和文档都没有覆盖的设计分支，才进入拷打提问，并且要明确说明这是“未实现/未定义”的决策点；
- 如果文档已经声明某个接口或能力存在，但代码中没有对应 router、service 或测试，要标记为“文档-实现缺口”，不能写成已完成。
- “拷打方案/评审文档”默认不是“实现代码”授权。除非用户明确说“可以改代码”“直接实现”或提出具体编码任务，否则只查事实、指出缺口、沉淀文档，不要顺手修改业务代码。
- 如果拷打过程中发现后端、前端、迁移或测试缺口，应先把缺口和推荐实现口径说清楚；是否进入代码实现由用户单独确认。

## 实现中禁区

以下行为会破坏重构目标，应主动避免：

- 普通用户从网页端先注册普通账号，再绑定微信；
- 用 `identity_code >= 1`、职务数字大小或部门数字直接做接口鉴权；
- 把 `998/999` 当成日常业务审批角色；
- 业务域直接修改积分账户余额，绕过积分流水和账本服务；
- 前端页面只写静态数据、mock 数据或“待接入”按钮，却在文档里写成已完成；
- 把成员网页端第一阶段做成完整会员门户、完整社区、项目工作台或复杂视觉展示页，却没有先接通资料、积分、物资浏览、借用申请和我的借用记录；
- 为了让 `apps/web` “像网站”，在第一阶段堆大量静态内容、假活动、假项目、假博客或假社区，而不是先做轻量公开主站入口并完善成员中心和物资借用主线；
- 把公开主站 / 社区入口设计成完整 BBS/论坛，提前实现发帖、评论、举报、社区审核、社区积分等后续社区能力，导致当前端到端主线被推迟；
- 小程序只接入登录和邮箱绑定，却声称小程序业务页已完成新后端适配；
- 在当前最小主线中迁移小程序物资、项目、活动、任务等业务页面，导致网页端和后台控制台验收被打散；
- 后台管理端铺完整菜单、静态表格或 mock 审核按钮，却没有接入真实登录、权限和借用审核 API；
- 成员网页端和后台管理端各自复制一套 `fetch`、响应信封解析、`ApiRequestError`、令牌续签、`/auth/me`、`/permissions/me` 和错误码处理逻辑；
- 新小程序业务页继续在页面中直接手写 `wx.request` 并判断旧式 `data.code === 200`，而不是通过统一请求适配处理新版 `success/error/request_id` 响应；
- 后台菜单或接口用 `resource_manager` 角色名、基管部部门名或职务数字硬编码判断权限；即使业务常态由基管部负责，也必须检查 `resources.material.manage`、`borrowing.application.review` 等权限点；
- 把项目立项、审核、材料上传或结项混入当前最小主线；项目域已经降级到 1.5 阶段，当前只保留设计草案；
- 新增主要目录、业务域或能力模块时不写 README；
- 修改需求、接口、数据库、权限点或运维行为时不更新文档；
- 新增后端 Python 文件时缺少文件路径头、中文模块说明、分区注释和关键函数 docstring。

## 交付前检查

交付说明中至少写清楚：

- 本次推进的是哪个端到端步骤；
- 哪些文件和文档被修改；
- 当前能力处于哪个交付状态标签；
- 还缺哪些步骤才能进入“可验收完成”；
- 运行了哪些 lint、test 或文档构建命令；
- 如果没有运行，原因是什么。

涉及后端代码时至少运行后端 lint/test；涉及文档时运行文档构建。

## 当前最容易误判的区域

- 后台管理端第一阶段只是主线审核控制台，不能只靠静态 Ant Design 表格判断完成；真实登录、`/auth/me` 会话恢复、`/permissions/me` 权限摘要和按权限点过滤菜单是 P0 前置，之后才算进入物资借用审核/归还、积分流水和审计接口验收。
- 后台 P0 壳是“最小可用后台”，不是完整后台框架。它只需要登录页、会话恢复、权限摘要、按权限过滤的主线菜单、无权限空态和退出入口；不要先铺成员管理、项目审核、运营大盘、系统配置等静态模块。
- 后台 P0 壳必须用不同权限账号证明菜单差异：`borrowing.application.review` 才显示物资借用审核/归还，`resources.material.manage` 才显示物资台账维护。不能用固定菜单、角色名、部门名或静态数据模拟权限效果。
- `packages/api-client` 已有第一版共享调用层；`apps/web` 只保留 base URL 和 storage key 配置，`apps/admin` 还没有真实调用层。进入后台 P0 登录、权限菜单和审核页时，必须复用现有共享客户端，并按需补齐物资借用、审核、归还、积分流水和审计 API，不能重新写一套后台专用客户端。
- 网页端、小程序端和后台管理端共享同一套 `/api/v1` 业务契约、状态机和业务事实；迁移小程序业务页时不要重做一套 miniapp 专用借用、积分、资源或审核语义。
- 共享 API 客户端要支持不同端侧适配：web/admin 使用 `fetch`，miniapp 后续使用 `wx.request`；认证存储 key 和存储介质可以按应用配置，但令牌过期判断、续签、清理和错误处理语义必须一致。
- 可以抽到共享调用层的是：响应解析、错误对象、错误码常量、分页参数、认证和权限接口、成员资料/积分/物资/借用/审核等 typed API、业务状态和轻量标签映射。不能抽到 `api-client` 的是 React 页面、Ant Design 表格、路由菜单、小程序页面结构、弹窗和表单布局。
- 端侧功能是包含关系，不是交叉关系：小程序用户端是成员网页端高频轻量子集，小程序管理端是后台管理端高频轻操作子集；移动端可以有摘要/聚合读取，但不能绕过统一权限、状态流转和后端校验。
- 新 API 失败响应只有 `success=false`、`error.code`、`error.message`、`error.details`、`request_id` 这一种结构；新增错误码时不能恢复旧小程序的顶层 `code/msg/data` 返回。
- 基管部通常同时负责物资台账和借用审核，所以第一阶段可以给测试账号授予 `resource_manager`；但这只是预置角色打包，不是代码鉴权依据。
- 成员网页端第一阶段仍以端到端验收为主；资源借用页面和我的借用记录没有接通前，不要扩展动态活动、完整社区、项目流程、完整个人中心或复杂协会工作台体验。
- `apps/web` 长期应拆成公开主站首页 / 社区入口、成员中心和协会工作台三层；公开主站首页要提供 6 个全局业务入口：个人借物、团队借物、场地申请、协会活动、项目立项、查看项目。旧小程序已有这些入口，可作为现有业务参照，但网页端是完整上位入口，小程序后续承接高频轻量子集。BBS-like 讨论区可以作为后续社区频道，但内容域和社区域必须分开，第一阶段不实现社区互动能力。
- `apps/web` 第一阶段可以做轻量公开主站首页，但只承载协会介绍、品牌视觉、登录/首次登录、小程序入口、6 个首页业务快捷入口和少量手工维护的内容预览；不要做完整 CMS、动态活动列表、动态项目展示、官方博客发布流或社区互动闭环。
- 当前 `apps/web` 第一屏仍偏“登录后任务验证页”；后续必须重排为公开主站、成员中心和协会工作台三层。普通成员自助区叫“成员中心”，协会工作台只面向协会工作人员或有权限用户，任务和值班等不能继续占据普通成员第一屏。
- `apps/web` 第一阶段实现必须按“信息架构壳 -> 认证入口迁移 -> 成员中心壳 -> 协会工作台入口收纳 -> 共享 API 补齐 -> 物资浏览 -> 借用申请 -> 我的借用记录 -> 端到端联调”的顺序推进；不要跳过共享 API 和真实接口去堆静态卡片。
- 小程序当前只验收新版登录、令牌校验/续签和邮箱绑定；业务页面仍大量使用旧接口是已知状态，后续要等网页端和后台控制台验证 API 后再逐页迁移。
- 成员端物资列表默认只展示 `status=available` 的物资；后端也必须按权限兜底过滤，没有 `resources.material.manage` 的用户即使手动传其他状态，也不能看到维修中、停用或下架台账数据。
- 物资详情同样要按权限兜底：普通成员只能读取可借物资详情；直接访问不存在物资或非 `available` 物资详情时都返回 `404 MATERIAL_NOT_FOUND`，后台台账视角才允许查看维修中、停用或下架物资。
- 物资借用申请必须在提交前展示预计押金、可用积分和当前可借数量；后端提交和修改接口也要拒绝押金不足或当前可借库存不足。审核通过时还要二次校验余额和库存，不足则自动驳回，不能先扣库存再发现押金冻结失败。
- 提交或修改时库存不足必须返回 `BORROW_MATERIAL_STOCK_NOT_ENOUGH` 和 HTTP `409`；`details` 必须包含 `material_id`、`material_name`、`requested_quantity` 和 `available_quantity`，不能让端侧解析中文提示。
- 人工驳回物资借用申请必须填写驳回理由；前端禁用空理由只是体验，后端也必须拒绝空理由驳回。系统自动驳回则由后端写入库存或积分不足原因。
- 确认归还不是普通成员自助动作；第一阶段只暴露在后台主线审核控制台，后续可暴露给小程序管理端。
- 后台归还 UI 必须拆成“正常归还”和“异常归还”两个动作；不要做一个含糊的“确认归还”按钮再让操作者临场选风险分支。
- 第一阶段不要急着新增正常归还/异常归还后端权限点；暂时共用 `borrowing.application.review`，但必须在页面动作、二次确认、备注和交付说明里体现风险差异。
- 审核和归还接口第一阶段的幂等口径是防重复副作用，不是重复请求必须返回第一次成功结果；重复点击后状态已流转可以返回 `409`，但绝不能重复扣库存、冻结押金、恢复库存或解冻/扣除押金。
- 正常归还的二次确认弹窗要说明会恢复库存并解冻押金，备注可选；异常归还的二次确认弹窗必须使用高风险文案，说明会全额扣除押金且不自动恢复库存，并强制填写原因。
- 第一阶段异常归还只做全额扣除本次申请押金；不要新增部分扣除金额、按损坏数量扣除、剩余押金解冻等分支。
- 异常归还必须要求填写备注，不能允许空原因扣押金和关闭库存；正常归还备注可以为空。
- 第一阶段不要给异常归还增加普通撤销、重新归还、状态回退按钮；误操作走受控库存调整和积分反向修正。
- 已驳回申请允许申请人取消收尾，但不能删除审核记录，也不能把取消写成“撤销驳回”。
- 旧小程序已经支持物资借用“修改申请”：详情页携带 `edit=true&sb_id` 回到申请页并调用 update 接口；新版不能把已驳回修改误改成“取消后新建”。
- 旧小程序物资申请强制选择归还日期；新版个人物资借用提交和修改也必须由后端校验 `expected_return_at` 非空。
- 用户取消借用申请时取消原因选填；前端不能把原因做成必填，后端也不能因为空原因拒绝合法取消。
- 成员端“我的借用记录”要展示全部状态；只能对 `pending_review` 和 `rejected` 展示修改和取消动作，不能给 `approved` 增加归还、自助归还或异常归还按钮。
- 修改 `rejected` 申请后应回到 `pending_review` 等待重新审核，并保留原审核记录；旧小程序 `state != 2` 这种宽按钮条件不能照搬，新版要收窄到待审核和已驳回。
- 修改申请必须由后端按当前物资台账重新校验身份、状态、物资可借、数量、当前可借库存、押金和可用积分；前端提示、按钮禁用或缓存数据不能代替后端判断。
- 修改申请不是局部补丁；前端必须提交 `reason`、`expected_return_at` 和整份 `items` 明细，后端必须拒绝缺失或空明细，也不能允许客户端修改借用类型、项目、申请人、状态或押金字段。
- 修改待审核或已驳回申请是第一阶段 P0；如果后端没有 `PATCH /api/v1/borrowing/applications/{application_id}`、测试和端侧入口，不能把物资借用标记为端到端可验收完成。
- 修改已驳回申请后，端侧主状态必须展示待重新审核；旧驳回意见只能作为历史审核意见展示，不能继续作为当前审核结果或醒目错误态展示。
- 修改申请成功后必须刷新 `submitted_at`；后台待审核队列必须按 `submitted_at desc, id desc` 排序，不能只按原始 `created_at` 排序。当前如果实现仍按 `created_at desc, id desc` 排序，属于第一阶段 P0 实现缺口。
- 后台审核工作台默认只展示 `pending_review` 待办；其他状态通过筛选查看。审核按钮只出现在 `pending_review`，正常/异常归还按钮只出现在 `approved`。
- 借用申请必须保存申请人姓名、学号、手机号、联系邮箱、年级和专业快照；历史展示用快照，权限和积分处理仍使用 `applicant_id`。
- 申请人快照必须从成员资料生成，借用表单不能自由填写身份/联系字段；联系邮箱默认取登录邮箱且可在资料中修改，修改联系邮箱不等同于更换登录邮箱；联系邮箱必须由后端校验格式，但不需要验证码确认；姓名、学号、手机号、年级或专业缺失，或联系邮箱和登录邮箱均缺失时，后端返回 `BORROW_PROFILE_INCOMPLETE`，端侧引导完善资料。
- 后台审核列表指后台管理端/后台主线审核控制台，不是成员中心页面。
- 借用列表范围必须由后端兜底：普通成员不传 `mine=true` 也只能看到本人申请；只有 `borrowing.application.review` 可以看全量；有审核权限但传 `mine=true` 时仍切回本人视角。
- 借用列表字段必须由后端兜底：成员本人列表和后台全量列表都只返回摘要，不返回学号、手机号、联系邮箱或 `applicant_current_contact`。
- 普通成员查看本人详情可以看到 `applicant_snapshot`，但端侧必须标注为“申请时资料”；`applicant_current_contact` 只给拥有 `borrowing.application.review` 权限的审核/归还人员看，并在详情接口默认返回，不需要额外开关参数；普通成员查看本人详情不返回该字段。
- 后台审核列表只展示摘要，不展示申请编号、学号、手机号、联系邮箱或当前联系方式；这些字段只在详情页展示。
- 后台审核列表不要拼接 `年级-专业-姓名`，应拆列展示姓名、年级、专业、提交时间、状态、类型、物资摘要和操作。
- 成员端“我的借用记录”列表是本人视角，不套用后台审核表格列；优先展示状态、提交时间、预计归还时间、物资摘要和可执行操作。
- 物资摘要使用借用明细快照生成：单项显示“物资名称 x 数量”；多项显示“首件物资名称 x 数量 等 N 项”，详情页再展示完整清单。
- 小程序是高频轻量入口，不承载物资台账维护、物资类别多层管理、库存调整、资源状态处理、复杂筛选导出和异常归还；待审核列表、通过/驳回和正常归还后续可以接入小程序管理端。
- 后台审核、归还和历史详情页主信息使用申请人快照；如展示当前手机号或联系邮箱，只能作为“当前成员资料”只读辅助信息，不能改写申请快照、审核记录、归还记录或审计日志。
- 借用明细必须保存申请时物资名称、分类名称和单件押金快照；历史展示用快照，库存扣减和归还仍通过 `material_id` 操作当前物资台账。
