# 后端基础设施建设清单

## 目标

本文档用于约束 MakersHub 后端基础设施建设，避免在核心底座没有明确完成状态时继续堆业务功能。

重构的目的不是“跑起来就行”，而是让后续维护者能清楚知道：

- 哪些基础能力已经可用；
- 哪些只是包边界或占位；
- 哪些能力在业务模块落地前必须先补齐；
- 哪些旧系统经验已经吸收，哪些不能照搬。

## 维护原则

- 空目录不作为交付物。保留目录必须至少有 `README.md` 或 `__init__.py` 说明边界。
- 文档中的目录规划和代码结构必须一致，不能同时存在 `logging.py` 和 `logging/` 这类漂移。
- 业务接口不能绕过基础设施，例如不能用身份数字比较替代权限点检查。
- 运行日志、业务审计和积分流水是三类不同东西，不能互相替代。
- 旧 Codex 对话记录可以作为上下文来源，但关键结论必须沉淀到仓库文档和代码注释中。

## 业界基线对照

本清单按以下公开基线持续对照：

- [OWASP API Security Top 10 2023](https://owasp.org/API-Security/editions/2023/en/0x00-header/)；
- [OWASP REST Security Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/REST_Security_Cheat_Sheet.html)；
- [OWASP HTTP Headers Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/HTTP_Headers_Cheat_Sheet.html)；
- [OWASP Logging Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Logging_Cheat_Sheet.html)。

当前已吸收的基础设施要求包括：

- API4/API6 相关的资源消耗和敏感业务流滥用防护：应用层全局限流、认证限流和请求体大小限制；
- API8 相关的安全配置防护：生产 CORS 禁止 `*`、生产 HSTS、基础安全响应头；
- API2 相关的认证安全：短期 access token、refresh token 哈希存储、刷新轮换、会话撤销；
- Logging Cheat Sheet 相关要求：运行日志分流、请求日志不记录 body/query、500 级错误入错误日志；
- REST Security 相关要求：限流错误使用 429，过大请求使用 413，错误响应不暴露内部堆栈。

仍未达到完整生产级的部分：

- 进程内限流不能覆盖多 worker、多实例和分布式攻击，生产必须接 Redis、网关或云安全产品；
- OpenTelemetry、集中日志、指标面板和告警还未接入；
- 依赖审计、可修复高危镜像扫描、secret 扫描和 SBOM 已有基础版，但生产镜像签名、全量 CVE 报告和制品准入还未落地；
- 文件上传还缺病毒扫描或异步安全扫描、临时文件清理和业务引用接入；
- 权限拒绝告警、聚合分析和对象级授权规则需要跟业务域一起细化；
- 备份恢复演练、灾备流程和生产 runbook 还需要落地。

## 近期必须补齐

### 1. 权限系统

已完成基础版：

- 权限点表；
- 角色表；
- 角色权限关系；
- 用户授权关系；
- 作用域字段，例如全局、部门、项目、资源；
- FastAPI `require_permission(...)` 依赖；
- `998/999` 系统职务到系统兜底权限的受控映射；
- `/auth/me` 和 `/permissions/me` 权限摘要。
- `require_permission(...)` 拒绝访问时写入 `permission.denied` 审计。
- 用户角色授权查询、授予和撤销接口；
- `permission.user_role_grant.create` 与 `permission.user_role_grant.revoke` 审计。
- `points.ledger.view` 和 `points.manual.adjust` 权限点；
- `points_manager` 只读角色，以及 `998/999` 的受控人工积分调整权限。

仍待完成：

- 权限拒绝告警和聚合分析；
- 后台菜单过滤；
- 项目、资源、场地等具体业务作用域规则；
- 自定义角色创建和角色权限编辑。

原因：

- 后台管理端不能等同于 `998/999`；
- 组织管理、资源管理、积分规则审批都依赖权限点；
- 需求文档明确禁止 `identity_code >= 1` 这种数字比较。

### 2. 审计系统

必须区分：

- 运行日志：服务运行、请求耗时、异常排查；
- 审计日志：谁在什么时候对什么业务对象做了什么；
- 业务流水：积分账户流水、借用状态流转等业务事实。

已完成基础版：

- `audit_logs` 表；
- `record_audit_log(...)` 服务；
- `GET /api/v1/audit/logs`；
- 绑定邮箱、首次邮箱登录、设置密码、退出登录和初始化 999 的审计写入。
- 权限授予、权限撤销、成员资料维护、成员部门调整和成员职务调整的审计写入。
- 人工积分调整的 `points.manual_adjustment.create` 审计写入。

后续第一批还应继续接入：

- 资源新增、下架、库存调整；
- 借用审核、归还、异常处理。

### 3. 积分账本基础

已完成第一阶段基础版：

- `point_accounts`：用户积分账户，保存总余额和冻结余额缓存；
- `point_holds`：业务冻结记录，支持解冻和冻结转扣除；
- `point_ledger_entries`：积分流水事实表，记录余额快照和业务来源；
- `points` 服务层：人工调整、冻结、解冻、冻结转扣除和幂等键校验；
- 当前用户积分账户和流水接口；
- 后台积分账户、流水查询和受控人工调整接口；
- 人工调整审计和权限点接入。

仍待完成：

- 固定积分规则和每周积分领取规则；
- 临时积分规则申请、审批、撤回和通知；
- 反向流水修正接口；
- 积分统计、通胀风险视图和导出；
- 借用、3D 打印、任务等业务域接入冻结和自动发放。

原因：

- 积分相当于协会内部货币，不能再放在 `users.score` 上直接覆盖；
- 后续借用押金、打印消费、悬赏任务和人工异常修复都依赖同一个账本事实；
- 流水和审计互相补充，流水证明“积分事实”，审计证明“谁触发了敏感操作”。

### 4. 共享工具清理

需要逐步清理：

- 同名 `.py` 文件和目录不能并存；
- 未实现目录必须有 README 说明“不负责什么”和“何时实现”；
- 新增业务域前先确认是否需要共享编号、时间、分页、文件摘要等基础能力。

## 旧记录处理规则

已经确认旧对话记录位于宿主机 Codex 记录包中，且旧对话非常长。后续不能依赖“我记得”，需要按下面顺序处理：

1. 先读仓库内 `01-先读/01-需求核对清单.md` 和 `03-工程运维/02-后端代码规范.md`；
2. 再按本次要做的业务查看旧 `makershub-backend` 和小程序源码；
3. 如果需求仍有疑问，再检索旧 Codex 记录；
4. 得到的新结论必须写回仓库文档或代码注释。

这样即使对话压缩或换模型，项目约束也不会丢。
