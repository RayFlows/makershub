# MakersHub 重构需求核对清单

## 目标

本文档用于持续核对 MakersHub 的需求边界，避免在项目结构、权限、端侧入口、接口和后续增量规划上做错方向。

本次整理已经把前面多轮讨论中的行内备注、冲突表述和重复问题合并为一份较稳定的需求基线。后续如果继续确认需求，优先更新本文档对应条目，不再把新答案追加到旧问题后面。

## 使用方式

- “已确认需求”可以直接作为架构设计、数据库设计和增量拆分的输入。
- “权限功能编号”用于继续核对权限和端侧入口。
- “后续细化问题”只保留不阻塞第一阶段开工、但实现前还需要补充细节的问题。
- “暂缓需求”表示后续会做，但不放进当前最小重构闭环。

## 一、已确认需求

### 1.1 重构方式

- 采用严格的软件工程增量模型。
- 不在旧仓库上继续大改，而是新建 `makershub`。
- 后端技术栈暂定继续使用 FastAPI、SQLAlchemy、MySQL、MinIO。
- `makershub` 采用单仓库多应用结构，顶层分为 `apps/`、`packages/`、`infra/` 和 `scripts/`。
- 后端依赖管理采用 `uv + pyproject.toml`。
- 前端采用 `pnpm workspace` 管理多应用和共享包。
- 成员网页端和后台管理端拆成两个应用，共享 API 客户端、类型和 UI 组件。
- 前端技术方向采用 React、Vite、TypeScript 和 Ant Design Pro 风格。
- 旧业务数据可以清空重建，不要求迁移历史业务数据。
- 所有项目文档默认使用中文。
- 需要维护仓库内 Markdown 文档。
- 需要规划文档网站，文档网站放在新仓库内部。

### 1.2 业务背景

- 项目服务对象是开源硬件协会。
- 当前协会有四个部门：
  - 宣传部；
  - 基管部；
  - 项目部；
  - 运维部。
- 项目后续会扩展很多能力，包括博客、社区、电控、AI、设备管理、积分制度和更多内部运营功能。
- 因为项目会长期增长，所以项目结构必须按业务域组织，不能只按当前页面或当前旧接口拆文件。

### 1.3 架构原则

- 部门不是代码模块，业务域才是代码模块。
- 身份与权限、组织与成员要分开。
- 积分制度要作为独立账本系统设计，不能只是用户表上的 `score` 字段。
- 资源和借用要分开。
- 内容和社区要分开。
- 项目和工作台要分开。
- 电控必须单独成域。
- AI 不能成为业务事实来源。
- 当前旧后台管理系统在重构初期可以完全删除，后续按新业务重写。
- 重构初期需要保留的是统一身份、权限系统，以及后续后台管理会调用的业务接口。

### 1.4 端侧规划

MakersHub 至少有三类用户入口：

| 入口 | 用途 |
| --- | --- |
| 小程序 | 短流程、轻表单、移动场景、简单审核、查看和报名 |
| 网页端 | 长表单、材料上传、复杂内容编辑、需要反复修改的申请 |
| 后台管理系统 | 列表管理、批量操作、统计导出、敏感配置、审计追踪、异常处理 |

已确认原则：

- 端侧能力是包含关系：小程序提供轻量入口，网页端覆盖更完整的成员操作，后台管理系统覆盖管理和运维。
- 第一阶段优先实现成员网页端，用网页端验证登录、借用、项目和积分账本等核心业务逻辑。
- 后台管理系统直接重写，不迁移旧后台前端。
- 旧后台的 React 和 Ant Design 方向可以保留，但不继续使用旧的 CRA/react-scripts 工程结构。
- 当前小程序已有的简单审核可以保留在小程序端。
- 大型表单管理、复杂材料提交、批量管理和数据导出更适合网页端或后台管理系统。
- 后台管理系统访问不等于 `998` 管理员身份；业务人员也可以进入自己有权限的后台模块。
- 网页端和后台管理系统第一版不能只依赖微信登录，需要支持账号密码登录。
- 小程序端继续采用微信无感登录，不要求普通小程序用户设置账号密码。
- 所有用户理论上都可以拥有账号密码，但微信端不强制使用账号密码。
- 账号密码登录主要服务网页端、后台管理系统、第一个 `999` 初始化，以及后续需要网页操作的用户。
- 账号密码登录和微信登录应绑定到同一个用户主体上：前期网页端可以先用账号密码，后续再绑定微信登录。
- 用户主体不要直接等同于微信 `openid` 或 `unionid`；建议内部使用稳定的用户主键，再挂接微信身份、本地账号密码等登录方式。
- 当前暂时拿不到稳定的微信 `unionid`，所以第一个 `999` 和网页端登录不能依赖 `unionid` 才能启动。
- 本地账号登录名采用邮箱。
- 账号绑定流程以小程序微信无感登录为前置：用户先进入小程序个人主页绑定邮箱，系统通过协会邮箱发送验证码。
- 网页端首次登录接受邮箱验证码，验证通过后强制设置密码，从而把网页端账号和微信小程序用户绑定到同一个用户主体。
- 第一版不考虑先注册网页账号再绑定微信；后续如果有独立 App 再重新评估。
- 第一版不允许解绑微信。
- 用户可以设置、修改密码，也可以通过邮箱验证码重置密码。
- 邮箱验证码有效期为 5 分钟。
- 邮箱验证码同一邮箱 1 小时最多发送 10 次，每次重新请求至少间隔 1 分钟。
- 第一版允许用户更换绑定邮箱。
- 更换邮箱需要验证旧邮箱和新邮箱；更换成功后新邮箱作为登录名，原密码保持不变。

### 1.5 身份、职务和权限模型

当前确认的身份和职务编号如下：

| 代号 | 名称 | 类型 | 当前理解 |
| --- | --- | --- | --- |
| 0 | 外部成员、协会会员 | 基础身份 | 已加入协会生态，但不一定是部门干事 |
| 1 | 协会成员、干事 | 基础身份 | 各部门干事 |
| 2 | 部长、部门负责人 | 部门职务 | 各部门负责人，2 及以上属于核心成员 |
| 3 | 副会长 | 协会职务 | 权限高于部长、低于会长；可以和部长兼任 |
| 4 | 会长 | 协会职务 | 协会最高业务负责人 |
| 5 | 指导老师 | 特殊身份 | 通过权限点授权；可以参与所有审核类事项，但不负责运维 |
| 998 | 管理员 | 底层管理身份 | 系统运维、敏感配置、异常修复和受控兜底操作 |
| 999 | 超级管理员 | 底层管理身份 | 母账号；比管理员多“指定管理员”的能力，其余底层能力与管理员一致 |

已确认原则：

- 一般业务权限尽量做包含关系：高一级身份拥有前一级身份的大部分权限。
- `1` 和 `2` 是明显分水岭：`2` 及以上是核心成员，`1` 及以下不是核心成员。
- 副会长参与权限继承，不再仅作为展示标签。
- 副会长每周积分应高于部长，具体数值后续建模。
- 副会长的场地权限通过“场地管理关系”确定：如果副会长被分配或申请到某个场地的管理关系或对应工位，则拥有该场地相关权限。
- 指导老师是特殊身份，不按普通层级继承，但审核类事项权限较高。
- 指导老师可以参与积分人工扣减、积分异常撤回等审核或复核事项；最终落库操作需要结合积分部分的执行边界。
- `998` 和 `999` 确认为绑定到已有用户上的附加系统管理身份，不是独立后台账号。
- `998` 和 `999` 属于底层管理身份，不作为日常业务审批、业务查看、业务导出的默认角色。
- `998` 主要用于运维、系统配置、异常修复、敏感数据处理。
- `999` 主要用于初始化、权限恢复、指定管理员、破窗处理。
- 全系统只能有一个 `999`。
- `999` 平时一般不使用，它的核心工作是指定或恢复 `998`。
- 第一个 `999` 不应从普通业务界面产生，建议通过部署初始化命令或一次性初始化流程产生。
- 第一个 `999` 初始化时应优先创建或绑定本地账号密码登录方式，后续再支持绑定微信身份。
- 灾备重置 `999` 必须要求服务器 sudo 管理员操作。
- 灾备重置 `999` 后，旧 `998` 需要全部失效。
- 灾备重置必须记录执行人、原因、时间和目标账号。
- 代码中不能用 `identity_code >= 1` 这类数字比较直接做接口鉴权。
- 接口应该统一检查权限点，再由身份、职务、部门、作用域规则授予权限点。

建议把以下概念分开：

- 账号身份：用户是谁，如何登录。
- 协会身份：是否为协会会员、协会成员、部门成员。
- 部门归属：宣传部、基管部、项目部、运维部。
- 职务身份：部长、副会长、会长、指导老师等。
- 系统权限：能不能执行某个功能。
- 作用域权限：只能管理本部门、某个项目、某个场地，还是全局。

“组织管理权限”说明：

- 它不是新的身份代号，也不是 `998/999`。
- 它是一组可以授予现有用户的权限点，用于维护成员资料、部门归属、职务身份等组织基础数据。
- 已确认可以保留这组权限，由会长授予可信用户。

### 1.6 小程序改造范围

已确认方向：

- 如果新后端不兼容旧接口，小程序至少需要重构 API 层和数据适配层。
- 小程序不一定需要完全重写 UI。
- 小程序源码可以预留在 `apps/miniapp`，微信开发者工具可以直接打开该子目录。
- 小程序接入顺序排在成员网页端验证核心业务逻辑之后。
- 如果只是统一接口契约和字段，很多页面可以保留 UI，只换 API 调用和数据映射。
- 如果借用、积分、项目、值班、社区等流程重新设计，对应页面需要同步重做。
- 当前仓库没有小程序源码，所以还不能判断具体页面改造量。

建议策略：

1. 先统一小程序 API 客户端，所有请求走同一层。
2. 为小程序定义新接口契约，不继续让页面直接适配旧响应。
3. 按业务域逐个替换页面数据来源。
4. 能保留 UI 的页面先保留，业务流程变化大的页面再重做。
5. 如果旧小程序必须继续上线运行，再考虑临时兼容层。

### 1.8 积分制度

已确认方向：

- 积分相当于协会内部货币。
- 每个用户都应有积分账户，外部成员 `0` 也有积分账户。
- 积分变动必须有流水。
- 不允许业务代码直接修改余额。
- 人工调整必须写原因。
- 重要积分操作必须进入审计。
- 同一业务事件发放积分必须幂等，不能重复发放。
- 不允许用户之间转账。
- 不允许积分透支。
- 积分永久有效。
- 积分数量是核心。
- 积分等级、称号可以作为后续激励能力规划，但不是第一优先级。
- 每周积分领取和固定积分规则第一版可以先统一用 `10` 作为简单占位值调试，但必须做成可配置规则，不能硬编码在业务代码里。
- 用户提交相关表单时，应该显示积分余额、预计扣除或冻结额度。
- 借用押金采用冻结余额机制。
- 冻结余额不是扣除积分，而是暂时锁住一部分积分。
- 正常归还后解冻；逾期、损坏或丢失时，冻结积分的一部分或全部可以转为扣除。
- 普通固定规则内的积分发放应自动化，例如日常任务固定积分。
- 涉及改动积分规则、创建临时积分规则、发放规则之外的积分、人工补发和人工扣分，都必须进入受控审批或系统兜底流程。
- `998/999` 不作为日常积分规则审批角色，只负责系统级兜底、异常修复和受控人工调整。
- 积分规则审批默认由会长持有；指导老师可以参与复核；副会长可由会长授权参与审批。
- 特殊非模板任务不能直接发布，需要先提交临时积分规则申请；审批通过后生成一次性任务模板，再由任务发布人使用该模板发布任务。
- 临时积分规则至少需要记录任务原因、积分数、适用对象、最大发放人数、总积分上限、有效期、申请人、审批人和审批理由。
- 积分规则、临时积分规则、一次性任务模板的创建、审批、发布、撤回都必须保留日志。
- 已发布的积分规则或临时积分规则允许撤回；撤回不能删除原记录，只能标记撤回并记录撤回原因、撤回人、撤回时间和影响范围。
- 临时积分规则撤回不等于自动追回已经发放的积分；默认含义是停止该规则后续继续使用。
- 只有规则错误、重复发放、恶意滥用或审批失误等明确异常场景，才允许追回已经发放的积分。
- 如果确实需要追回已经产生的积分流水，必须通过反向流水修正，不能直接删除原流水。
- 临时积分规则撤回以及由撤回引发的任务取消、待处理、积分修正，都需要通知受影响用户。
- 后台人工发放积分前需要开会讨论，量化贡献后再发放。
- 违规扣分不需要二次确认，但需要允许撤回。撤回应该通过反向流水修正，而不是删除原流水。
- 违规扣分撤回的最终执行由 `998/999` 处理；指导老师可以参与积分异常的审核或复核。
- 积分获取和消耗需要后续进行数学建模，避免滥用和积分通货膨胀。

已确认的积分来源或消耗方向：

- 每周领取积分，按身份或职务区分。
- 打扫卫生或大扫除可以获得积分。
- 日常任务完成可以获得固定积分。
- 宣传链接通过、审核、审批等贡献可以获得积分。
- 审批失误、资源丢失、逾期未归还、损坏物资、恶意占用工位等可以扣积分。
- 借物、场地或工位押金通过借用流程处理，不属于“积分兑换审核”。
- 3D 打印额度等可以作为积分消费场景。
- 3D 打印积分消费第一版不需要库存、次数或额度限制，但需要支持“维护中/关闭消费”开关。
- 3D 打印、服务器借用等服务型资源主要由运维部维护和操作，对应的服务开关也应授予运维部相关权限。
- 3D 打印和服务器借用第一阶段保留业务域和接口占位，但不作为核心闭环优先实现。
- 服务型资源更像“服务订单/接单执行”，不是普通业务审批。
- 3D 打印流程倾向于：用户提交切片文件，系统按配置核算积分，运维负责人接单后打印，完成后通知申请人领取。
- 3D 打印字段后续再和运维部对接；当前不在核心重构闭环中强行定死。
- 服务器借用流程倾向于：按租用服务扣积分，可按时长或周期计费。
- 服务器借用申请字段后续再和运维部对接；当前不在核心重构闭环中强行定死。
- 3D 打印和服务器借用价格应由系统核算，用户提交时就能看到预计消耗，不需要“待计价”“待确认”状态。
- 3D 打印接单后冻结积分，打印完成后扣除积分。
- 3D 打印取消后释放冻结积分；未接单时用户可以取消，接单后需要联系运维处理人取消。
- 服务器借用按时间或周期扣积分。
- 服务器借用需要到期前提醒。
- 服务器借用没有传统逾期概念，可以保留 3 天宽限期；宽限期结束后直接释放对应实例。

### 1.9 场地和工位

当前工位类型调整为四类：

| 类型 | 积分变化 | 是否允许放个人物品 | 当前理解 |
| --- | --- | --- | --- |
| 流动工位 | 增加积分或额度 | 不允许 | 可通过 NFC 或后续视觉方案记录使用和值班情况 |
| 流动固定工位 | 扣积分 | 允许 | 包含部长、会长等固定使用诉求；促进成员在三个场地之间流转 |
| 管理工位 | 不扣积分，不加积分 | 允许 | 每个场地仅一个；只有会长或副会长可以申请；不是自动分配 |
| 项目工位 | 不直接影响积分 | 允许 | 通过项目或工位申请获得，需要关联项目并签订开源协议 |

已确认方向：

- 工位不是普通场地资源，需要独立字段描述类型、积分规则、占用规则、是否可存放个人物品、是否需要冷却期。
- 场地借用后续基本会收敛为工位申请。
- 不同工位类型需要不同字段，例如项目工位需要项目编号，管理工位不需要项目编号。
- 流动工位不需要用户填写申请；它更接近值班或到场使用场景，后续通过 NFC、视觉检测等方式记录使用并结算积分。
- 流动固定工位冷却期暂定 3 个月。
- 流动固定工位一次最长申请 3 个月，冷却期与申请时长对等。
- 管理工位按会长或副会长职务长期有效，不单独设置固定到期时间；职务变更后自动取消或到期。
- 项目工位周期与项目周期绑定。
- 项目工位原则上不允许随意换场地或换工位，但需要保留受控换位能力。
- 项目工位换位由项目负责人发起，需要重新审核。
- 项目工位换位后原工位需要立即释放。
- 项目工位换位原因必须记录。
- 当前场地名称可预填为 `101`、`i创街`、`208`，但场地可能变化，只能作为初始数据。
- 基地/场地/工位管理属于基管部，不属于运维部。
- 硬件相关能力暂时不考虑，仅保留业务域入口。
- NFC、WiFi 在线时长、视觉监控等打卡或检测能力放到后续增量。

### 1.10 资源和物资

已确认方向：

- 物资列表和物资详情供小程序查看。
- 物资详情建议展示名称、分类、剩余数量、描述、位置、柜号、层数、可借状态、维护/下架状态、借用规则提示。
- 物资新增、编辑、库存调整属于基管部日常管理。
- 物资类别删除或整类下架风险更高，应限制为基管部 `2` 及以上。
- “调整物资库存”指修改某种物资的数量，用于补货、盘点、报废、丢失修正；它不是删除物资类别。
- “维修中、停用、下架”等物资状态可以并入物资编辑或下架功能，不必强行拆成复杂模块。
- 资源操作日志当前范围指场地和物资的新增、修改、数量调整、下架、删除等记录。

### 1.11 借用和预约

已确认方向：

- 物资借用和场地/工位借用的数据格式不同，不能粗暴合成一张完全相同的业务表。
- 可以统一生命周期和通用接口概念，例如申请、审批、取消、归还、逾期、押金、审计。
- 物资借用取消需要保留取消记录。
- 借用需要积分押金或信用分。
- 取消记录至少需要保存取消人、取消时间、取消原因、取消前状态。
- 物资借用和场地/工位借用都属于基管部业务。
- 查看全部借用记录应限制为 `2` 及以上。
- 物资借用审核可以由基管部 `1` 及以上处理。
- 场地/工位借用审核建议由基管部 `2` 及以上处理。
- 普通归还可以轻量处理；涉及损坏、丢失、扣分、异常记录时应进入后台管理系统。
- 借用规则配置由会长 `4` 和基管部 `2` 及以上处理。
- 通过项目申请物资或工位时，可以走项目相关减免规则，不直接扣普通个人积分。
- 项目相关的物资借用或工位申请需要关联项目，并要求负责人签署开源协议。
- 项目借用必须关联已经通过项目申请并获得项目编号的项目。
- 项目借用字段应尽量从项目和个人资料自动带出，避免申请人手填造成不一致。
- 个人借用不需要填写指导老师信息；项目借用的指导老师信息应从项目资料自动带出。
- 项目借用完全免扣个人积分，不额外占用个人积分。
- 项目负责人完成开源协议签署并通过项目申请后，后续该项目借用可基于项目编号走项目借用路径。
- 项目借用主要靠开源协议约束，一般问题不做积分惩罚。
- 违反开源协议或其他重要规定时，不只是项目拉黑，项目负责人和成员也应一起进入限制或封禁状态，不允许再向协会提起申请。
- 重大破坏或重要违规由拥有项目风控处理权限的人员结合实际情况手动判断；该权限只授予项目部部长及以上权限，需要系统级封禁时由 `998/999` 兜底执行。
- 项目、负责人和成员的限制或封禁允许申诉。
- 封禁或限制解除权限限制为会长及以上。
- 项目风控记录需要保存原因、证据、处理人、处理时间、影响范围和解除条件。
- 项目申诉记录需要保存申诉说明、附件、处理意见、处理人和处理时间。
- 项目中断记录需要保存中断原因、剩余资源处理、恢复条件和审核意见。
- 项目中断后，无法收回或已经损耗的借出物资可以由基管部部长及以上标记为已损耗，并从库存中扣除。

- 第一版可以保留逾期扣分公式的配置占位，但默认不启用具体扣分公式。

### 1.12 项目

已确认方向：

- 项目创建后进入待审核，后续可以显式拆分草稿和提交审核。
- 项目结项需要审核。
- 项目结项可以被打回。
- 项目结项只需要一个有权限的审核人审核通过即可，不需要多方会签。
- 项目招募需要申请加入，但可以先作为后续待办。

- 项目成员第一版只保留“负责人”和“成员”，不扩展维护者、硬件负责人等复杂角色。

- 项目类型第一版保留个人项目和比赛项目，用于区分兴趣项目和比赛项目；协会项目暂缓。
- 项目完成是否奖励积分后续讨论。
- 项目团队需要每 3 个月提交一次项目进展。
- 项目进展至少需要填写一种材料，例如文字、图片、文件、链接或视频。

- 项目进展到期前 7 天开始提醒。
- 项目进展逾期 1 周再次提醒，逾期 2 周严重提醒，逾期 3 周作为最后期限。
- 项目进展逾期 1 个月后直接进入黑名单或限制名单。
- 项目进展逾期不扣积分，直接按提醒、限制、拉黑处理。
- 项目进展黑名单或限制名单只能由会长及以上权限解除。
- 项目进展不对外展示，但项目结项后需要对外展示。
- 项目材料需要长期归档，所有材料默认永久保存。
- 项目材料上传权限应限制为项目负责人。

- 项目工位申请只需要支持上传材料和展示文字提示，不需要系统固定材料类型。
- 项目工位申请应在工位申请中选择项目编号，从而和项目关联。
- 项目工位审核不应在项目域重复建一套流程，应归入工位/场地借用审核。

- 项目立项不需要新增“预期成果”字段。
- 开源协议由协会拟定，项目负责人需要下载、签字后上传签署文件。
- 开源协议签署文件限制为 PDF。
- 开源协议签署文件和协议版本都需要保存。
- 项目需要有“中断”状态，由项目部人员负责管理。
- 负责人毕业、退会或失联等异常情况，除特殊原因外按照项目进展逾期规则处理。
- 项目中断后，项目工位可以直接释放。
- 项目中断后，已经借出的资源需要按资源类型和实际情况处理；项目工位直接释放，无法收回或已经损耗的借出物资可以由基管部部长及以上标记为已损耗，并从库存中扣除。

### 1.13 工作台、任务和值班

已确认方向：

- 所有部门都需要排班。
- 排班基本单位是任务。
- 所有部门都支持自由发布任务。
- `2` 及以上可以发布任务。
- 发布指定任务时，原则上只能高权限给低权限发布，不能平级发、给自己发、向上发。
- 会长和指导老师的发任务权限不受上述上下级限制。
- 指定任务不需要接收，发布后直接安排，并通过微信服务通知提醒。
- 指定任务发布后直接进入“待完成”，不需要单独保留“已发布”状态。
- 悬赏任务可以不指定人，需要“待领取”状态；被领取后直接进入“待完成”，不需要单独保留“已领取”状态。
- 悬赏任务需要区分协会内可见和公开可见；`0` 外部成员只能领取公开任务。
- 任务完成流程应拆成“执行人提交完成申请或材料”和“任务发布人审核完成”。
- “待审核”表示执行人已经提交完成材料，等待任务发布人审核；它比“待确认”更清晰。
- 最终标记任务完成的人应是任务发布人。
- 宣传部的文案、推文、新闻稿是便捷任务类型；不是这三类时使用自由任务。
- 宣传链接即秀米链接，通常由宣传部长派任务给干事做，再由部长审核。
- 值班申请是有人发布值班消息，然后成员报名。
- 值班消息需要和场地关联，不需要绑定部门。
- 值班消息可以选择多个场地。
- 多场地值班的人数上限由各场地的流动工位数量决定。
- 成员报名多场地值班时需要选择具体场地。
- 同一个值班消息下，一个成员只能报名一个场地。
- 值班报名后允许退出。
- 值班人数满员后自动关闭报名。
- 值班记录正常业务中不开放随意增删改查，只有错误修正时才由 `998/999` 介入。
- 打扫卫生或大扫除属于统一任务的一种任务类型，不单独建立清洁业务域。
- 悬赏任务发布人可以选择公开范围。
- 允许公开给 `0` 外部成员的任务类型后续由积分规则审批权限在模板配置中维护。
- 外部成员完成公开悬赏任务后可以获得积分，同时保留完成记录。
- 部门工作应在部门内入口发布，包括部门内悬赏工作和指定工作。
- 能公开的工作应在外部入口发布，属于公开悬赏工作。
- 任务模板按当前理解分为三类：部门指定任务、协会指定任务或协会悬赏任务、对外悬赏任务。
- 悬赏任务类型和积分条目只能由积分规则审批权限维护，不由普通任务发布人临时修改。
- `998/999` 只负责系统级兜底，不作为日常任务模板和积分规则审批角色。
- 悬赏任务类型和积分条目修改后，只对新任务生效，不影响已经发布的任务。
- 发布任务时必须完全使用模板积分，不能在模板基础上临时改积分。
- 特殊非模板任务不能直接发布，需要先提交临时积分规则申请；审批通过后生成一次性任务模板。
- 临时积分规则和一次性任务模板必须保留审批日志，并允许撤回；异常需要追回积分时通过反向流水修正。
- 临时积分规则撤回默认只停止后续使用，不自动追回已经发放的积分。
- 只有规则错误、重复发放、恶意滥用或审批失误等明确异常场景，才允许通过反向流水追回积分。
- 临时积分规则撤回后，未领取任务直接关闭；已领取未提交任务通知执行人并取消；待审核任务进入“规则撤回待处理”；已完成且已发放积分的任务默认保留积分和完成记录。

任务状态建议：

| 任务类型 | 状态流转 |
| --- | --- |
| 指定任务 | 待完成 -> 待审核 -> 已完成；异常分支为已打回、已取消 |
| 悬赏任务 | 待领取 -> 待完成 -> 待审核 -> 已完成；异常分支为已打回、已取消 |

说明：

- “已发布”更适合作为操作记录或发布时间，不适合作为指定任务的业务状态。
- “已领取”也更适合作为操作记录；领取完成后，任务对执行人来说就是“待完成”。

### 1.15 文档和运维

已确认方向：

- 运维手册第一版需要覆盖本地开发、测试环境、预发布环境、生产环境和灾备恢复。
- 需要发布记录模板。
- 需要故障记录模板。
- 系统目标是高可用系统。

VitePress 说明：

- VitePress 是一个基于 Markdown 的文档网站生成工具。
- 写文档时主要还是写 `.md` 文件。
- 它可以把 Markdown 自动生成一个静态文档网站。
- 它适合放在仓库内部，和代码一起版本管理。
- 它支持导航、侧边栏、全文搜索、代码高亮和静态部署。
- 对当前项目来说，它适合作为开发文档、接口说明、运维手册、发布记录、故障记录的文档站底座。

## 二、已处理的冲突和重复

| 冲突或重复 | 处理结果 |
| --- | --- |
| 副会长到底是标签还是权限等级 | 调整为协会职务，参与权限继承，高于部长低于会长，同时可兼任部长 |
| 管理员和超级管理员是否参与业务审批 | 不参与日常业务审批，只作为底层管理身份；999 只比 998 多指定管理员能力 |
| 指导老师是否属于普通权限等级 | 不属于普通层级，作为特殊身份，通过审核类权限点授权 |
| 网页端是否必须使用微信登录 | 不必须；第一版需要账号密码登录，后续再绑定微信登录 |
| 小程序用户是否也要账号密码 | 不需要，小程序继续微信无感登录 |
| 本地账号登录名 | 采用邮箱，通过邮箱验证码绑定和重置密码 |
| 邮箱验证码规则 | 有效期 5 分钟；同一邮箱 1 小时最多发送 10 次；每次重新请求至少间隔 1 分钟 |
| 是否允许更换邮箱 | 允许；需要验证旧邮箱和新邮箱；更换后密码不变 |
| 是否允许解绑微信 | 第一版不允许解绑微信 |
| `999` 是否可以有多个 | 全系统只能有一个，主要用于指定或恢复 `998` |
| 工位类型中是否单独保留部长/会长工位 | 不单独保留，归入流动固定工位 |
| 场地借用和工位申请是否重复 | 后续场地借用基本收敛为工位申请，不同工位类型使用不同字段 |
| 项目工位审核是否属于项目域 | 不重复建流程，归入工位/场地借用审核；项目域只提供项目编号和材料关联 |
| 服务型资源是否属于审核 | 不按普通审批理解，更像系统核算积分、运维部接单执行服务 |
| 服务型资源是否需要待计价/待确认 | 不需要，系统提交时核算并展示预计积分 |
| 服务型资源是否进入核心闭环 | 第一阶段只保留业务域和接口占位，具体字段后续和运维部对接 |
| 服务器借用是否有逾期 | 不按逾期处理，保留 3 天宽限期，结束后释放实例 |
| 积分兑换是否需要审核 | 普通兑换不需要独立审核；借物、场地、工位的审核属于借用流程 |
| 物资维护状态是否单独成复杂功能 | 不强行复杂化，可并入物资编辑、下架和删除 |
| 任务完成由谁标记 | 执行人提交完成申请，任务发布人审核并最终标记完成 |
| 任务是否需要“已发布”“已领取”“待确认” | 指定任务发布后直接待完成；悬赏任务领取后直接待完成；“待确认”改名为“待审核” |
| 值班记录是否开放 CRUD | 正常业务不开放随意 CRUD，只有错误修正时由 998/999 介入 |
| 打扫卫生是否单独成域 | 不单独成域，作为统一任务类型处理 |
| 流动工位是否需要申请 | 不需要，作为到场/值班使用场景，后续由 NFC 或视觉方案记录 |
| 项目负责人是否允许变更 | 第一版不允许变更负责人 |
| 项目进展逾期是否扣积分 | 不扣积分，按提醒、限制、拉黑处理 |
| 开源协议上传格式 | 限制为 PDF |
| 项目中断后是否释放资源 | 工位直接释放；无法收回或已损耗的借出物资由基管部部长及以上标记为已损耗并扣减库存 |
| 值班消息是否只能绑定一个场地 | 可以选择多个场地 |
| 同一成员能否在同一值班消息报名多个场地 | 不可以，只能报名一个场地 |
| 任务发布时能否临时改模板积分 | 不可以，必须完全使用模板积分；特殊非模板任务需要先申请临时积分规则 |
| 特殊非模板任务由谁审核 | 默认由会长审批，指导老师可复核，副会长可由会长授权；`998/999` 只做系统兜底 |
| 临时积分规则是否允许撤回 | 允许；必须保留日志；撤回默认停止后续使用，不自动追回已发积分 |
| 临时积分规则撤回后是否追回积分 | 默认不追回；只有规则错误、重复发放、恶意滥用或审批失误等异常场景才通过反向流水修正 |
| 临时积分规则撤回是否需要通知用户 | 需要通知受影响用户，包括已领取、待审核、已完成或已获得积分的用户 |

## 四、后续细化问题

本节只保留不阻塞第一阶段开工、但实现前还需要补充细节的问题。已经确认的回答已经移动到前面的“已确认需求”和“权限功能编号”。

### 4.1 服务型资源

Q001：服务订单字段后续再和运维部继续对接。

当前已确认：

- 3D 打印和服务器借用第一阶段只保留业务域和接口占位。
- 这部分不如核心的借物、借场地和项目重要，不作为当前核心重构闭环。
- 3D 打印和服务器借用价格由系统核算，用户提交时就能看到预计积分。
- 不需要“待计价”“待确认”状态。
- 服务订单状态采用待接单、执行中、待领取或使用中、已完成、已取消。
- 3D 打印接单后冻结积分，打印完成后扣除积分。
- 3D 打印取消后释放冻结积分；未接单时用户可以取消，接单后需要联系运维处理人取消。
- 服务器借用按时间或周期扣积分。
- 服务器借用需要到期前提醒，没有传统逾期概念，允许拖 3 天，3 天后释放对应实例。

后续需要继续确认：

- 3D 打印是否还需要材料类型、颜色、备注等字段，以及这些字段是否由用户填写还是运维接单时补充。
- 服务器借用需要哪些配置字段，例如用途、规格、系统镜像、周期、访问方式；这部分暂时留空，等运维部确认。

### 4.2 临时积分规则

Q002：临时积分规则的表单和撤回处理规则。

当前已确认：

- 任务模板按当前理解分为三类：部门指定任务、协会指定任务或协会悬赏任务、对外悬赏任务。
- 普通任务必须完全使用模板积分，发布人不能临时改积分。
- 特殊非模板任务不能直接发布，需要先提交临时积分规则申请。
- 临时积分规则审批通过后生成一次性任务模板。
- 临时积分规则默认由会长审批；指导老师可以复核；副会长可以由会长授权；`998/999` 只做系统兜底。
- 临时积分规则必须保留审批、发布、撤回日志。
- 临时积分规则允许撤回；撤回默认停止后续使用，不自动追回已经发放的积分。
- 临时积分规则撤回需要通知受影响用户。
- 只有规则错误、重复发放、恶意滥用或审批失误等明确异常场景，才允许追回已经发放的积分。
- 如果确实需要追回已经产生的积分流水，必须通过反向流水修正，不能删除原流水。

建议申请表单字段：

- 基础信息：规则名称、任务类型、所属部门或协会级别、申请人、申请原因、为什么不能使用现有模板。
- 积分信息：单次完成积分、最大发放人数、总积分上限、有效期。
- 任务要求：适用对象、完成条件、需要提交的材料、审核人或审核范围。
- 风控信息：是否公开给外部成员、是否允许多人完成、是否允许重复领取、撤回影响说明。

撤回后的任务处理建议：

| 任务状态 | 处理方式 |
| --- | --- |
| 未领取 | 关闭任务，展示为“规则已撤回，任务已关闭”，不产生积分 |
| 已领取但未提交 | 通知执行人，任务标记为“规则撤回取消”，不产生积分；如工作已经实际发生，发布人可重新申请临时积分规则补偿 |
| 已提交待审核 | 进入“规则撤回待处理”，暂停审核和发分；由审核人判断取消、打回，或重新申请临时积分规则 |
| 已完成但未发分 | 暂停发分，进入“规则撤回待处理”；确认任务仍有效后可用新规则结算 |
| 已完成且已发分 | 默认保留积分和完成记录；只有规则错误、重复发放、恶意滥用或审批失误等异常场景，才通过反向流水追回积分 |
