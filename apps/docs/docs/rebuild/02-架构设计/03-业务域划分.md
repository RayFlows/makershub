# MakersHub 业务域划分

## 目标

MakersHub 不是一个简单的小程序后端，而是开源硬件协会长期使用的平台。它现在需要支撑小程序和统一权限体系，后续还要扩展新的后台管理、博客、社区、电控、AI、设备管理、积分流通、项目协作和更多协会内部运营能力。

因此重构不能只围绕“把旧接口重新写一遍”，而应该采用严格的增量模型：每个增量交付一个可以运行、可以验收、可以继续扩展的业务闭环。

## 核心原则

**部门不是代码模块，业务域才是代码模块。**

协会目前有四个部门：

- 宣传部；
- 基管部；
- 项目部；
- 运维部。

这些部门应该进入组织和权限模型，而不应该直接变成后端模块名称。因为一个功能经常跨部门，例如：

- 活动发布主要由宣传部负责，但项目展示可能和项目部有关；
- 物资、场地属于基管部管理，但借用审批会涉及用户、资源、审批、通知和审计；
- 运维部后续可能负责电控、设备状态和系统维护，但这些不应该和普通任务模块混在一起。

所以代码应该按业务域划分，而不是按部门划分。

业务域也不是最后一级目录。每个业务域内部必须继续按能力、聚合或状态机拆分，
避免重构后再次形成巨大的 `service.py` 和 `repository.py`。具体规则见
[后端业务域内部架构](./04-后端业务域内部架构.md)。

## 顶层业务域

| 业务域 | 负责 | 不负责 | 当前功能来源 | 未来扩展 |
| --- | --- | --- | --- | --- |
| 身份与权限 | 登录、账号、会话、角色、权限 | 部门业务规则、项目业务规则 | 用户登录、微信登录、后台登录 | 多登录方式、统一认证、通行密钥 |
| 组织与成员 | 部门、成员、职务、协会身份、成员状态 | 登录凭证、资源审批逻辑、积分账本 | 用户资料、成员列表、部门字段 | 成员成长、内部人事 |
| 积分与账本 | 积分账户、积分流水、冻结、结算、规则、兑换 | 用户身份、部门组织、业务事实本身 | `users.score` 字段 | 协会内部货币、贡献激励、兑换权益 |
| 资源 | 物资、场地、设备、资产元数据、库存状态 | 借用流程、审批流程 | 物资、场地 | 工具、设备、实验室资产、IoT 设备 |
| 借用 | 借用申请、审批、归还、冲突检测、借用记录 | 资源主数据、用户身份 | 物资借用、场地借用 | 统一预约、审批流 |
| 项目 | 项目申请、成员、审核、材料、结项、招募 | 用户账号管理、普通内容发布 | 项目相关接口 | 项目展示、阶段里程碑、经费/奖项 |
| 工作台 | 任务、排班、值班申请、值班记录、内部运营 | 官方内容、项目生命周期 | 任务、排班、值班相关空模块 | 考勤、工作量、运营看板 |
| 内容 | 活动、宣传链接、公告、官方博客 | 社区审核、项目审核 | 活动、宣传链接 | 博客、新闻、活动归档 |
| 社区 | 帖子、评论、点赞、收藏、举报、社区审核 | 官方公告、账号登录 | 暂未实现 | 论坛、问答、项目讨论 |
| 电控 | 硬件设备、控制命令、遥测、设备访问策略 | 普通借用流程，除非共享设备资源 | 暂未实现 | 门禁、电源控制、传感器、实验室自动化 |
| AI | 知识库、AI 助手、推荐、自动检查 | 核心权限来源、业务事实来源 | 暂未实现 | AI 问答、项目助手、智能审核 |
| 通知 | 消息模板、站内信、订阅消息、投递状态 | 业务决策本身 | 零散/未实现 | 微信订阅消息、邮件、系统通知 |
| 审计 | 操作日志、审批轨迹、安全事件、重要数据变更 | 实际业务修改的归属 | 零散日志 | 合规追踪、误操作恢复、管理追责 |

## 业务域说明

### 身份与权限

身份与权限负责“用户是谁”和“用户能做什么”。

它应该提供：

- 微信小程序登录；
- 后台账号密码登录；
- 账号绑定；
- JWT 或会话签发；
- 角色和权限检查；
- 初始化超级管理员；
- 权限点注册表。

它不应该知道项目如何审核，也不应该知道借用如何审批。它只负责回答用户是否拥有某个能力，例如：

- `project.audit`；
- `borrowing.resource.review`；
- `content.event.publish`；
- `device.command.execute`。

建议概念：

- 用户 `User`；
- 账号 `Account`；
- 角色 `Role`；
- 权限 `Permission`；
- 用户角色 `UserRoleGrant`；
- 会话 `Session`。

### 组织与成员

组织与成员负责协会内部结构和成员身份。

它应该提供：

- 部门；
- 成员资料；
- 成员状态；
- 职务；
- 部门成员关系；
- 成员搜索；
- 成员花名册。

四个部门建议建成稳定标识：

- `publicity`：宣传部；
- `base_management`：基管部；
- `project`：项目部；
- `operations`：运维部。

部门可以影响权限，但不能替代权限系统。例如，一个基管部成员不一定天然拥有 `resources.manage`，这应该由角色或权限授予。

组织与成员可以展示成员积分余额，但不应该直接修改积分。积分变更必须进入积分与账本域。

### 积分与账本

积分制度是协会内部的“货币系统”，不能只作为 `users.score` 这样的普通字段处理。

它应该提供：

- 积分账户；
- 当前余额；
- 可用余额；
- 冻结余额；
- 积分流水；
- 积分发放；
- 积分扣减；
- 积分冻结和解冻；
- 积分兑换；
- 积分规则；
- 人工调整；
- 审计追踪。

积分变更必须采用账本模型。原则上不允许业务代码直接改用户表上的积分余额，而应该追加一条积分流水，再由账本服务计算或更新余额缓存。

建议概念：

- 积分账户 `PointAccount`；
- 积分流水 `PointLedgerEntry`；
- 积分规则 `PointRule`；
- 积分冻结 `PointHold`；
- 积分兑换 `PointRedemption`；
- 积分调整申请 `PointAdjustment`。

典型积分来源：

- 完成任务；
- 完成值班；
- 完成项目阶段；
- 参与活动；
- 贡献文档或教程；
- 社区优质内容；
- 管理员人工奖励。

典型积分消耗：

- 兑换物资或权益；
- 借用押金或信用冻结；
- 活动报名消耗；
- 违规扣分；
- 兑换协会周边或服务。

积分域可以监听其他业务域的事件，例如 `task.finished`、`duty.record.approved`、`project.closed`，但不能让其他业务域直接写积分余额。

因为积分相当于协会内部货币，所以它必须从第一阶段就具备：

- 明细可追溯；
- 余额不可随意覆盖；
- 幂等处理；
- 人工调整必须有原因；
- 高风险操作必须进入审计。

### 资源

资源负责协会拥有的“东西”。

它应该提供：

- 物资分类；
- 物资条目；
- 场地和工位；
- 设备元数据；
- 库存数量；
- 可用状态；
- 维护状态。

资源不负责借用审批。审批属于借用域。

资源应该暴露稳定的资源 ID，客户端不应该用字符串拼接来表达借用物资。比如旧代码里的 `分类 - 名称 - 数量` 应该改成结构化数据：

```json
{
  "items": [
    { "resource_id": "xxx", "quantity": 2 }
  ]
}
```

### 借用

借用负责共享资源的使用生命周期。

它应该提供：

- 借用申请；
- 预约；
- 审批；
- 取消；
- 归还；
- 冲突检测；
- 逾期状态；
- 借用历史；
- 审批审计轨迹。

当前的 `stuff-borrow` 和 `sites-borrow` 可以先作为借用域里的两个子流程，也可以最终收敛成统一模型：

- 借用申请 `BorrowApplication`；
- 借用条目 `BorrowItem`；
- 审批记录 `BorrowReview`；
- 归还记录 `BorrowReturn`。

长期看统一模型更好，但实现可以增量推进：

1. 先重建物资借用；
2. 再重建场地借用；
3. 等重复逻辑稳定后抽出统一工作流。

### 项目

项目负责项目生命周期。

它应该提供：

- 项目创建；
- 成员管理；
- 立项审核；
- 招募状态；
- 项目材料；
- 结项申请；
- 项目归档。

项目成员不是协会部门成员。项目内部应该有项目作用域角色，例如：

- 项目负责人；
- 维护者；
- 成员；
- 指导老师。

### 工作台

工作台负责协会内部日常运营。

它应该提供：

- 任务；
- 任务分配；
- 排班；
- 值班申请；
- 值班记录；
- 工作量统计。

这个业务域主要服务协会内部管理，需要依赖组织域的成员数据，也需要依赖身份域的权限检查。

### 内容

内容负责官方发布。

它应该提供：

- 活动；
- 活动海报；
- 宣传链接；
- 公告；
- 官方博客；
- 内容审核。

内容必须和社区分开。内容是官方发布，社区是用户生成内容。

### 社区

社区用于后续用户互动。

它可能提供：

- 帖子；
- 评论；
- 点赞；
- 收藏；
- 举报；
- 审核队列。

社区可以依赖身份和组织，但不能直接修改成员身份。

### 电控

电控用于实验室自动化和硬件控制。

它可能提供：

- 设备注册；
- 控制命令；
- 遥测数据；
- 设备访问权限；
- 操作日志；
- 紧急停用规则。

这个业务域会控制真实硬件，所以从第一天开始就必须有严格权限和审计。

### AI

AI 用于智能化能力。

它可能提供：

- 知识库；
- AI 助手；
- 项目推荐；
- 智能搜索；
- 内容自动检查；
- 流程建议。

AI 不能成为业务事实来源。它应该通过明确接口读取业务数据，写回动作必须走可审计的应用服务。

### 通知

通知是共享支撑域。

它应该提供：

- 通知模板；
- 用户站内信；
- 微信订阅消息；
- 邮件或短信集成；
- 重试和投递状态。

业务域不应该直接发送消息，而应该通过事件或应用服务请求通知。

### 审计

审计是共享支撑域。

它应该提供：

- 管理员操作日志；
- 审批历史；
- 安全敏感事件；
- 电控操作日志；
- 重要数据变更记录。

审计记录默认应该只追加，不随意修改。

## 推荐项目结构

`makershub` 采用单仓库多应用结构。后端、成员网页端、后台管理端、文档站和小程序源码放在同一个仓库中，前端共享包统一放在 `packages/` 下。

完整目录规划见 [项目结构规划](../03-工程运维/01-项目结构规划.md)。

顶层结构建议如下：

```text
makershub/
  apps/
    api/
    web/
    admin/
    docs/
    miniapp/
  packages/
    api-client/
    ui/
    config/
  infra/
    docker/
    nginx/
    mysql/
    minio/
  scripts/
  README.md
  pnpm-workspace.yaml
  pyproject.toml
```

每个模块建议包含自己的说明文档：

```text

  README.md

  schemas.py
  repository.py
  service.py
  permissions.py

```

```text
modules/borrowing/
  README.md
  applications/
  reviews/
  returns/
  conflicts/
```

### 前端结构

当前仓库里的旧后台管理系统不建议迁移。新后台管理端直接重写，不把旧 React 后台按原样搬过去。

旧后台的 React 和 Ant Design 经验可以保留，但工程底座升级为 React、Vite、TypeScript 和 Ant Design Pro 风格。网页端和后台管理端拆成两个应用，共享 API 客户端、类型和 UI 组件。

```text
apps/
  web/
  admin/

packages/
  api-client/
  ui/
  config/
```

成员网页端 `apps/web` 第一阶段优先实现，用来验证登录、物资借用、积分账本等核心业务闭环。它的长期产品形态应拆成公开主站首页 / 社区入口、成员中心和协会工作台三层：公开主站首页面向游客和外部用户，并提供个人借物、团队借物、场地申请、协会活动、项目立项、查看项目等全局业务入口；成员中心面向普通已登录成员；协会工作台只面向协会工作人员或被授予对应权限的用户。网页端是完整上位入口，小程序后续承接高频轻量子集；旧小程序只能作为现有业务入口参照，不能反过来限定网页端能力。BBS-like 讨论区属于后续社区频道，不作为当前主站骨架。

后台管理端 `apps/admin` 长期承接成员管理、权限管理、资源管理、审核、导出、审计和异常处理；第一阶段仅作为端到端主线审核控制台落地。后台访问不等于 `998/999`，业务人员可以进入自己有权限的后台模块。

### 小程序结构

```text
apps/miniapp/
  src/
    app/
    shared/
      api/
      auth/
      components/
      constants/
      utils/
    features/
      profile/
      points/
      resources/
      borrowing/
      projects/
      workbench/
      content/
      community/
```

小程序源码后续可以放在 `apps/miniapp`。微信开发者工具可以直接打开该子目录，不需要打开整个仓库。

小程序、`apps/web` 和后台管理端必须共享同一套 `/api/v1` 业务契约、状态机和业务事实。端侧差异只能体现在身份入口、权限点、页面密度、表单拆分、字段摘要和传输适配，不能变成多套后端数据模型或小程序专用业务语义。

如果新后端不做旧接口兼容，小程序至少需要重构 API 层和数据适配层。页面是否大修取决于业务流程变化：如果只改接口契约，页面可以逐步保留；如果借用、积分、项目、值班等流程也重画，相关页面就需要同步重做。

### 文档网站

文档网站应该作为产品的一部分维护。

建议包含：

- 产品概览；
- 架构说明；
- 业务域说明；
- 权限模型；
- API 契约；
- 数据库模型；
- 部署说明；
- 运维手册；
- 开发指南；
- 发布记录；
- 故障记录。

可选实现：

- VitePress；
- Docusaurus；
- MkDocs Material。

这个项目推荐优先考虑 VitePress，因为它简单、原生支持 Markdown，也方便前端同学维护。

建议放在：

```text
apps/docs/
  docs/
  .vitepress/
```

## 增量交付顺序

第一阶段详细实施顺序和验收标准见 [第一阶段实施路线图](../01-先读/02-第一阶段路线图.md)。

### 增量 0：架构基线

交付内容：

- 业务域划分；
- 目标项目结构；
- 目录 README 规范；
- 文档网站选型；
- `apps/api`、`apps/web`、`apps/admin`、`apps/docs`、`apps/miniapp` 的骨架规划；
- `packages/api-client`、`packages/ui`、`packages/config` 的共享边界；
- 统一 API 契约草案；
- 旧接口清单和迁移矩阵。

以下增量清单是目标态能力拆解，包含旧系统入口和后续规划，不代表当前第一阶段小程序业务页已经迁移；当前实施范围以 `01-先读/02-第一阶段路线图.md` 为准。

### 增量 1：身份、组织与积分基础

交付内容：

- 用户；
- 账号；
- 微信登录；
- 后台登录；
- 角色；
- 权限；
- 部门；
- 成员花名册；
- 初始化超级管理员。
- 积分账户初始化；
- 积分账本基础模型。
- 成员网页端基础登录、个人资料、积分概览页面。

### 增量 2：资源

交付内容：

- 物资分类；
- 物资条目；
- 场地和工位；
- 后台资源管理；
- 成员网页端资源浏览；
- 小程序资源浏览。

### 增量 3：借用

交付内容：

## 后续问题

以下问题不阻塞创建新项目骨架，可以在对应增量开始前继续细化：

- 物资借用、场地借用和工位申请第一阶段保留不同申请字段，但共享生命周期、审批、取消、归还、押金和审计概念；后续根据重复程度抽统一工作流。
- Redis 不作为增量 0 必选项，等通知、限流、后台任务或缓存确实需要时再引入。
- 服务型资源的具体字段后续和运维部对接。
- 临时积分规则的细表单字段和撤回通知文案后续在实现前细化。
- 项目进展的自动提醒和拉黑执行流程可以在项目基础流程稳定后继续细化。
