# MakersHub 当前功能工作流

## 目标

本文档梳理当前代码库和旧 Apifox 接口文档中能看到的功能流程。

它的用途是帮助确认业务域划分是否正确。它不是最终产品需求文档。有些流程已经在代码中实现，有些只出现在旧接口文档中，有些只是因为空模块存在而被推断出来。

## 状态标记

- 已实现：当前仓库中有实际代码。
- 部分实现：有代码，但缺少关键生命周期，或者存在明显不一致。
- 仅旧接口存在：旧 Apifox 文档中有接口，但当前仓库没有实现。
- 空模块：本地文件存在，但模型、路由或服务为空。
- 未来规划：协会后续需要，但当前没有实现。

## 身份与权限工作流

### 微信小程序登录

状态：已实现。

当前接口：

- `POST /users/wx-login`

当前流程：

1. 小程序获取微信临时 `code`。
2. 客户端把 `code` 发给后端。
3. 后端调用微信 `jscode2session` 接口。
4. 后端拿到 `openid`。
5. 后端查询 `users.userid == openid`。
6. 如果用户不存在，创建新用户：
   - `userid = openid`；
   - 生成 `maker_id`；
   - `real_name` 为空；
   - 使用默认头像；
   - 其他字段使用模型默认值。
7. 后端用 `openid` 签发 JWT。
8. 后端返回 token 和 `user_info`。

当前问题：

- 新用户会继承模型默认值，其中包括 `role=1`，这可能导致新微信用户默认成为协会成员。
- 鉴权时没有检查用户 `state`，被封禁用户可能仍可使用 token。
- 登录流程把账号身份和成员资料创建混在一起。

目标归属：

- 身份与权限负责微信账号绑定和 token 签发；
- 组织与成员负责成员状态、部门和资料。

### 后台登录

状态：已实现，但和小程序身份体系割裂。

当前接口：

- `POST /admin/api/login`
- `GET /admin/api/verify`
- `POST /admin/api/logout`

当前流程：

1. 管理员输入用户名和密码。
2. 后端把用户名密码和环境变量比较。
3. 后端签发独立的 admin JWT，包含 `type=admin`。
4. 后台前端保存 token。
5. 后台接口使用独立 admin token 鉴权。
6. 登出只需要前端删除 token。

当前问题：

- 后台管理员不是系统里的真实用户。
- 后台权限没有连接到部门、角色、用户状态。
- 系统存在两套认证。

目标归属：

- 身份与权限应该统一微信账号和后台密码账号；
- 后台访问应该是权限点，例如 `admin.access`。

### 用户资料

状态：已实现。

当前接口：

- `GET /users/profile`
- `PATCH /users/profile`
- `POST /users/profile-photo`

当前流程：

1. 用户使用小程序 JWT 鉴权。
2. 后端按 `userid` 加载用户。
3. 读取资料时返回：
   - 真实姓名；
   - 角色；
   - 手机号；
   - 学院；
   - 学号；
   - QQ；
   - 年级；
   - 账号状态；
   - 头像 URL；
   - 个性签名；
   - 积分。
4. 更新资料时，请求体使用嵌套 `data` 对象。
5. 后端校验学号、QQ、年级等数字字段。
6. 后端更新用户记录。
7. 上传头像时，文件写入 MinIO 的 `AVATARS` 桶，然后更新 `profile_photo`。

目标归属：

- 身份与权限负责鉴权；
- 组织与成员负责成员资料；
- 文件存储属于基础设施。

### 成员花名册

状态：已实现。

当前接口：

- `GET /users/get-makers`

当前流程：

1. 调用方需要 `role >= 2`。
2. 后端查询 `state=1` 且 `role in [1, 2]` 的用户。
3. 后端按数字部门分组。
4. 返回每个成员的姓名和 `maker_id`。

当前问题：

- 权限绑定到 `role>=2`，不是 `member.roster.view` 这样的能力权限。
- 部门是代码里的数字常量。

目标归属：

- 组织与成员负责部门、成员身份、花名册和职务。

### 项目成员搜索

状态：已实现。

当前接口：

- `GET /users/find-by-phonenum`

当前流程：

1. 已登录用户输入部分手机号。
2. 后端用 SQL `LIKE` 搜索 `users.phone_num`。
3. 后端排除当前用户。
4. 返回最多 10 个用户，包含姓名、学院、手机号和 `maker_id`。

目标归属：

- 组织与成员负责成员搜索；
- 项目域在添加成员时消费这个能力。

### 后台用户管理

状态：已实现。

当前接口：

- `GET /admin/api/user/list`
- `GET /admin/api/user/detail/{userid}`
- `PUT /admin/api/user/role/{userid}`
- `PUT /admin/api/user/state/{userid}`
- `PUT /admin/api/user/update/{userid}`
- `GET /admin/api/user/stats`

当前流程：

1. 管理员使用 admin token 鉴权。
2. 管理员可以按角色、状态、部门、关键词筛选用户。
3. 后端返回用户列表、角色统计和部门统计。
4. 管理员可以修改：
   - 角色；
   - 状态；
   - 部门；
   - 积分字段；

目标归属：

- 身份与权限负责角色和权限；
- 组织与成员负责部门、成员资料和协会身份。

## 积分与账本工作流

状态：当前只有 `users.score` 字段，缺少真正的积分制度。

当前代码现状：

- 用户模型中有 `score` 字段；
- 用户资料会返回积分；
- 后台用户管理可以直接修改积分；
- 没有积分流水；
- 没有积分规则；
- 没有冻结、兑换、结算、撤销和审计。

目标定位：

积分相当于协会内部货币，应该作为独立业务域建设，而不是作为用户资料的一部分。

目标流程：

1. 每个用户拥有一个积分账户。
2. 积分账户保存当前余额、可用余额和冻结余额。
3. 所有积分变动都必须追加流水。
4. 流水记录至少包含：
   - 用户；
   - 变动方向；
   - 变动数量；
   - 业务来源；
   - 业务单据 ID；
   - 原因；
   - 操作人；
   - 幂等键；
   - 创建时间。
5. 任务完成、值班通过、项目结项、活动参与、社区贡献等业务可以触发积分发放。
6. 借用押金、兑换权益、违规扣分等业务可以触发积分冻结或扣减。
7. 管理员人工调整积分必须填写原因，并写入审计。
8. 积分流水原则上不可删除，只能通过反向流水冲正。

建议状态和类型：

| 概念 | 示例 |
| --- | --- |
| 流水方向 | 增加、扣减、冻结、解冻、冲正 |
| 业务来源 | 任务、值班、项目、借用、活动、社区、人工 |
| 账户状态 | 正常、冻结、停用 |
| 兑换状态 | 待处理、已完成、已取消、已驳回 |

关键规则：

- 业务域不能直接修改积分余额；
- 积分域可以监听业务事件，也可以提供应用服务供其他域调用；
- 同一个业务事件必须幂等，避免重复发积分；
- 涉及扣减和冻结时，必须检查余额；
- 高风险操作必须记录审计。

目标归属：

- 积分与账本负责账户、余额、流水、规则、冻结和兑换；
- 组织与成员可以展示积分，但不拥有积分变更；
- 审计域记录人工调整和高风险操作；
- 通知域负责积分到账、冻结、兑换完成等消息提醒。

- `GET /stuff/get-all`

当前流程：

1. 已登录用户请求物资列表。
2. 后端读取所有物资。

目标归属：

## 借用工作流

### 物资借用

状态：已实现，但请求契约较弱。

当前接口：

- `POST /stuff-borrow/apply`
- `GET /stuff-borrow/view`
- `GET /stuff-borrow/view-all`
- `GET /stuff-borrow/detail/{sb_id}`
- `POST /stuff-borrow/review`
- `POST /stuff-borrow/return`
- `POST /stuff-borrow/cancel/{sb_id}`
- `PATCH /stuff-borrow/update/{sb_id}`

当前申请流程：

1. 已登录用户提交借用申请。
2. 请求包含申请人信息、归还时间、原因和 `materials`。
3. 个人借用使用 `type=0`。
4. 团队借用使用 `type=1`，可以包含项目编号、指导老师姓名和指导老师电话。
5. 后端生成 `sb_id`。
6. 后端把每个物资字符串按 `分类 - 名称 - 数量` 解析。
7. 后端用分类和名称查找物资。
8. 后端创建 `StuffBorrow`，状态为 `0`。
9. 后端创建关联的 `BorrowItem`。
10. 创建申请时不扣库存。

当前审核流程：

1. 管理员或成员审核申请。
2. 请求使用 `borrow_id`、`action=approve|reject` 和 `reason`。
3. 批准时锁定申请和相关物资行。
4. 后端检查库存是否充足。
5. 如果库存不足：
   - 系统自动打回申请；
   - 写入系统审核意见；
   - 返回业务错误结果。
6. 如果库存充足：
   - 扣减每个物资的 `number_remain`；
   - 申请状态变成 `2`；
   - 保存审核意见。
7. 打回时状态变成 `1` 并保存理由。

旧系统物资归还流程（仅作为迁移参照，不代表新版第一阶段端侧暴露对象）：

1. 旧系统由管理员确认归还。
2. 后端检查申请状态必须是 `2`。
3. 后端把申请状态改成 `3`。
4. 后端再单独恢复物资数量。
5. 如果库存恢复失败，接口返回 `202`，提示人工检查。

当前用户修改和取消流程：

1. 用户只能修改自己的申请。
2. 申请必须是 `0` 或 `1`。
3. 修改物资列表会删除并重建借用条目。
4. 用户只能取消 `0` 或 `1` 状态的申请。
5. 当前取消是直接删除申请，而不是保留“已取消”状态。

当前问题：

- 物资选择是字符串契约，不是结构化数据。
- 归还状态变更和库存恢复分成两步，可能破坏一致性。
- 物资借用和场地借用的审核请求格式不同。
- 物资借用没有明确的取消状态。

目标归属：

- 借用域负责申请、审批、归还、取消和审计；
- 资源域负责物资数量，但借用域通过明确的库存操作来修改数量。

`PATCH /api/v1/borrowing/applications/{application_id}` 已后端 HTTP 落地，当前只能标记为
“HTTP 已开放”。成员网页端修改入口、我的借用记录、后台审核控制台真实接入、提交申请时库存不足直接返回结构化 409 的后端实现、提交/修改结构化错误的更多回归测试和字段可见性缺口仍待补，因此不能写成“端侧已接入”或“可验收完成”。

1. 成员通过 `POST /api/v1/borrowing/applications` 提交物资借用，明细使用
   `material_id + quantity`，不再解析 `分类 - 名称 - 数量` 字符串。
2. 提交申请校验物资存在、状态可借、数量为正，并校验当前可用积分足以覆盖预计押金；
   如果可用积分不足，直接拒绝提交，不创建待审核申请。
3. 旧小程序申请页强制选择归还日期；新版第一阶段个人物资借用提交和修改时，后端必须要求
   `expected_return_at` 非空。
4. 提交申请不扣库存、不冻结押金，避免待审核申请提前占用资源。
5. 拥有 `borrowing.application.review` 的管理员审核通过时，先再次校验申请人可用积分，
   再在同一事务中锁定申请和
   物资，扣减 `materials.available_quantity`，并调用积分域冻结押金。
6. 审核时可用积分不足或库存不足会把申请落为 `rejected`，写入系统审核意见，不扣库存、不冻结押金。
   人工驳回必须填写驳回理由；审核通过意见选填。
7. 申请人可以修改待审核或已驳回申请。旧小程序通过详情页“修改申请”携带
   `edit=true&sb_id` 回到申请页，并调用 `/stuff-borrow/update/{sb_id}`；新版应保留
   “修改原申请”语义，不要求取消后新建。
8. 修改申请可以更新借用理由、预计归还时间和物资明细；修改已驳回申请后状态回到
   `pending_review`，原审核记录保留，新操作写审计。
9. 修改申请按当前物资台账重新提交原申请处理，后端重新校验身份、状态、物资可借、数量、押金和可用积分；
   前端可以提示风险和限制操作，但所有判断都必须由后端兜底。
10. 修改申请使用完整提交语义，申请人提交且只能提交 `reason`、`expected_return_at` 和整份 `items` 明细；
   后端不接受只增减一条明细的增量补丁，也不允许额外字段修改借用类型、项目关联、申请人、状态或押金字段。
11. 修改已驳回申请后，主状态展示为待重新审核；旧小程序只在 `state === 1` 时展示“审核反馈”，新版也不能在
    `pending_review` 主状态下继续把上一轮驳回意见当作当前结论，只能在“历史审核意见”中展示。
12. 修改申请成功后刷新 `submitted_at`，表示最新重新提交时间；`created_at` 保留原始创建时间，后台待审核队列按
    `submitted_at desc, id desc` 排序。
13. 借用申请页只填写借用理由、预计归还时间和物资明细等业务字段；姓名、学号、手机号、联系邮箱、年级和专业从成员资料自动带出。
14. 联系邮箱默认取登录邮箱，成员可以在资料中修改，修改联系邮箱不等同于更换登录邮箱；联系邮箱必须由后端校验邮箱格式，但不需要验证码确认，因为它不是登录凭证。
15. 提交或修改申请时，后端从成员资料生成申请人信息快照，联系邮箱为空但登录邮箱存在时回退使用登录邮箱。
16. 提交或修改申请时，成员资料缺少姓名、学号、手机号、年级或专业，或联系邮箱和登录邮箱均缺失时拒绝提交或修改；如果历史数据里已有非法联系邮箱，也应按资料不完整处理，并由端侧引导先完善资料。
17. 修改申请成功时，申请人信息快照按当时最新成员资料刷新；历史详情、审核和归还处理的主申请人信息展示快照，不被后续个人资料修改覆盖。
18. 后台审核列表指后台管理端/后台主线审核控制台，不是成员中心页面。
    后台审核列表沿用旧小程序摘要边界，但不要继续使用旧小程序 `年级-专业-姓名` 的拼接字符串，应拆成独立列：姓名、年级、专业、提交时间、状态、类型、物资摘要和操作；申请编号、学号、手机号、联系邮箱以及当前联系方式放详情页，不作为列表主列。
    成员端“我的借用记录”列表是本人视角，不套用后台审核表格列；优先展示状态、提交时间、预计归还时间、物资摘要和可执行操作，学号、手机号、联系邮箱以及当前联系方式只进入详情页。
19. 列表里的物资摘要使用借用明细快照生成：单项显示“物资名称 x 数量”；多项显示“首件物资名称 x 数量 等 N 项”，其中 `N` 是本次申请的明细项总数。完整物资清单、分类、单件押金和每项数量只在详情页展示。
20. 小程序后续定位为高频轻量入口，可以接入轻量查看、提交、状态跟踪、简单审核和正常归还；物资台账维护、物资类别多层管理、库存调整、资源状态处理、复杂筛选导出和异常归还不放在小程序端，统一放在浏览器端，管理类主入口为 `apps/admin`。
21. 如果申请人提交后修改了手机号或联系邮箱，后台审核页、归还页和历史详情页可以额外只读展示“当前成员资料中的联系方式”，但必须标注来源；当前联系方式只用于人工联系，不改写申请快照、审核事实、归还事实或审计日志。
22. 借用明细在提交或修改成功时保存申请时快照，至少包含物资名称、分类名称和单件押金积分；
    历史详情、审核和归还展示使用快照，库存操作仍通过 `material_id` 访问当前物资台账。
23. 后台借用审核工作台默认展示 `pending_review` 待办列表；全部、已通过待归还、已驳回、已取消、已归还和异常关闭通过筛选查看。
24. 审核动作只在 `pending_review` 展示；正常归还和异常归还动作只在 `approved` 展示，历史状态默认不混进待办列表。
25. 申请人可以取消待审核或已驳回申请，取消原因选填，状态变成 `cancelled`，保留申请、明细和审核记录；
   取消已驳回申请不是删除记录，也不是撤销审核结论；空取消原因不能阻塞合法取消。
26. 成员端“我的借用记录”展示全部状态，但只对 `pending_review` 和 `rejected` 展示修改和取消动作；
   `approved` 只表示等待后台归还处理，不提供成员自助归还入口。
27. 旧小程序物资详情页用 `state != 2` 展示操作按钮，这个条件过宽；新版只能在
    `pending_review` 和 `rejected` 展示修改或取消，不能对 `returned`、`cancelled`
    或 `exception_closed` 展示操作。
28. 归还动作不是普通借用人自助动作；第一阶段由后台主线审核控制台处理，后续可接入小程序管理端。
29. 后台界面必须把“正常归还”和“异常归还”拆成两个动作，不能只给一个含糊的“确认归还”按钮。
30. 第一阶段后端暂不拆正常归还和异常归还权限点，两者都暂时检查 `borrowing.application.review`；
   但端侧必须用动作拆分、二次确认和备注必填表达风险差异。
31. 正常归还提交前使用轻量二次确认弹窗，提示将恢复可借库存并解冻本次冻结押金；备注可选。
32. 异常归还提交前使用高风险二次确认弹窗，提示将全额扣除本次冻结押金且不自动恢复可借库存；必须填写原因。
33. 正常归还时恢复可借库存并解冻押金，状态变成 `returned`。
34. 损坏、遗失、消耗类归还会全额扣除本次申请冻结押金，状态变成 `exception_closed`，
   不自动恢复可借库存，后续由资源管理员通过库存调整入口处理。
35. 第一阶段不支持部分扣除、按损坏数量扣除或管理员手填扣除金额；这些细则进入 1.5 阶段。
36. 异常归还必须填写备注，正常归还备注可选；空备注不能触发押金扣除和异常关闭。
37. `exception_closed` 是第一阶段终态，不支持普通撤销或重新归还；误操作通过受控积分反向修正、
    库存调整和审计记录兜底，专门撤销流程进入 1.5 阶段。

### 场地借用

状态：已实现。

当前接口：

- `POST /sites-borrow/post`
- `GET /sites-borrow/view`
- `GET /sites-borrow/view-all`
- `GET /sites-borrow/detail/{apply_id}`
- `PATCH /sites-borrow/review/{apply_id}`
- `PATCH /sites-borrow/return/{apply_id}`
- `POST /sites-borrow/cancel/{apply_id}`
- `PATCH /sites-borrow/update/{apply_id}`
- `POST /admin/api/site/review-borrow/{apply_id}`
- `POST /admin/api/site/return-borrow/{apply_id}`

当前申请流程：

1. 已登录用户提交场地借用表单。
2. 请求包含申请人信息、指导老师信息、场地 ID、场地名称、工位号、开始时间、结束时间、用途和可选项目 ID。
3. 后端解析 ISO 时间字符串。
4. 后端校验开始时间早于结束时间。
5. 后端校验目标场地和工位存在。
6. 后端创建 `SiteBorrow`，状态为 `0`。
7. 申请创建时不占用工位。

当前审核流程：

1. 管理员或成员审核申请。
2. 请求使用 `state=1|2` 和 `review`。
3. `state=1` 表示打回，并要求审核意见。
4. `state=2` 表示批准，后端锁定申请和目标工位。
5. 后端校验工位仍存在且未占用。
6. 后端设置 `Site.is_occupied = true`。
7. 后端把申请状态改成 `2`。
8. 后端保存审核意见。

旧系统场地归还流程（仅作为迁移参照；场地/工位不进入当前最小物资借用验收主线）：

1. 旧系统允许用户或管理员确认归还。
2. 后端检查申请状态必须是 `2`。
3. 后端把申请状态改成 `3`。
4. 后端把目标工位 `is_occupied` 改成 `false`。

当前修改和取消流程：

1. 用户只能修改自己的申请。
2. 申请必须是 `0` 或 `1`。
3. 如果已打回申请被修改，状态重置为 `0`，审核意见清空。
4. 用户可以取消状态 `0`、`1` 或 `2` 的申请。
5. 取消时状态变成 `4`。
6. 如果原状态是 `2`，后端释放工位。

当前问题：

- 场地借用比物资借用更像完整状态机。
- 审核格式和物资借用不一致。
- 如果批准时发现工位已被占用，当前代码尝试自动打回，但随后回滚事务，自动打回可能不会保存。

目标归属：

- 借用域负责申请和审批；
- 资源域负责工位记录。

- `PUT /project/update/{project_id}`

当前流程：

1. 已登录用户创建项目。
2. 当前用户自动成为项目负责人。
3. 请求包含：
   - 项目名称；
   - 项目类型；
   - 项目描述；
   - 开始和结束时间；
   - 指导老师信息；
   - 是否招募；
   - 初始成员 maker id 列表。
4. 如果负责人把自己放进成员列表，后端拒绝。
5. 后端创建项目，状态为 `0`。
6. 后端为有效成员创建项目成员记录。
7. 审核员查看项目审核列表，可以按状态筛选。
8. 审核员审核：
   - `state=1` 表示通过并进入进行中；
   - `state=2` 表示打回。
9. 负责人只能在状态 `0` 或 `2` 修改项目。
10. 修改后项目状态重置为 `0`。

当前问题：

- 审核权限基于全局角色，不是项目部或项目审核权限。
- 项目基础信息更新和成员更新不是一个统一命令。

目标归属：

- 项目域负责项目生命周期；
- 组织与成员提供用户和成员信息。

### 项目成员管理

状态：已实现。

当前接口：

- `POST /project/member/add/{project_id}`
- `DELETE /project/member/{project_id}`

当前流程：

1. 项目负责人提交要添加或移除的 maker id。
2. 后端检查项目存在。
3. 后端检查当前用户是项目负责人。
4. 添加成员时：
   - 忽略负责人自己；
   - 忽略已存在成员；
   - 把有效用户加入项目成员。
5. 移除成员时：
   - 删除匹配的项目成员；
   - 忽略不在项目中的用户。

目标归属：

- 项目域负责项目内成员；
- 组织与成员负责协会级成员身份。

### 项目招募

状态：已实现一部分。

当前接口：

- `PUT /project/action/toggle-recruit/{project_id}`

当前流程：

1. 项目负责人或 `role >= 1` 的用户切换招募状态。
2. 后端更新 `is_recruiting`。

当前问题：

- 没有实现申请加入项目的流程。
- 现在只是一个开关。

目标归属：

- 项目域负责项目招募；
- 社区后续可以承载项目讨论和公开展示。

### 项目材料与结项

状态：部分实现。

当前接口：

- `POST /project/material/upload/{project_id}`
- `DELETE /project/material/{material_id}`
- `DELETE /project/material/all/{project_id}`
- `PUT /project/action/submit-closure/{project_id}`

当前流程：

1. 只有项目负责人可以上传材料。
2. 项目必须是状态 `1`。
3. 后端把文件保存到 MinIO 的 `MATERIALS` 桶。
4. 后端创建 `ProjectMaterial` 记录。
5. 项目状态为 `1` 时，负责人可以删除单个材料或清空所有材料。
6. 提交结项时：
   - 项目必须存在；
   - 当前用户必须是负责人；
   - 项目必须是状态 `1`；
   - 至少存在一个材料记录。
7. 后端把项目状态改成 `3`，保存结项描述。

当前问题：

- 模型里有 `state=4` 和 `finish_review`，但当前路由没有最终结项审核。
- 旧 Apifox 有结项相关动作，但当前代码只实现了提交，没有实现结项审批/打回。
- 后台清理任务会清理项目材料，需要在重构前重新确认策略。

目标归属：

- 项目域负责结项生命周期；
- 文件存储属于基础设施；
- 审计域记录审核决定。

## 对业务域划分的验证

这份工作流分析支持当前的业务域划分：

| 业务域 | 为什么需要 |
| --- | --- |
| 身份与权限 | 登录、账号绑定、JWT、角色、权限检查 |
| 组织与成员 | 成员、部门、花名册、成员统计 |
| 积分与账本 | 积分账户、积分流水、冻结、兑换、规则和审计 |
| 资源 | 物资、场地、设备主数据和可用状态 |
| 借用 | 物资/场地申请、审批、归还、取消 |
| 项目 | 项目生命周期、项目成员、材料、结项 |
| 工作台 | 任务、排班、值班、内部运营 |
| 内容 | 活动、宣传链接、未来博客 |
| 社区 | 未来用户生成内容，和官方内容分开 |
| 电控 | 未来物理硬件控制，需要强权限和审计 |
| AI | 未来助手和知识库，不能成为业务事实来源 |
| 通知 | 跨业务域消息和提醒 |
| 审计 | 跨业务域操作和审批轨迹 |

最重要的边界是：**资源和借用必须分开**。

当前代码中借用流程直接修改库存和工位占用状态，但业务含义不同：

- 资源回答“有哪些东西，它们现在是否可用”；
- 借用回答“谁申请使用了什么，谁审批了，是否归还”。

同样，**身份与组织也必须分开**：

- 身份回答“这个主体能否登录，以及拥有哪些权限”；
- 组织回答“这个人在协会内部是什么身份”。

**内容和社区也必须分开**：

- 内容是官方发布；
- 社区是用户生成内容和互动。

## 待确认产品问题

在创建新项目骨架前，需要确认：

- 物资借用和场地借用是否立刻统一成一个借用模型，还是先分别重建再合并；
- 物资借用取消是否应该保留取消记录，而不是直接删除申请；
- 四个部门分别能审批哪些流程；
- 积分是否支持转账、透支、冻结、兑换和人工冲正；
- 哪些业务行为会产生积分，哪些行为会消耗积分；
- 值班申请到底是换班申请、值班报名、请假申请，还是其他流程；
- 清洁模块是否仍然是有效产品需求；
- 项目结项是否需要单独的审核和打回流程；
- 项目招募是否需要申请加入流程；
- 旧小程序是否需要在重构期间继续运行。
