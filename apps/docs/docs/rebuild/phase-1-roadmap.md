# MakersHub 第一阶段实施路线图

## 目标

第一阶段目标是交付一组可以运行、可以验收、可以继续扩展的核心业务闭环。

第一阶段不追求把所有规划功能一次性做完，而是先把系统底座和高频主流程做稳：身份、组织、权限、积分账本、资源、借用、项目基础和成员网页端验证链路。

## 当前基线

已完成：

- 创建 `makershub` 单仓库骨架；
- 创建 `apps/api`、`apps/web`、`apps/admin`、`apps/docs`、`apps/miniapp`；
- 创建 `packages/api-client`、`packages/ui`、`packages/config`；
- 已将原 `MakersHub_Front-end` 小程序源码导入 `apps/miniapp`；
- 迁移重构文档到 `apps/docs/docs/rebuild`；
- 创建 FastAPI 最小应用入口和健康检查接口；
- 创建成员网页端和后台管理端最小 React/Vite 入口；
- 创建 Docker 本地开发编排，覆盖 MySQL、MinIO、后端、成员网页端、后台管理端和文档站；
- 创建 GitHub Actions CI 基线，覆盖前端构建、文档构建、后端 lint/test 和 Docker Compose 配置检查；
- 创建后端数据库连接、Alembic 迁移、统一响应、统一错误处理、请求 ID、分页结构和 V1 路由注册入口；
- 创建身份域首批模型和迁移，覆盖用户、本地账号、微信账号、邮箱验证码、职务和用户职务关系；
- 创建初始化唯一 `999` 超级管理员的服务和 CLI；
- 创建环境、部署与发布周期文档，明确本地开发、预发布、生产、灰度、回滚和备份的基本原则；
- 确认后端采用 FastAPI、SQLAlchemy、MySQL、MinIO；
- 确认后端依赖管理采用 `uv + pyproject.toml`；
- 确认前端采用 `pnpm workspace`、React、Vite、TypeScript 和 Ant Design Pro 风格；
- 确认 Docker 作为本地开发、联调和后续部署的基础设施入口；
- 确认成员网页端优先，后台管理端重写，小程序基于旧源码后续适配新 API。

## 第一阶段范围

必须交付：

- 身份与登录：邮箱账号、密码、微信身份挂接、会话、唯一 `999` 初始化；
- 组织与成员：部门、职务、成员资料、成员状态、花名册；
- 权限系统：权限点、角色或授权关系、作用域权限、后台访问权限；
- 积分账本：积分账户、可用余额、冻结余额、流水、幂等业务事件；
- 资源基础：物资、场地、工位、库存、资源状态；
- 借用基础：物资借用、场地或工位申请、审批、取消、归还、押金冻结；
- 项目基础：立项、审核、成员、材料、结项、归档；
- 成员网页端：用网页端跑通登录、资料、资源浏览、借用申请、项目申请、积分流水；
- 后台管理端：按业务域提供最小管理和审核入口，不迁移旧后台 UI；
- 文档站：持续维护架构、接口、数据库、开发和运维文档。
- Docker 本地环境：至少编排 MySQL、MinIO、后端服务、成员网页端、后台管理端和文档站。

暂不深入：

- 3D 打印和服务器借用具体字段；
- 完整项目招募流程；
- 项目进展自动提醒和自动拉黑执行；
- 博客、社区、电控、AI；
- NFC、WiFi 在线时长、视觉检测等硬件联动；
- 完整积分经济模型、积分等级和称号。

## 实施顺序

### 0. 架构基线收尾

交付物：

- `project-structure.md`；
- `domain-division.md`；
- `requirements-checklist.md`；
- `workflows.md`；
- `phase-1-roadmap.md`；
- `environment-release-ops.md`；
- `repository-versioning.md`；
- Docker 本地环境规划；
- GitHub Actions CI 基线；
- 文档站导航。

验收标准：

- 所有文档默认中文；
- 文档中的项目结构统一指向 `makershub` 单仓库；
- 不再把旧后台前端作为迁移对象；
- Docker 相关内容进入 `infra/` 和文档，不作为后期补丁；
- 第一阶段范围和暂缓范围清晰可查。

### 1. 数据库设计

交付物：

- `database-design.md`；
- 身份、组织、权限、积分、资源、借用、项目的核心表设计；
- 枚举、状态流转、唯一约束、审计字段、软删除或归档策略；
- 初始化数据说明，例如部门、基础身份、唯一 `999`。

验收标准：

- 积分不是用户表字段，而是独立账户和流水；
- 借用押金支持冻结余额；
- 身份、职务、权限、作用域分离；
- 物资、场地、工位和借用生命周期分离；
- 项目和项目工位申请不重复建审核流程。

### 2. API 契约设计

交付物：

- `api-contract.md`；
- 统一响应格式；
- 认证和错误码约定；
- 分页、筛选、排序、上传文件约定；
- 第一阶段核心 API 清单。

验收标准：

- 网页端、后台管理端和后续小程序使用同一套业务事实；
- 小程序不再直接适配旧响应结构；
- 后台管理端只通过权限点和作用域访问业务接口；
- 重要写操作都能关联审计。

### 3. 后端最小骨架

交付物：

- FastAPI 应用入口；
- 后端 Dockerfile；
- 本地 `compose` 编排；
- 配置读取；
- 数据库连接；
- 模块注册机制；
- 权限点注册机制；
- 基础错误和响应封装；
- Alembic 初始化。

验收标准：

- 后端可以本地启动；
- 可以通过 Docker 启动后端、MySQL 和 MinIO；
- 健康检查接口可用；
- 空模块可以被统一注册；
- 后续业务模块能按 `modules/<业务域>` 增量加入。

当前进展：

- 已完成 FastAPI 入口、`/health`、`/api/v1/health`；其中 `/api/v1/health` 是 readiness，依赖异常时返回 503；
- 已完成 Dockerfile 和 `infra/docker/compose.dev.yml`；
- 已完成 SQLAlchemy 异步数据库连接、会话工厂和数据库健康检查；
- 已完成 MinIO 健康检查；
- 已完成统一成功响应、统一错误响应、请求 ID 和基础分页结构；
- 已完成运行日志基础设施，包含 Loguru 初始化、文件分流、轮转保留、标准 logging 桥接和请求耗时日志；
- 已完成 HTTP 安全边界基础版，包含安全响应头、生产 HSTS、请求体大小限制和应用层兜底限流；
- 已完成工程化守卫测试，检查后端 Python 文件头、模块说明和基础目录 README；
- 已完成权限基础版，包含权限点入库、默认角色、用户角色授权、`998/999` 映射和 `require_permission(...)`；
- 已完成审计基础版，包含 `audit_logs`、审计服务和审计日志读取接口；
- 已完成文件元数据基础版，包含 `files` 表、对象 key 生成和元数据登记服务；
- 已完成 V1 路由注册入口，后续业务模块可以按业务域挂载；
- 已完成 Alembic 初始化配置和版本目录；
- 权限写接口、后台菜单过滤、业务作用域规则和统一上传接口仍待实现。

### 4. 核心业务后端

交付物：

- 身份与登录；
- 组织与成员；
- 权限系统；
- 积分账户和流水；
- 资源基础；
- 借用基础；
- 项目基础。

当前进展：

- 已完成身份域首批数据库表；
- 已完成基础职务种子数据；
- 已完成唯一 `999` 初始化服务；
- 已完成微信登录创建或复用用户主体的服务层逻辑；
- 已完成微信用户绑定邮箱生成待设置密码本地账号的服务层逻辑；
- 已完成短期 access token、长期 refresh token、会话表、令牌轮换、退出撤销和当前用户依赖；
- 已完成微信登录 HTTP 接口、`/auth/refresh`、`/auth/logout` 和 `/auth/me`；
- 已完成已登录用户绑定邮箱的验证码发送、限流、消费和绑定接口；
- 已完成网页端首次邮箱验证码登录、首次设置密码和邮箱密码登录接口；
- 权限点注册、业务鉴权、密码重置和更换邮箱仍待实现。

验收标准：

- 普通用户先通过小程序微信无感登录建立用户主体，再绑定邮箱；
- 已登录小程序用户可以通过邮箱验证码绑定本地账号入口；
- 用户可以用已绑定邮箱的验证码首次登录网页端并设置密码；
- 网页端第一版不提供普通用户自助注册入口；
- 第一个 `999` 通过受控运维初始化命令创建本地账号，后续再绑定微信；
- 用户主体可以挂接微信身份和本地账号；
- 小程序端 token 过期后可以自动重新登录，不要求用户手动清理缓存；
- 网页端和后台管理端可以通过 refresh token 维持登录态，并支持退出登录后立即撤销会话；
- 权限检查不使用 `identity_code >= 1` 这类数字比较；
- 借用申请可以冻结积分，取消或归还可以解冻；
- 项目立项、审核、材料上传和结项可以形成闭环。

### 5. 成员网页端

交付物：

- 登录和首次设置密码；
- 个人资料；
- 积分余额、冻结积分和流水；
- 资源浏览；
- 借用申请和记录；
- 项目立项、材料上传、结项提交。

当前进展：

- 已完成成员网页端认证页面；
- 已接入邮箱密码登录、首次邮箱验证码登录和首次设置密码；
- 已接入 access token 启动校验、refresh token 自动续签和退出登录；
- 已完成当前用户最小资料展示；
- 已接入组织域成员资料读取与自助更新；
- 积分、资源借用和项目页面仍待实现。

验收标准：

- 成员网页端可以跑通第一阶段核心业务；
- 长表单和文件上传优先在网页端验证；
- 页面不依赖旧接口响应；
- 网页端和后续小程序共享 API 契约。

### 6. 后台管理端

交付物：

- 新后台布局和权限菜单；
- 成员管理；
- 资源管理；
- 借用审核；
- 项目审核；
- 积分和审计查看；
- 异常处理入口。

验收标准：

- 后台管理端直接重写，不迁移旧后台 UI；
- 业务人员只能进入自己有权限的模块；
- `998/999` 只作为底层系统兜底身份，不作为日常业务审批角色；
- 后台操作能产生审计记录。

### 7. 小程序接入准备

交付物：

- 小程序 API 适配策略；
- 小程序页面改造清单；
- `apps/miniapp` 接入说明。

验收标准：

- 微信开发者工具可以直接打开 `apps/miniapp`；
- 小程序优先接入已经由网页端验证过的 API；
- 能保留 UI 的页面优先保留，业务流程变化大的页面再重做。

## 验收总标准

第一阶段完成时，应满足：

- 后端、成员网页端、后台管理端、文档站都能本地启动；
- Docker 可以启动第一阶段所需基础服务；
- 身份、权限、积分、借用、项目基础闭环可演示；
- 所有核心变更都有中文文档；
- 数据库模型和 API 契约能支撑后续小程序接入；
- 暂缓需求有明确边界，不阻塞核心闭环。

## 文档维护要求

后续每次进行以下变更，都必须同步更新中文文档：

- 目录结构调整；
- 数据库表结构调整；
- API 契约调整；
- 权限点或身份规则调整；
- 积分规则、临时积分规则、撤回规则调整；
- 运维、部署、灾备流程调整。
