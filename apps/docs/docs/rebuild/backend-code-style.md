# 后端代码注释与工程化规范

## 目标

本文档记录 MakersHub 后端代码的注释风格、目录说明和工程化可交接要求。

后续无论是人工开发还是 AI 协作，都必须优先遵守本文档。原因很简单：本项目会长期维护，代码不仅要能运行，还要能让后续接手的人快速理解业务边界、框架机制、运维入口和风险点。

## 总体原则

- 后端代码默认使用中文注释和中文文档。
- 每个 Python 文件都应该有文件头和模块级说明。
- 注释重点解释“为什么这样设计”和“这个文件负责什么”，不要只重复代码正在做什么。
- 重构前应主动查看旧 `makershub-backend` 和小程序代码，把旧代码作为业务流程、注释风格、接口行为和边界条件的参照。
- 重构不是复制旧实现；可以调整架构、拆分领域和清理历史问题，但不能脱离旧业务语义，也不能丢掉旧代码中已经说明清楚的意图。
- 业务规则、权限边界、初始化流程、迁移原因、框架特殊机制必须写清楚。
- 新增主要目录时必须同步添加或更新 `README.md`。
- 修改需求、接口、数据库、运维行为时，必须同步更新仓库文档。

## 旧代码参照要求

旧 `makershub-backend` 和旧小程序源码是重构期间的重要业务参考，不是最终架构模板。

每次实现旧系统已经存在或相近的功能前，应先查看旧代码，至少确认：

- 旧接口路径、请求字段和响应字段；
- 旧业务状态、状态流转和异常处理；
- 旧权限判断和入口归属；
- 旧注释中解释过的框架机制、临时处理和历史修复；
- 小程序或后台页面实际依赖了哪些字段和行为。

查看旧代码后，应按新架构重新落地：

- 业务域边界以 `domain-division.md` 为准；
- 需求口径以 `requirements-checklist.md` 为准；
- 数据库和接口以当前重建设计文档为准；
- 注释风格参考旧项目的中文工程化说明，但避免保留已经过时的实现细节。

如果新实现与旧行为不同，必须在代码注释或重构文档中说明差异原因。

## 文件头规范

每个后端 Python 文件第一行使用路径标识：

```python
# app/core/database/session.py
```

迁移文件使用：

```python
# migrations/versions/20260429_0002_xxx.py
```

文件头之后必须有模块级 docstring，说明：

- 文件负责什么；
- 为什么需要这个文件；
- 和其他层的边界；
- 必要时列出主要功能。

示例：

```python
# app/core/database/session.py
"""
数据库连接与会话管理

本文件负责创建全局异步 engine、异步 Session 工厂，以及 FastAPI 依赖注入使用的
数据库会话。业务代码不应该自己创建 engine，避免连接池分散、事务边界混乱。

主要功能:
1. 根据 Settings 创建 SQLAlchemy AsyncEngine；
2. 提供每个请求独立使用的 AsyncSession；
3. 提供健康检查使用的数据库 ping；
4. 应用关闭时释放数据库连接池。
"""
```

## 模块分区注释

文件内部使用分区注释标明结构：

```python
# --- 全局数据库引擎 ---
# --- Session 工厂 ---
# --- 中间件注册 ---
# --- 异常处理与路由注册 ---
# --- V1 路由聚合 ---
```

分区注释用于帮助读者快速扫描文件结构，尤其适用于：

- 应用入口；
- 配置文件；
- 数据库基础设施；
- 路由聚合；
- 迁移脚本；
- 业务模型和服务文件。

## 函数和类注释

公共函数、服务函数、类、FastAPI 路由、CLI 命令都应写 docstring。

docstring 至少说明职责。涉及参数、返回值、风险点时，应补充 `Args`、`Returns`、`注意`。

示例：

```python
async def get_session() -> AsyncIterator[AsyncSession]:
    """
    FastAPI 数据库会话依赖。

    每个请求创建一个独立 AsyncSession，请求结束后自动关闭连接。
    事务提交/回滚由具体接口或服务层显式控制，避免隐藏副作用。
    """
```

对于复杂函数，应该写清楚业务分支：

```python
async def login_wechat_identity(...) -> WechatLoginResult:
    """
    处理小程序微信登录。

    - 已存在 openid：复用同一个用户主体，并刷新登录时间；
    - 首次出现 openid：创建用户主体和微信凭证；
    - unionid 暂时可空，但如果传入，必须和已有绑定关系一致。
    """
```

## 行内注释使用场景

行内注释不要滥用。只在以下场景使用：

- 解释框架机制，例如 FastAPI 请求体重置、SQLAlchemy 异步迁移桥接；
- 解释容易误改的业务约束，例如 `999` 初始化、`unionid` 冲突、密码哈希为空；
- 解释运维或安全原因，例如 secret 来源、连接池释放、健康检查边界；
- 标记模块结构或迁移顺序。

不需要写这种注释：

```python
user_id = user.id  # 获取用户 ID
```

应该写这种注释：

```python
# 普通用户第一版必须先由微信登录建立用户主体，不能从网页端直接注册。
```

## README 规范

每个主要目录的 `README.md` 至少说明：

- 这个目录负责什么；
- 这个目录不负责什么；
- 当前已落地的能力；
- 后续待实现内容；
- 如果有运维命令，需要给出本地开发和生产注意事项。

业务域 README 应该明确领域边界。例如：

- `identity` 负责登录身份，不负责部门、职务和业务权限授予；
- `resources` 负责资源基础数据，不负责借用生命周期；
- `points` 负责账本，不允许业务域直接修改用户积分余额。

## 迁移脚本注释

数据库迁移必须写清楚需求背景，不能只写自动生成的 revision 信息。

迁移文件应该包含：

- 文件路径头；
- 中文标题；
- revision 信息；
- 需求背景；
- `upgrade` 和 `downgrade` 的职责说明；
- 重要表、索引、回滚顺序的注释。

示例：

```python
# migrations/versions/20260429_0002_allow_pending_local_account_password.py
"""
允许本地账号处于待设置密码状态

需求背景:
普通用户第一版必须先通过小程序微信登录建立用户主体，再绑定邮箱；
网页端首次登录时才设置密码。因此 local_accounts 需要支持
“邮箱已验证但 password_hash 为空”的状态。
"""
```

## 身份域特别约定

身份与登录是第一阶段最容易跑偏的模块，代码注释必须反复明确以下约束：

- 用户主体不等于微信 `openid`、`unionid` 或邮箱；
- 普通用户第一版先通过小程序微信无感登录建立用户主体；
- 普通用户绑定邮箱后，本地账号可以处于“密码未设置”状态；
- 网页端首次登录只能消费已绑定邮箱，不能创建普通新用户；
- 第一个 `999` 是例外，只能通过受控运维初始化命令创建；
- `998/999` 是绑定到用户上的系统身份，不是独立后台账号；
- `unionid` 当前可能拿不到，因此系统启动不能依赖 `unionid`。

## AI 协作要求

后续 Codex 或其他 AI 继续开发前，必须先阅读本文档，并在实现时遵守以下规则：

- 实现具体业务前，先查看旧 `makershub-backend` 和相关小程序代码，提取业务语义后再按新架构实现；
- 新增 Python 文件时，先写文件头和模块 docstring；
- 新增业务逻辑时，同步补充关键业务意图注释；
- 新增主要目录或业务域时，同步写 README；
- 修改需求口径时，优先更新 `requirements-checklist.md`，再更新相关设计文档；
- 不能因为上下文压缩或会话切换而降低注释和文档要求；
- 代码最终提交前至少运行后端 lint/test，涉及文档时运行文档构建。

## 提交前检查清单

后端改动提交前至少检查：

- 新增或修改的 Python 文件是否有文件头；
- 文件是否有中文模块说明；
- 关键类、公共函数、路由、CLI 是否有 docstring；
- 业务约束是否写在代码附近，而不是只留在聊天记录里；
- 目录 README 是否需要更新；
- `apps/docs/docs/rebuild` 下的设计文档是否需要同步；
- `ruff` 和 `pytest` 是否通过；
- 文档改动是否通过 VitePress 构建。
