# modules

业务域模块目录。业务域是后端业务边界的第一层，不是最后一层。

重构的目的不是把旧代码搬进几个更大的 `service.py`，而是让每条业务链路都有清晰边界。
因此业务域内部还要继续按能力、聚合或状态机拆分。

业务模块按业务边界拆分，不按协会部门拆分。

当前已开始落地的基础业务域：

- `identity`：用户主体、微信登录、邮箱密码账号、双 token 会话；
- `organization`：成员资料、部门、职务关系；
- `audit`：审计日志和重要操作追踪；
- `files`：文件元数据、对象 key 生成和文件状态管理；
- `points`：积分账户、冻结记录、积分流水和受控人工调整。

## 域内拆分规则

小型业务域可以临时使用 `models.py`、`repository.py`、`service.py`。一旦出现以下情况，
必须继续拆二级目录：

- 单个 `service.py` 超过 350 行；
- 单个 `repository.py` 超过 300 行；
- 一个业务域同时处理 3 条以上独立业务链路；
- 一个业务链路有独立状态机，例如借用、项目审核、积分规则审批；
- 新增功能需要频繁修改同一个大文件的多个分区。

具体结构以 `apps/docs/docs/rebuild/02-架构设计/04-后端业务域内部架构.md` 为准。

当前 `identity` 和 `points` 已经完成第一轮二级能力拆分；`organization` 接下来也需要
继续拆成成员、部门、职务等能力模块。后续新增 `resources`、`borrowing`、`projects`
时不能再先写成单个大 service。
